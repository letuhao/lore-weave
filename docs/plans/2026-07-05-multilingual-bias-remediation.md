## Key design decisions

- **DD-1 — A5 is forward-only under `canonical_version=1`.** `canonical_version` is hardcoded `1` everywhere and **no re-canonicalization pass exists**. A full versioned migration (re-hash all entities, merge collisions across Neo4j + Postgres SSOT) is gate-#2 structural AND barred by the "no destructive ops on shared dev DB" rule. Instead: expand the honorific list, keep v1. New extractions dedup native-honorific variants correctly *immediately*; the few pre-existing stranded nodes are reconciled by a **one-time re-canon backfill script** — written + unit-tested here, **not run** against shared dev DB (tracked `D-ML-A5-RECANON-BACKFILL`, operator-run). Determinism is preserved (same input→same output forever under the new list).
- **DD-2 — Detection is degrade-open with bounded script-range extensions, NOT a new segmenter.** A2/A3/A4 extend the existing CJK-range + per-language `get_patterns(lang)` registry (add hiragana/katakana `぀-ヿ` + hangul `가-힯` ranges; per-language stop-particles; degrade-open negation NP). No jieba/MeCab dependency (that's the ideal, tracked separately if ever justified). Matches the repo's documented degrade-open philosophy.
- **DD-3 — A2/A3/A4 are ONE coupled slice.** The English stopwords exist to filter the English capitalized-word heuristic; per-language stops only make sense with per-language detection. Plan/build/commit together.
- **DD-4 — i18n translations generated by local LM Studio `google/gemma-4-26b-a4b-qat`** at dev-time (idiomatic per-locale, $0). Generated JSON is committed static content → no provider-gateway/runtime concern. Spot-check zh-CN terminology (model may drift Traditional).
- **DD-5 — Add `zh-CN` (Simplified, mainland) + `ko`; keep `zh-TW`.** zh-CN ≠ zh-TW (Simplified script + different UI terminology: 软件 vs 軟體, 视频 vs 影片). Simplified reaches ~1.4B vs ~30M Traditional, and the **backend already T2S-normalizes to Simplified** — FE shipping Traditional-only is an internal mismatch.
- **DD-6 — Full UI-locale set = 18** (user decision 2026-07-05, 4-tier strategy): **T1 global** en, zh-CN, zh-TW · **T2 SEA+LatAm** vi, id, ms, es, th · **T3 mature/high-ARPU** ja, ko, pt-BR, fr, de · **T4 huge-pop/long-term** hi, ar (RTL), ru, tr, bn. Scripts: Latin, Han (Hans/Hant), Japanese, Hangul, Cyrillic, Arabic (rtl), Devanagari, Bengali, Thai. Extraction patterns (Axis 3) do NOT expand — stay en/vi/zh/ja/ko core + **degrade-open**. All non-en generated by gemma (DD-4). Extraction patterns (Axis 3) do NOT expand — stay en/vi/zh/ja/ko core + **degrade-open** for the rest; you don't need per-language extraction rules to render the UI or translate *into* a language.
- **DD-7 — Canonical Language Registry (SoT keystone).** One shared list — `code` (BCP-47), `englishName`, `endonym`, `script`, `dir` (ltr/rtl), `uiLocale` bool, `translationTarget` bool — replaces the scattered hardcoded 5-lang lists (`patterns/__init__.py:71`, `session_translator.py:172`, FE `index.ts` langs). Powers: the UI-locale switcher, the new **translation-target picker** (Axis 2, surfaced per user decision), RTL direction, and backend language-name maps.
- **DD-8 — RTL/bidi** for `ar` (and future he): `dir` from the registry applied at the app root + logical-property CSS audit. Thai/Hindi are LTR complex scripts (no bidi, but no-space Thai affects word-count display only, not UI strings).

**Axis 2 decision:** surface a translation-target picker now (the LLM already handles arbitrary targets; this just makes the registry's `translationTarget` languages selectable and replaces the `"en"` free-string default).
