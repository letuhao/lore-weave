## 10 ┬╖ CLARIFY тАФ `21`-**G2**: the PlanForge proposer is blind to the book it is planning *(O-1)*

> PO approved a CLARIFY now (Q4). This is the CLARIFY. It ends at a PO checkpoint (┬з10.5).

### 10.1 The defect (verified against code, not the spec's word)

`plan_forge_service.propose(created_by, **book_id**, source_markdown, тАж)` тАФ **`book_id` is already in
scope**, and the service already calls `_ensure_work(book_id)`. But it imports only `WorksRepo` /
`PlanRunsRepo` / `GenerationJobsRepo`; and `engine/plan_forge/propose.py` + `propose_llm.py` read
**only the caller's markdown**. No `OutlineRepo`, no `MotifRepo`, no glossary.

тЗТ **Proposing a spec for a 40-chapter in-progress book is architecturally identical to proposing for
a blank one.** This is *not* "blocked on a missing signal" тАФ the signal and the plumbing are both
there. It is unbuilt work. (`blocked-on-sdk-field-often-self-owned`.)

### 10.2 What actually happens today, end-to-end тАФ **and the safety net is in the wrong place**

The linker's upsert key is **`(book_id, plan_run_id, plan_arc_id)`** тАФ `plan_run_id` is *in the key*.
A second proposal тЗТ a new `plan_run_id` тЗТ **`ON CONFLICT` can never match** тЗТ a **second, disjoint set
of arcs and chapters is inserted onto the same book.**

There *is* a guard тАФ PF-10's `_FIND_DUPES` тАФ and it is honest (*"we surface it; we never merge it"*).
**But it matches on `lower(btrim(title))`.** A proposer that never read the book invents *fresh*
titles ("Arc 1 тАФ The Beginning") for a book already at chapter 40 тАФ **so the titles don't collide and
the detector never fires.**

> **The safety net is downstream of the blindness, and it is keyed on the one signal the blindness
> destroys.** It can only catch a blind proposer that *happens to guess the same titles*.

### 10.3 The three scenarios (this is the actual product question)

| # | Scenario | Today |
|---|---|---|
| 1 | **Cold start** тАФ blank book + braindump тЖТ full spec | тЬЕ works. **Must keep working.** |
| 2 | **Mid-book** тАФ 40 chapters written, "plan the next arc" | ЁЯФ┤ proposes a whole new plan from scratch |
| 3 | **Retro-fit** тАФ imported book, prose but no plan | ЁЯФ┤ same. *(The other path to this was `26`-D3's arc decompiler тАФ **O-2, orphaned**.)* |

### 10.4 Options

| | Approach | Size | Trade |
|---|---|---|---|
| **A** | **Refuse + redirect.** If the book already has spec rows, `propose` 409s: *"this book has a plan тАФ use `plan_apply_revision`."* | **XS** | Honest and safe, but leaves scenarios 2 + 3 **unserved**. |
| **B** тнР | **Ground the proposer.** Feed a compact book-state digest (existing arcs ┬╖ chapter spine ┬╖ cast roster ┬╖ active motifs) into the charter proposer, so it proposes a spec that **continues** the book. | **M** | тЪб **`composition_package_tree` (28-AN-2) ALREADY builds exactly this digest тАФ spec + manuscript + index + coverage in ~138 tokens.** The digest exists, the summarizer exists, `book_id` is in scope. **This is a wiring job, not a new engine.** |
| **C** | **Delta-propose** тАФ propose only the delta beyond what exists. | **L** | Most correct; needs diff semantics nobody has specced. |

**Recommendation: B, with A as the fail-closed fallback** тАФ ground the proposer; and if the digest
cannot be built (a degraded read), **refuse rather than propose blind**. *Absent тЙа zero: "I could not
look at the book" must never become "the book is empty."* That is exactly the bug class this cluster
has shipped twice.

**Effect test (the DoD, per `checklist-is-self-report-enforce-by-tests`):** seed a book with an arc
and 40 chapters тЖТ `propose` тЖТ **the packed proposer prompt CONTAINS the existing arc's title.**
Change the book's state тЗТ the prompt changes. (The same discipline as BA12 and PF-13 тАФ prove it by
*effect*, never by a source assertion.)

### 10.5 Rules mode тАФ **RESOLVED** (2026-07-13, by reading what it actually is)

**What rules mode is:** `plan_propose_spec` runs two modes over the same input тАФ a markdown document
the **author wrote themselves**. `rules` **parses** it deterministically (**$0**, no model,
**synchronous**); `llm` has a model **write** the spec from it (billed, async, needs `model_ref`).

тЗТ **Rules mode is a TRANSCRIBER, not a generator.** Its pitch is: *"I already wrote my plan. Structure
it for me тАФ **don't let an AI reinterpret it.**"* And `propose.py`'s own header (written by 27-B5, the
fixture severing) states the law:

> *"parse what the document says; **emit nothing where it says nothing**. Absent тЙа invented тАФ an honest
> empty tells the author exactly what their braindump is missing."*

*(Before B5, rules mode was a **fixture wearing a parser**: for ANY document it returned another
novel's variables, anchors, forbids and protagonist. Not a crash, not an empty тАФ **a confident wrong
answer.** P-06.)*

**So the question answers itself.** **Grounding rules mode = inventing content the document does not
contain** тАФ which is *precisely* the bug B5 just killed. Rules mode must **not** be grounded, not as a
compromise but because that is the only thing consistent with what it **is**.

**The resolution тАФ pre-flight, not grounding:**

| Mode | Book-state treatment |
|---|---|
| **rules** | **PRE-FLIGHT.** Read the book's state; do **not** feed it to the parser. **Report** the collision тАФ *"this book already has 3 arcs / 40 chapters; your document proposes 2 arcs matching none of them"* тАФ and stop behind an explicit confirm. **Never invents** (rules mode's contract) and **never silently re-plans** (*absent тЙа zero*). |
| **llm** | **GROUND** (option B) тАФ digest in, a spec that *continues* the book out. Fail-closed: a degraded read тЗТ refuse, never propose blind. |

**This also fixes ┬з10.2's real defect**, which is not in the proposer at all: `_FIND_DUPES` keys on
**title**, so it can only catch a blind proposer that *happens to guess the same titles*. The
pre-flight compares **book state**, not strings тАФ so it fires exactly when the blindness would have
been silent.
