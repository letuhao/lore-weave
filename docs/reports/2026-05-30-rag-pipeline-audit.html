LoreWeave — RAG Pipeline Architecture & Evaluation Audit

Technical review of the knowledge-extraction + retrieval pipeline and a critical assessment of the evaluation methodology.
Generated 2026-05-30 · repo lore-weave @ 0d456f27 (main) · audit scope: knowledge-service, loreweave_extraction SDK, eval framework, provider gateway.

Verdict — both pipelines are architecturally sound; the eval has two fixable setup flaws (not a model-quality problem)

The extraction is producing genuinely good output — the models are near-top-tier open models (Google Gemma-26B, Alibaba Qwen-30B/35B), and independent strong-model review (Claude, across sessions — including manual spot-checks this session that agreed with the pipeline's keep/drop decisions) corroborates real quality. Model quality is not the concern. What is not reliable is the eval's absolute numbers and sub-noise deltas, for two architectural reasons: (1) the eval is wired so the extractor and the precision filter are also 2 of the 3 judges — grading your own output inflates the score regardless of how strong the grader is; (2) the gold set is tiny (9 chapters, 1–6 items each) with no confidence intervals, so ship decisions on ±0.1–0.3pp F1 sit inside the measurement noise. Both are fixable wiring/methodology issues. There is also one real RAG correctness bug (2-hop graph retrieval is dead, §9). Full reasoning in §8.

1 · System overview & data flow

LoreWeave builds a per-user knowledge graph + memory layer over novels and feeds assembled context to chat LLMs. The relevant subsystem is two halves that share a graph store:

  • Write path (extraction): chapter text → LLM extraction (entities/relations/events/facts) → entity anchoring → Neo4j graph + passage vector index.
  • Read path (RAG retrieval): chat turn → multi-layer context assembly (summaries, glossary, graph facts, vector passages) → token-budgeted memory block → chat-service.

3 · Extraction pipeline

3.1 Two-pass architecture

Pass 1 (pattern/regex) — app/extraction/triple_extractor.py. Pure-function SVO extraction over English-only capitalised-subject + verb (-ed/-s/-ing/irregular) + noun-phrase regex, cross-checked against entity candidates. Emits quarantine triples at confidence=0.5, pending_validation=True. CJK falls straight through to Pass 2. Per-language pattern detectors live under patterns/{en,ja,ko,vi,zh}.py. In practice Pass 1 is a low-trust seeder; Pass 2 is the real pipeline.

Pass 2 (LLM) — two orchestrator layers:

  • SDK loreweave_extraction/pass2.py::extract_pass2 — library-pure. Order: entity-first (anchors the rest) → gate (no entities ⇒ return empty) → asyncio.gather(relations, events, facts) → optional recovery → optional precision filter. Returns a Pass2Candidates dataclass.
  • Service app/extraction/pass2_orchestrator.py — adds per-op caching (extraction_leaves, keyed by compute_task_id(text, op, extractor_version, model_ref) so editing one prompt invalidates only that op's cache), glossary anchoring, Neo4j persistence, and async summary enqueue.

Chunking is gateway-side: the caller sends the whole chapter as one user message + a ChunkingConfig(strategy="paragraphs"); provider-registry splits on paragraph boundaries, replays the system prompt verbatim per chunk, and merges per-op JSON across chunks (jsonListAggregator). Chunk size scales to the model's loaded context and CJK token density.

3.2 The four extractors

All share the shape load_prompt → _extract_via_llm_client → tolerant_parse → _postprocess(resolve/canonicalize/dedup/sort -confidence). Output schemas (Pydantic):

ExtractorOutput model — fieldsEnums / guards
entityLLMEntityCandidate: name, kind, aliases[], confidence, canonical_name, canonical_idkind ∈ person·place·organization·artifact·concept·other
relationLLMRelationCandidate: subject, predicate, object, polarity, modality, confidence, subject_id?, object_id?, relation_id?polarity affirm·negate; modality asserted·reported·hypothetical. Null-object relations dropped at postprocess (local models emit null objects for intransitive verbs).
eventLLMEventCandidate: name, kind, participants[], participant_ids[], location?, time_cue?, event_date?, summary, confidence, event_id?kind action·dialogue·battle·travel·discovery·death·birth·other. event_date validated against a truncated-ISO regex, malformed ⇒ None.
factLLMFactCandidate: content, type, subject?, subject_id?, polarity, modality, confidence, fact_idtype description·attribute·negation·temporal·causal. Facts are never filtered.

Tolerant parsing: iterate job.result[<op>]; drop non-dicts and items missing key fields; default aliases→[], confidence→0.5; clamp confidence to [0,1]; on enum ValidationError drop + emit knowledge_extraction_dropped_total{op,reason}. Markdown-fence recovery happens gateway-side (aggregator.go::extractJSONObject — a fence bug that had silently dropped all fenced-model output was fixed 2026-05-23).

Prompts (prompts/<op>_extraction_system.md) all open with an anti-reasoning preamble (suppress <think>), keep names/subjects/summaries in original script, force English keys/enums, inject {known_entities}, require confidence ∈ [0.5,1.0]. The relation prompt defines a 28-predicate canonical vocabulary (kinship/mentorship/authority/spatial/action/social) with CJK (拜…為師→disciple_of) + VN (con riêng→stepchild_of) worked examples added in cycle 70. The entity prompt's load-bearing rule is scene-relevance ("bias toward omission" of backstory/geographic asides).

3.3 LLM gateway path

Extractors call llm_client.submit_and_wait(operation, model_source, model_ref, input, chunking, transient_retry_budget=1). Dedicated JobOperation enums exist for the four extractors + summarize_level; the precision filter and entity recovery reuse operation="chat". model_ref is a provider-registry model UUID resolved server-side against user_models (which holds the upstream name + BYOK credential). The client retries transient errors (honouring retry_after_s) and exponential-backs-off stale-pool connect timeouts, but does not raise on status=failed — the extractor inspects. max_tokens=4096 caps exist specifically to stop LM Studio reserving full context per slot (3× R/E/F gather would OOM the GPU KV cache).

3.4 Entity resolution & anchoring

  • Canonicalization canonical.py: lowercase → strip honorifics (a deterministic tuple, longest-first, because a set would be hash-randomised across restarts) → collapse whitespace → strip punctuation. IDs are sha256("v1:user:project:kind:canonical")[:32] — tenant+project scoped, idempotent.
  • Two-stage anchoring: each extractor snaps names to known-entity canonical spelling; relation/event/fact endpoints resolve against a case-insensitive name/alias lookup, then the known_entities anchor map; unresolved ⇒ id=None.
  • Glossary anchors (two-layer pattern): a Pass-0 loader pulls active glossary entities into an in-memory AnchorIndex keyed (folded_name|alias, kind); on hit, the writer reuses the anchor's canonical_id and stamps a glossary_entity_id FK (knowledge-service entity layer anchored to the authored glossary SSOT). Glossary down ⇒ graceful mint-on-no-match.

3.5 Neo4j graph storage

app/extraction/pass2_writer.py::write_pass2_extraction runs one Cypher transaction per chapter. All persisted text is run through neutralize_injection first. Order: hierarchy MERGE (Book/Part/Chapter/Scene) → provenance source → entities (resolve_or_merge_entity + :EVIDENCED_BY) → relations → events → facts. Schema: :Entity(canonical_id, name, kind, confidence, glossary_entity_id, anchor_score, auto_created, pending_validation), :Event, :Fact; edges :RELATES_TO (keyed by relation_id), :EVIDENCED_BY. Re-extraction is idempotent (MERGE on structural IDs); evidence edges dedupe by job_id.

Endpoint repair / autocreate (cycle 73e): Tier A.1 chapter-local name-fold repair (free) → Tier A.2 anchor-index reuse (free) → Tier B env-gated MERGE of a new :Entity(kind=concept, auto_created=true, confidence≤0.3) with a per-chapter cap + noise-subject heuristic. Tier B is OFF by default; the cycle-74a F1 eval found it neutral (§7).

3.6 Precision filter

loreweave_extraction/pass2_filter.py::apply_precision_filter — a post-extraction LLM pass that re-grades each extracted item against the source text and drops the unsupported ones. Best-effort (LLM failure ⇒ degraded, candidates unchanged), immutable, per-category concurrent. Each item is judged supported|partial|unsupported in batches of 3; keep/drop by partial_policy (drop in production). Ships ON by default, relation-only (CATEGORIES=relation, cycle 73b — 0.916 F1 ≈ full-filter 0.917 at 55% less latency). Runtime-reconfigurable via Redis pub/sub (cycle 73f; the disable-semantics bug found+fixed cycle 74b this session). The filter prompt is byte-identical to the eval judge prompt — by design, but see §8.

Entry point POST /internal/context/build app/routers/context.py, called by chat-service before each turn. Dispatch on project state:

4.2 Embeddings & vector retrieval

Query embedded via provider-registry /internal/embed (bge-m3, 1024-d), cached per-worker (TTLCache, 30s). Neo4j native vector index per dimension (passage_embeddings_1024, cosine). Because Neo4j vector indexes are global, the query oversamples ×10 then post-filters on user_id/project_id for tenant isolation. Raw cosine score is then re-weighted in the selector:

  • Hub penalty — score ×0.3 (SPECIFIC_ENTITY) … ×0.9 (GENERAL) for is_hub chunks.
  • Recency weight — decay = 1/(1+age_in_chapters); signed by intent (HISTORICAL prefers old, RECENT_EVENT prefers new).
  • MMR diversification — λ=0.7, greedy lam·rel − (1−lam)·max_redundancy; redundancy = embedding cosine (Jaccard fallback). Early-exit at top_n (pool=40@3072-d costs 1.2s uncapped vs 57ms capped).
  • Optional listwise LLM rerank (opt-in per project), 1.0s inner timeout so a slow rerank falls back to MMR order.

Pool/top-N scale by intent: SPECIFIC_ENTITY pool 20→top 5; RELATIONAL/GENERAL pool 40→top 10. Intent classification is a pure regex cascade (<15ms, no LLM).

4.3 Graph fact retrieval (L2)

Resolves intent entity names → canonical IDs, then 1-hop :RELATES_TO (both directions, UNION dedupes self-loops; confidence≥0.8, valid_until IS NULL, non-pending, non-archived), optional 2-hop fan-out gated by predicate set, plus negation facts (post-filtered to resolved entities — first-class anti-hallucination signal). Note two issues: the 2-hop call is broken (§9), and temporal bucketing (current/recent vs background) is unimplemented — all relations land in background pending chapter_index on evidence edges.

4.4 Token budgeting

6000-token budget (tiktoken cl100k). When over budget the assembler re-renders after each trim, dropping lowest-value first: passages → summaries → absences → background facts → glossary. Never dropped: L0, project instructions, L1 summary, current/recent/negation facts, CoT instructions. Trimming only touches content after </project> so the cacheable stable prefix stays byte-identical (chat-service applies Anthropic cache_control there).

6 · Evaluation framework

6.1 Judges

An LLM judge reads the full chapter text and grades extraction. Precision: each extracted item → supported|partial|unsupported ("judge by meaning, not surface wording"). Recall: each gold item → "captured under ANY phrasing?" with a matched index. Batched (3/call); items the judge omits become unjudged and are excluded from the denominator (missing data ≠ miss). The precision-judge system prompt is the same file as the production filter prompt.

6.2 Scoring formulas (compute_ensemble_macros.py)

PRECISION_CREDIT = { supported:1.0, partial:0.5, unsupported:0.0, unjudged:0.0 }
RECALL_FOUND     = { "covered", "supported" }

# per chapter, over JUDGED (non-unjudged) verdicts:
chapter_p = Σ credit / n_judged
chapter_r = n_found  / n_judged

# MACRO over the ≤9 chapters (arithmetic mean), then harmonic F1:
macro_P  = mean(chapter_p)          macro_R = mean(chapter_r)
macro_F1 = 2·P·R / (P + R)          # NOT mean of per-chapter F1

Macro-averaging over 9 chapters with single-digit per-chapter denominators is the core statistical weakness (§8.3).

6.3 Three-judge ensemble

  • Fleiss κ over items all judges scored: κ = (P̄ − Pₑ)/(1 − Pₑ), Landis & Koch buckets. Measures inter-rater agreement, not correctness.
  • Majority vote: majority = top if top_count > n_complete/2 else None(disputed); unjudged excluded.
  • D11 gate: ensemble acceptable iff ≥2 judges complete; never silently downgrade a 2-judge result to look like 3-judge.
  • D12 bias: strictness_gap (flag >0.15), language_bias en/vi/zh (flag >0.15), rp_bias (informational).

6.4 External anchors

Two published benchmarks run against the real production extractor (not a stub): CoNLL-2003 NER (100 samples, seqeval strict) and DocRED unlabeled-triple (50 samples, direction-agnostic pairs). They are informational only; the sole hard gate is a sanity floor (F1≥0.10 AND extracted≥0.1×gold) to catch "regressed to empty". Both are English-only — zero signal on the CJK/VN chapters where judge leniency is worst. DeepEval G-Eval adds three more LLM-judged metrics (one per judge) — explicitly "another judge layer, not a fix for the single-judge problem."

6.5 Gold sets

10 human-annotated chapter excerpts (9 used by default; the 54 KB Sherlock chapter is excluded as too long). Annotated conservatively ("only unambiguously present; when in doubt add a trap"). The Vietnamese chapter texts are original paraphrases composed for fixtures, not canonical source.

8 · Architecture-correctness assessment

The right question is not "can we trust the local model" — the extractor and judges are near-top-tier open models (Gemma-26B, Qwen-30B/35B), and independent strong-model review (Claude, across sessions) corroborates that the extraction output is genuinely good. The question is whether the two pipelines are wired correctly. The RAG pipeline is architecturally sound (one real bug). The eval pipeline's design is above-average — inter-rater reliability, bias metrics, external anchors, saved-dump fixtures, an anti-self-reinforcement subset check that has actually changed ship decisions — but its setup has two correctness flaws that make the absolute numbers and sub-noise deltas unreliable. Both are fixable without touching the models.

8.1 · RAG retrieval pipeline — architecture is correct

The design is sound and uses standard, appropriate techniques: mode dispatch by project state, layered retrieval (L0–L3) with per-layer timeouts and graceful degradation, regex intent classification, cosine vector search with oversample-then-tenant-filter (correct for global Neo4j indexes), hub-penalty + recency-decay + MMR (λ=0.7) re-ranking, priority-ordered token budgeting with a cache-stable prefix, and glossary/L1 dedup. Multi-tenant isolation is enforced on every query. No architectural objections. The defects are a bug and an unfinished feature, not design errors: the 2-hop graph path is dead code (§9, HIGH) and temporal fact-bucketing is unimplemented (all relations land in background). Fix those two and the read path is solid.

8.2 · Eval — self-reinforcement is a WIRING flaw (independent of model quality)

This is the single most important correctness issue, and it is about how the eval is wired, not how good the models are — a frontier model grading its own output is still biased. In a default stack the models grade their own work:

The locked metric (3-judge median) includes both self-overlapping judges, and the median is frequently pinned to the filter model. The team's 2-judge "exclude-claude" recompute caught a fake +0.9pp lift in c73d — but it only excludes the filter model, never the extractor-as-Judge-B, and it is run ad-hoc per cycle rather than baked into the metric of record.

8.3 · Sample size far too small for the precision claimed

9 chapters, single-digit gold per category, several chapters with exactly one gold relation, macro-averaged, no confidence intervals — yet ship/no-ship decisions turn on ±0.1–0.3pp F1. The framework's own data shows alice_ch01 precision swinging 0.385→0.250 (−29pp) from pure LLM nondeterminism with zero fixture change. That single-run noise dwarfs every cycle's claimed lift. Deltas the docs themselves call "within Fleiss-κ noise" are nonetheless used as ship gates.

8.4 · "unjudged-excluded" can flatter the numbers

Omitted verdicts shrink the denominator instead of counting as misses. Coverage is surfaced, but historically ran 56–68% while P≈0.97 was reported — i.e. a third of items were never graded. Recent cycles are at ~100% coverage, but the mechanism remains a latent confounder.

8.5 · Multilingual axis is unanchored AND is where leniency is worst

All three judges accept CJK/VN items 8–18pp more readily than English; the only external anchors are English-only. So the Vietnamese/Chinese chapters' frequent P=R=1.00 scores are simultaneously the least trustworthy and the least independently checkable numbers in the set. The team flags this but cannot resolve whether it is real over-extraction quality or judges failing to reject non-English errors.

8.6 · Filter-output F1 over-states realized graph quality

The judge grades filter output, but the Neo4j writer cascade-drops 10–22% more at persist time (unresolved endpoints, FK cascades). "Realized" re-judges exist and are lower, but the headline cycle numbers are filter-output, not what actually lands in the graph.

Bottom line

The extraction output is genuinely good — strong-model review corroborates it, and both pipelines are well-architected. The problems are specific, structural, and fixable without changing the models: (RAG) the 2-hop graph bug; (eval) the self-reinforcement wiring (extractor + filter are also judges) and the undersized gold set with no confidence intervals. Today the eval is reliable as a relative, same-judge drift signal but its absolute F1 is inflated by self-grading and its sub-0.5pp deltas are inside the noise — so "F1≈0.91" should be read as "strong local models broadly agree the output is good," not as a calibrated accuracy figure, and tiny cycle deltas should not drive ship decisions. Make the judges disjoint from the extractor/filter, grow the gold set, and add CIs, and the numbers become defensible as absolute claims. Model quality is not the limiting factor.
