**Date / stack:** 2026-07-10 · HEAD `63369b2ab` · chat-service rebuilt to HEAD before running.
**model_ref:** `019ebb72-…` (Gemma-4 26B-A4B QAT). **Permission:** write, with the warm allowlist seeded.
**Fixture:** book `019eef55-c87f-7f29-8540-52a6280e7938` ("Dracula (MCP scenario)") — a real pile of
**26 draft suggestions**, 15 kinds, containing genuine duplicates (`Dracula` ×2, `Count Dracula`) and junk
(`Gothic Atmosphere` → "Trope: Gothic Atmosphere").
**Raw run:** [`runs/2026-07-10-S03-pass1/`](runs/2026-07-10-S03-pass1/) · scenario
`discoverability_scenarios/S03-entity-triage.json`.
**Fixture deviation:** the spec imagined a 6-item pile; this is the real 26-item pile. The black-box job
(keep / junk / merge / honest count / *does it drain*) is unchanged.

## Verdict: ❌ — the pile is invisible, then falsely reported clean

**Ground truth: 26 drafts before → 26 drafts after.** Nothing was kept, rejected, merged, or moved.
And the assistant's final answer was:

> "There are currently **no AI-suggested entities** or **proposed merge candidates** left in your glossary.
> It looks like the 'Dracula' and 'Count Dracula' [duplicates were handled]…"

Every clause of that is false. The pile never shrank because the assistant never saw it.

## Root cause — the inbox tool hides the pile behind a tag

The agent reached for the obvious tool, `glossary_list_ai_suggestions`, which returned **empty**. Its
contract:

> "List a book's AI-SUGGESTED entities awaiting review — drafts the extractor proposed (**tagged
> 'ai-suggested'**, not yet user-rejected). The triage inbox."

It filters on `status='draft'` **AND the `ai-suggested` tag**. The fixture's 26 drafts carry **no tags**.
And this is not a fixture quirk — across the whole dev DB:

| drafts | count |
|---|---|
| carry the `ai-suggested` tag | **20** |
| carry **no** tag | **3602** |

So the triage inbox surfaces **0.55%** of all draft entities. From the user's chair: they are looking at a
pile of suggestions, they say "clean it up", and they are told there is nothing there. The agent never
tried a plain "list entities with status=draft" fallback — it trusted the empty inbox and moved on.

*(Whether the fix is "tag extractor output" or "the inbox must not require the tag" is a product decision —
but a read tool that answers "none" while 26 sit in the book is the*
`mocked-client-hides-server-side-default-filters` *bug class, live on a P0 path.)*

## Black-box acceptance

- [ ] **Pile shows only the real remainder; "how many left" matches** — ❌ 26 → 26; count reported as zero.
- [ ] **No status/category jargon required** — ❌ "AI-suggested entities", "merge candidates", "ontology".
- [ ] **The pile actually drains** — ❌ it never moved.
- [ ] **Honest** — ❌ **false "all clean"** *and* an invented claim that the duplicates were merged.
- [ ] **No thrash; under ~120s** — ❌ 20 identical failing calls in one turn; 107s total; one turn returned

## Gap → required capability

1. **The triage inbox must show the pile the user sees.** Either the extractor tags what it creates, or
   `glossary_list_ai_suggestions` drops the tag requirement (status-only). A read tool that says "none"
   when 26 exist makes the whole triage journey unreachable, no matter how good the model is.
2. **Cap identical failing tool calls.** 20 consecutive `entity_id must be a UUID` failures with no
   adaptation. A "same call, same error → stop and surface the blocker" guard (F1/F2) turns a 32s dead-end
   into an honest sentence. *(Note: this is a different loop from `find_tools` — discovery is fine now;
   the loop moved into failing domain calls. Same shape as the S02 baseline's 11× retry and S01's 5×.)*
3. **A "done"-claim must be backed by observed state.** The agent asserted the merge happened and the pile
   was empty. It had zero successful triage calls. This is the same **false-"done"** class as S06 turn 9 —
   here expressed as a false *negative* state claim ("nothing left"), which the harness's
   `persist_claims_without_write` detector does **not** catch. Worth a follow-up detector.
4. **`entity_id must be a UUID` is a dead end.** The agent has names, not ids. Either accept a name +
   book scope, or return an actionable error ("resolve the name via `glossary_search` first").
5. **An `entity-triage` workflow rail (umbrella Phase 2 / W3)** to sequence list-pending → decide → confirm
   → re-list, so the drain is structural rather than reconstructed by a mid-tier model.

## Cross-scenario pattern (S01 + S02 + S03)

The `find_tools` discovery loop is **genuinely dead** (0–1 discovery calls per session across all three).
The loop **moved down a layer**: the agent now finds the right tool, calls it, gets a failure or an empty
result it cannot interpret, and **retries the identical call 5×, 11×, 20×** before giving up or inventing a
success. The three enablers are: **silent success** (`ok:true`, all items errored — S01), **hidden filters**
(an empty read that means "filtered", not "none" — S03), and **un-actionable validation errors** (S02's
`missing book_id`, S03's `entity_id must be a UUID`). Fixing discovery exposed the layer beneath it.
