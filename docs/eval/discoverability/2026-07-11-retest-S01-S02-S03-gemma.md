## S01 — still ❌: the agent proposes entities *before* adopting kinds

`book_kinds=0` after all 4 turns. The agent repeatedly calls `glossary_propose_entities` for
`cultivation_system`, `realm`, `sect`, … on a book with **no kinds adopted**; every item returns
`unknown kind: …`, the tool returns envelope **`ok:true`** anyway (silent success — **9** times), so the
agent gets no failure signal and never corrects. It never successfully adopts the ontology. By turn 3 it
admits *"I haven't actually created anything in your glossary yet."* — honest, but the world is empty.

**Auto-approve proved the confirm gate is NOT the blocker.** The driver now plays the human on suspended
cards (see below). The two `confirm_action` suspends it hit carried an **empty `confirm_token`** and a
hallucinated `glossary_propose_entities.batch` payload — the model invented a confirm for a batch it never
legitimately proposed, so dismissing them is correct. The blocker is upstream: **tool ordering
(adopt-before-propose) + silent success**, which only a workflow rail (WS-5) reliably fixes for a mid-tier
model.

## Harness improvement this cycle: auto-approve suspended calls (warm pass, headless)

The baseline runs exposed that a **propose→confirm / Tier-A approval flow suspends on a card a human
clicks** — and a headless driver can't, so the run hangs (`ok=None`, 0 chars). That silently confounded
S01/S03. The driver now:
- captures `runId` from the `RUN_STARTED` SSE event;
- on a suspend (a tool call with START/ARGS/END but no RESULT), POSTs `/{session}/tool-results` to play
  the human — `tool_approval` cards → `approved_always` ("always allow"); frontend-edit cards
  (`propose_edit`, `glossary_propose_entity_edit`, hallucinated `confirm_action`) → `dismissed` (never
  fabricate a save);
- loops until the turn finishes or a resume cap (`QG_MAX_RESUMES=16`), and records `resumed_tool_calls`
  (excluded from `unresolved_tool_calls`).

Env: `QG_AUTO_APPROVE=1` (default on) · `QG_APPROVE_OUTCOME=approved_always`. This is what makes a WARM
pass actually **complete** confirm-gated flows headlessly; without it, every write scenario stalls on the
first card. Documented in the scenarios README.

## Conclusion — the fixes moved the wall, not removed it

The 118 commits fixed the **tool-execution layer**: `book_id` injection (S02 ✅), a missing-required-args
intercept, C4 error envelopes. But the **agent-sequencing layer** is still exposed: on a mid-tier model,
without a rail the agent proposes entities before adopting kinds (S01), creates instead of triaging (S03),
and — per the S06 baseline — defaults to pure conversation. Two cheap, high-leverage product fixes recur in
every scenario and are still open:

1. **Silent success (P0).** `glossary_propose_entities` returning `ok:true` while every item errors is the
   single enabler of the S01 loop and corrupts any "successful call" metric. A batch whose items all fail
   must return the C4 error envelope. This one fix would let the agent self-correct in S01.
2. **The invisible triage inbox (P0 for S03).** `glossary_list_ai_suggestions` requires the `ai-suggested`
   tag; the vast majority of drafts are untagged, so the inbox shows ~nothing.

And the structural gap the flagship needs: **WS-5 — author the workflow rail** (glossary-bootstrap,
vision-to-book) so a mid-tier model is *sequenced* through adopt→propose→confirm instead of reconstructing
it. That is the unbuilt Track C deliverable, and every servant scenario now points at it.
