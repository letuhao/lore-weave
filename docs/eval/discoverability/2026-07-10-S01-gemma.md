## Three distinct defects

### 1. Wrong tool / wrong order — entities before kinds *(the primary failure)*
In the warm pass the agent went straight to `glossary_propose_entities` on a book with **no kinds adopted**.
Every item came back `{"status":"error","error":"unknown kind: character"}`. It then read the ontology
(`kinds: []` — confirming empty), and **retried the identical `propose_entities` call four more times**,
each returning the same 4/4 item errors. It never called `glossary_adopt_standards` at all. The user asked
for *categories* (kinds); the agent proposed *entities*.

### 2. Silent success — `ok:true` with every item errored *(why it retries)*
`glossary_propose_entities` returns envelope **`ok:true`** while all items inside failed. The agent has no
failure signal, so it cannot learn and loops. This is the repo's known `silent-success-is-a-bug` class,
live on a P0 path. It also means any harness/metric that credits "successful tool calls" will report writes
that never happened — this run is why the driver now detects it and refuses to count such a call.

### 3. Confirm deadlock *(cold pass only)*
`glossary_confirm_action` suspends on the Tier-A approval card. In a headless driver nothing answers it, so
it never resolves — fine, that's the driver. **But the agent's behavior around it is a real finding:** it
re-emitted its previous proposal *verbatim* (byte-identical, 2063 chars), and by the next turn told the
user *"Because a book starts empty, I cannot 'set up' the data until you approve the structure"* — after
the user had already said "go ahead" **twice**. From the user's chair: they approved, and were told to
approve.

## Gap → required capability

1. **Fix silent success (P0, cheap, cross-cutting).** A batch tool whose items all error must not return
   `ok:true`. Return the C4 error envelope (`BUSINESS_RULE` / `VALIDATION`) or at minimum surface
   `isError`. Without a failure signal the agent *cannot* self-correct — this converts one bad call into
   an N-call retry loop, and it will corrupt every future eval that counts "ok" calls as writes.
2. **A `glossary-bootstrap` workflow rail (umbrella Phase 2 / W1).** The correct order —
   list standards → propose categories in plain language → ONE confirm → adopt → write attribute
   descriptions → read back — must be a rail, not something a mid-tier model reconstructs. Left to itself
   gemma picks the wrong tool and skips the adopt step entirely.
3. **`unknown kind` should be actionable, not terminal.** The error names the missing kind; the tool (or
   the rail) should say "no categories exist yet — adopt them first", which is a signal the agent can act on.
4. **Jargon:** "ontology" leaks straight to the user. The rail's `notes_md` must own that vocabulary (F6).

## Note on the two-pass protocol (harness)

The cold pass is **not a product verdict** past its first suspend — a headless driver can't click the
approval card. The driver now flags `unresolved_tool_calls` as a hard signal so a cold run can never be
mistaken for a clean one. The warm pass (pre-seeded `user_tool_approvals`) is the pass to judge
goal-achievement on; it is product-faithful, since "always allow" writes the same rows. **Both passes
failed here, and for the same primary reason** (entities-before-kinds + silent success), so the verdict is
robust to the permission axis.
