## Verdict

**PASS (correctness/safety) · INCONCLUSIVE (token-saving) on this workload.**

- ✅ **The gate mechanism works end-to-end, live, cross-service.** The `entity_presence`
  frame surfaces the decision per turn; decisions are correct — `status_op` gates OUT
  (`no_entity_no_anaphora`), every lore/discovery/continuity turn stays grounded
  (`lore_intent` / `anaphora` bias-to-include). **Zero false-negatives** on the lore
  turns — the bias-to-include design held (a discovery question with no nameable entity,
  "who is the main character", correctly opens via lore-intent — the weakness found and
  fixed during this run).
- ✅ **No answer-correctness regression.** The gated-out `status_op` turn produced a
  coherent reply identical in quality to baseline; lore turns are byte-comparable
  baseline↔candidate.
- ⚠️ **Token saving NOT demonstrated on this book.** Gating out `status_op` saved **0
  tokens** — because `build_context`'s retrieval is ALREADY query-relevance-scaled: a
  no-lore query (`"give me a 3-step plan"`) retrieves little whether the gate fires or
  not, so the light path and the full path are the same size for it. The heavy pulls
  (87–120K) happen only on lore-matching queries, which the gate correctly KEEPS.

## Interpretation (honest)

The intent gate's hypothesized token win is **redundant with the existing FTS relevance
filter** for a *prepend/hybrid* retrieval model (sealed decision #1: prepend for all).
Where the gate genuinely pays off:
1. **The D1 strong-model JIT `pull` mode (deferred):** there retrieval is *unconditionally*
   expensive per turn, so gating no-lore turns is a real cut. The gate is the required
   substrate for that mode.
2. **Retrieval COMPUTE/latency, not tokens:** the light path skips the embedding vector
   search + LLM summarization (`static_mode` takes no embedding/LLM client). Token-neutral
   here, but it avoids those calls — a cost the token frame doesn't capture.
3. **A workload where a no-lore op superficially matches lore** and would trigger a heavy
   pull without the gate. Not present in this small POC book.

## Decision

Ship the gate — it is **correct, safe, flag-gated (default on), fully unit-tested**, and
the `entity_presence` telemetry feeds the Inspector. The **token-saving GATE is deferred**
to a workload that exhibits the cost (a seeded extraction-enabled KG with real passages —
`D-EVAL-BOOK`) or the D1 pull mode. The quality harness is ready to re-run the moment that
workload exists. This is a defer of a *measurement*, not of the build.
