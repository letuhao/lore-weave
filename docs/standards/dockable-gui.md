# Dockable Panel Standard (Writing Studio dock authoring)

**Status:** ACTIVE ¬∑ **Date:** 2026-07-04
**Governs:** authoring or migrating ANY dock panel in the Writing Studio dockview workspace (`frontend/src/features/studio/`) ‚Äî new panels and migrations of an existing route/page alike. Indexed in [`README.md`](./README.md). Consolidates decisions already locked in [`00_OVERVIEW.md`](../specs/2026-07-01-writing-studio/00_OVERVIEW.md) (D1-D30), [`08_studio_state_architecture.md`](../specs/2026-07-01-writing-studio/08_studio_state_architecture.md) (S1-S7, D22 checklist), [`11_dockable_migration.md`](../specs/2026-07-01-writing-studio/11_dockable_migration.md) (the W1 8-point checklist), and [`12_json_document_standard.md`](../specs/2026-07-01-writing-studio/12_json_document_standard.md) (S1-S6) into ONE place, and adds rules DOCK-8/DOCK-9 that those specs did not yet cover.

> **Why a standalone doc now, not just specs.** Three panel waves (11, 12 cycle 1) each re-derived the same checklist from scratch, and this repo's own gap-tracker flagged it: *"Dockable GUI has no standard doc... if dockable-panel authoring keeps recurring, promote a `docs/standards/dockable-gui.md`"* ([README.md Known gaps #5](./README.md#known-gaps--caveats), now resolved by this file). The trigger: scoping the Glossary migration (cycle queue item #2 after chapter editor) surfaced **two anti-patterns no existing panel had hit yet** ‚Äî a page that swaps its ENTIRE subtree by internal view-state (`GlossaryTab`'s `view` switch) and SIX independent hand-rolled `fixed inset-0` viewport modals ‚Äî neither was previously codified as a rule, only implicitly assumed. Both are defined below (DOCK-8, DOCK-9) with the concrete technical evidence, not just style preference.

## Rules

### Carried forward from 08/11/12 (already locked, now cited by ID here)

- **DOCK-1 ¬∑ Catalog entry.** Every panel is one row in [`catalog.ts`](../../frontend/src/features/studio/panels/catalog.ts): `id` = the dockview component id, i18n `titleKey`/`descKey` in the `studio` namespace, `hiddenFromPalette` only for singleton/retargeting panels (e.g. `json-editor`, `skill-editor`). (11 checklist #1)
- **DOCK-2 ¬∑ Thin view, no forks.** A panel component owns no domain state and forks no copy of an existing feature's logic ‚Äî it reuses the existing hook/component AS-IS. Domain state lives in a Tier-4 hoist above dockview (08 D4); the panel just renders it. (11 #2, 08 D3/D4)
- **DOCK-3 ¬∑ Register on mount.** `useRegisterStudioTool` (or the `useStudioPanel` helper wrapping it) ‚Äî palette command + agent-rack visibility. Unregister on unmount. (11 #3, 08 D8, D22 items 2/7)
- **DOCK-4 ¬∑ Bus, not props, for cross-panel signals.** Cross-panel context flows through `StudioContextBus` publish/subscribe ‚Äî never by importing another panel's hook and mutating it, never by prop-drilling through `StudioDock`. (11 #4, 08 S2/D19, D22 item 4/5)
- **DOCK-5 ¬∑ Self-title.** `props.api.setTitle(label)` in an effect ‚Äî an agent- or navigator-triggered open sets the dock tab title before the panel mounts, so the panel must claim its own localized title once mounted. (11 #5)
- **DOCK-6 ¬∑ Agent-openable ‚áí enum + contract lockstep.** A palette-openable panel that should also be agent-openable joins the `ui_open_studio_panel` BE enum; enum ‚äÜ catalog is machine-checked (`panelCatalogContract.test.ts`); regen `contracts/frontend-tools.contract.json`. A panel that takes constructor params instead of a bare id (singleton/retargeting) stays `hiddenFromPalette` and OUT of the enum until a params-bearing open tool exists (12 R3/R4 precedent). (11 #6)
- **DOCK-7 ¬∑ Route-decoupling.** A panel MUST NOT call `useNavigate()`, read `useParams()`, or render `<Link>`. Panel input comes from `props.params` (F1, see [11 ¬ßF1](../specs/2026-07-01-writing-studio/11_dockable_migration.md#f1--openpanel-params)); internal view/tab state is local `useState` seeded from params; outbound cross-panel navigation goes through the **studio link resolver** (F3) ‚Üí `host.openPanel(...)`, never a route hop. A panel that calls `navigate()` is a defect ‚Äî the studio never unmounts itself to satisfy one panel's link. (11 #7, F3 fallback rule W1-2)
- **DOCK-10 ¬∑ Hoist ownership.** If the panel edits in-flight data that must survive a dock-tab close/move/float, that data lives in ONE Tier-4 provider keyed by resource id ‚Äî not scattered `useState` across the panel and its children. A page-local `useState` bag that several unrelated capabilities read through prop-callbacks (the `GlossaryTab` pattern ‚Äî 15 state slots serving 7 different sub-features) is the anti-pattern; each capability gets its own hoist or its own self-contained hook, same as `useMergeCandidates`/`useUnknownReview` already do. (08 D22 items 1/3, S1)
- **DOCK-11 ¬∑ Testing discipline.** Unit tests for the panel + its hook, an E2E for the user-visible behaviour, `/review-impl` at the end of the milestone. Inter-panel links: build the data wiring + unit test now, defer only the cross-panel E2E until both panels exist (LIFO debt stack in `00_OVERVIEW.md`). (00_OVERVIEW Testing discipline, 11 #8, 08 D22 item 9)

### New ‚Äî closing the gap this doc exists for

- **DOCK-8 ¬∑ One capability = one panel; no internal page-replacement.** A panel component must not branch on internal state to swap its ENTIRE rendered subtree for a different, unrelated capability's full-page view (e.g. `view === 'ontology' ? <FullPageA/> : view === 'unknown' ? <FullPageB/> : <FullPageC/>`). Dock composition (tabs, splits, stacks) is dockview's job; a panel that internally re-implements "which page is showing" duplicates the dock and is invisible to the palette/agent-enum/registry for every capability except the default one. **Fix:** each distinct capability becomes its OWN catalog entry (DOCK-1), opened as a sibling dock tab / dock group; a thin "hub" panel may exist only as a **launcher** (buttons that call `host.openPanel(...)` for each capability), never as the capability's actual host. *(Found live in `GlossaryTab.tsx`'s `view` switch across entities/ontology/unknown/ai_suggestions/merge_candidates ‚Äî each of those five is dock-tab-sized on its own, not a fifth of one component.)*
- **DOCK-9 ¬∑ No hand-rolled viewport overlay inside a panel ‚Äî adopt the existing shared dialog.** A panel (or anything it renders) must not construct its own `fixed inset-0` backdrop+dialog pair. Two independent, verifiable reasons, not just taste:
  1. **Z-index has no shared scale for hand-rolled overlays.** The studio's own Command Palette claims `z-[60]` ([`StudioPaletteShell.tsx`](../../frontend/src/features/studio/palette/StudioPaletteShell.tsx)); dockview reserves `--dv-overlay-z-index: 999` for its own drag/drop feedback layer ([`dockview-core` styles](../../frontend/node_modules/dockview-core/dist/styles/dockview.css)). Glossary alone had **six independent hand-rolled instances** at ad hoc `z-40`/`z-50` (`EntityEditorModal` ‚Äî migrated to raw `Dialog.*` in 13_glossary_panels.md A1; `ResolveKindModal` ‚Äî migrated to `FormDialog` in A4, the DOCK-9 adoption precedent; `CreateEntityModal`, `ExtractionWizard`, `GlossaryTranslateWizard`, `BatchTranslateDialog` remain, queued for Phase B) that never coordinated against either number.
  2. **It's a direct [SDK-First](./sdk-first.md) (SDK-1) violation independent of dockable concerns** ‚Äî six copies of the identical `fixed inset-0 z-40 bg-black/60` + `fixed inset-0 z-50 flex items-center justify-center` pair is exactly the "used by ‚â•2, must be one shared primitive" case, not a per-panel style choice.

  **Fix ‚Äî ADOPT, do not rebuild (resolved 2026-07-04):** [`components/shared/FormDialog.tsx`](../../frontend/src/components/shared/FormDialog.tsx) + [`ConfirmDialog.tsx`](../../frontend/src/components/shared/ConfirmDialog.tsx) already exist, are Radix (`@radix-ui/react-dialog`)-based, and are already load-bearing across **21 non-glossary call sites** (knowledge/wiki/world/composition/import/settings). They render via `Dialog.Portal` ‚Üí `ReactDOM.createPortal(..., document.body)`, which sidesteps the transformed-ancestor risk entirely (a `position: fixed` descendant of a CSS-`transform`ed ancestor is clipped to that ancestor, not the viewport ‚Äî dockview applies `transform: translate3d(...)` on several of its internal containers for GPU-accelerated drag/scroll; portaling to `document.body` is what actually makes `fixed` positioning safe here, independent of z-index). Both dialogs sit at `z-50` ‚Äî correctly BELOW the studio palette's `z-[60]` (a global "what do I want to do" affordance should win over a scoped form) and BELOW dockview's `999` (transient, drag-only; app content should never bury the docking engine's own drop-target feedback). **`FormDialog` gained a `size` prop** (`sm`|`md`|`lg`|`xl`|`2xl`|`3xl`, default `lg` = same rendered width as every pre-existing call site) so wizard-width content (glossary's `max-w-2xl`/`max-w-3xl` needs) doesn't require a new one-off component.

  **When FormDialog's title+body+footer template doesn't fit (custom chrome):** a dialog with its own multi-row header/meta-bar/tabs (e.g. `EntityEditorModal` ‚Äî kind badge, status dropdown, save/discard, tab bar) shouldn't be force-fit into FormDialog's fixed layout just to avoid a second pattern. The codebase already has this case solved: use `Dialog.Root`/`Dialog.Portal`/`Dialog.Overlay`/`Dialog.Content` **directly** from `@radix-ui/react-dialog` (same `z-50`/portal-to-body guarantees, same library ‚Äî just without FormDialog's opinionated title block), with an sr-only `Dialog.Title`+`Dialog.Description` for a11y when the visible chrome already carries the name. Precedent: `features/knowledge/components/EntityDetailPanel.tsx` (a drawer variant, `z-50`, custom chrome) predates this standard and already does this; `EntityEditorModal` adopted the same pattern in Phase A (13_glossary_panels.md A1). **Either shape (`FormDialog`/`ConfirmDialog` wrapper, or raw `Dialog.*` primitives) satisfies DOCK-9** ‚Äî what's banned is a raw hand-rolled `<div className="fixed inset-0">`, not Radix itself. `dockablePanelHygiene.test.ts` skips its `fixed`+`inset-0` token check for any panel file that imports `@radix-ui/react-dialog` or `@/components/shared`, since that literal className pair is exactly what `Dialog.Overlay`/`Dialog.Content` legitimately render.

  A panel needing a dialog: simple form ‚Üí `FormDialog`/`ConfirmDialog` from `@/components/shared`; custom multi-part chrome ‚Üí raw `Dialog.*` from `@radix-ui/react-dialog`. Building a THIRD primitive is explicitly NOT the next step; migrating remaining hand-rolled callers (glossary's other five, per-cycle) is.

## Checklist ‚Äî authoring or migrating a panel

- [ ] Catalog entry with i18n keys (DOCK-1)
- [ ] Thin view ‚Äî reuses an existing hook/component, forks nothing (DOCK-2)
- [ ] `useRegisterStudioTool` / `useStudioPanel` on mount, unregister on unmount (DOCK-3)
- [ ] Cross-panel signals via the bus, never foreign-hook mutation or prop-drilling (DOCK-4)
- [ ] Self-titles via `props.api.setTitle` (DOCK-5)
- [ ] If agent-openable: added to the enum + contract regen; if params-based singleton: `hiddenFromPalette` (DOCK-6)
- [ ] Zero `useNavigate`/`useParams`/`<Link>`; params in via F1, links out via F3 (DOCK-7)
- [ ] One capability per panel ‚Äî no internal full-page view-switch; a multi-capability page becomes N catalog entries + a launcher (DOCK-8)
- [ ] Zero hand-rolled `fixed inset-0` ‚Äî uses `FormDialog`/`ConfirmDialog` from `@/components/shared`, or raw `Dialog.*` from `@radix-ui/react-dialog` for custom multi-part chrome (DOCK-9)
- [ ] In-flight edits live in ONE Tier-4 hoist per resource, not scattered page-local state (DOCK-10)
- [ ] Unit tests + E2E + `/review-impl`; cross-panel E2E may defer to the debt stack (DOCK-11)
