# Multilingual / Anti-Language-Bias Standard

**Status:** ACTIVE В· **Date:** 2026-07-04
**Governs:** anything language-dependent вҖ” normalization, segmentation, rule-based input/output checks, stopwords, honorifics, intent, encoding, i18n вҖ” so it works for ALL languages the product handles (source zh/en/ja/ko/вҖҰ, target vi/вҖҰ вҖ” any pair), not hardcoded for one. Indexed in [`README.md`](./README.md).

> **Why.** The product is a multilingual novel platform, but rule-based logic keeps getting written English-first, silently degrading for other languages: the intent classifier is 100% English keywords (so zh/ja/ko/vi queries get no retrieval routing); proper-noun extraction uses `[A-Z][a-z]+` (misses Vietnamese diacritics + Japanese kana + Korean hangul); the honorific-strip list вҖ” which feeds the entity dedup id вҖ” has no native еӨ§дәә/лӢҳ/ж§ҳ; a live event publisher uses `ensure_ascii=True` (2вҖ“3Г— byte bloat on CJK). The repo ALSO has excellent multilingual patterns (a shared NFKC+CJK-fold spine, a per-language pattern registry, a genuinely multi-language injection sanitizer). This standard makes the good patterns the rule and bans the English-first shortcuts.

## Rules

- **ML-1 В· Language is a first-class explicit parameter, never assumed.** Any function doing language-specific rule logic (segmentation, keyword/intent matching, stopwords, honorifics, negation) takes an explicit `lang` (or detects it via the shared `detect_primary_language()`) and dispatches per language вҖ” the [pattern-registry](../../services/knowledge-service/app/extraction/patterns/__init__.py) shape (`get_patterns(lang)`, one module per language). No English-only rule may sit on a path all languages traverse **without a documented degrade-open fallback** for unknown scripts (e.g. `entity_presence`'s `_has_non_ascii()` opens the grounding gate for non-English rather than gating it out).
- **ML-2 В· Unicode-correct normalization only.** User/entity/name/text normalizes through the shared spine (`sdks/python/loreweave_extraction/name_normalize.py`: NFKC вҶ’ Unicode `casefold()` вҶ’ CJK traditionalвҶ’simplified fold, **diacritics preserved** вҖ” vi `mГЎ`вү `ma`), **never bare `.lower()`/`.strip()`**. Any runtime that normalizes names mirrors + parity-tests against the Python/Go SoT (`namenorm.go`'s `TestT2SParityWithSoT` is the model). Honorific/stop lists must include **native-script** forms (дёӯ: еӨ§дәә/е…¬еӯҗ, ko: лӢҳ, ja: ж§ҳ/гҒ•гӮ“, vi: Гҙng/bГ /cГҙ), not just English + romanized.
- **ML-3 В· Script-agnostic segmentation.** No word/sentence splitting that assumes spaces on user prose: **banned** вҖ” `.split(' ')`, `\w` for tokenizing prose, `[A-Z][a-z]+` for proper nouns, word-count by whitespace. Use the CJK-aware helpers (`translation-service/app/workers/chunk_splitter.py`, `sentence_buffer.py`); word/token counts count CJK/kana **per character** (`\p{L}` + explicit CJK ranges вҖ” see the FE `WordCountStatusItem` which avoids `\w` "would shred 'Uyб»ғn' вҶ’ 2 words").
- **ML-4 В· No English-only rule logic on shared paths.** Keyword/intent/stopword lists are per-language or degrade-open. A rule-based **input/output check** (validation, sanitization, moderation) covers every supported language вҖ” the model is the multi-language injection sanitizer (`sdks/python/loreweave_grounding/sanitize.py`: explicit en+zh+ja+ko+vi + zero-width/bidi prenormalize + base64 re-scan). Copy that shape for any new content/validation check; never ship a Latin-only regex as the check.
- **ML-5 В· `ensure_ascii=False` wherever user text is serialized** вҖ” wire, storage, hashing, token measurement. Bare `json.dumps` on a body carrying prose is a defect (the `\uXXXX` inflation tax; [Context Budget Law](../specs/2026-07-03-context-budget-law.md) L3 / [MCP Tool I/O OUT-3](./mcp-tool-io.md)).
- **ML-6 В· Language-tagged data + per-locale i18n.** Detected/source/target language travels with the payload (the reader-language axis `book-service/reader_language.go` вҶ” `knowledge-service/labels/reader_lang.py` + BCP-47 handling is the model). User-facing strings resolve per user locale via a **key+params** model, not baked English ([Notification NOTIF-1](./notification.md)). No default-to-English where the product's primary target differs (translation target, retrieval boost). The UI-locale set + each language's `script`/`dir`(RTL)/`endonym`/`uiLocale`/`translationTarget` is the **Registry SoT** `frontend/src/lib/languages.ts` (`LANGUAGE_REGISTRY`); both the locale switcher and the translation-target picker derive from it, and the FE i18n loader is a **Vite glob** (`i18n/index.ts`) вҖ” adding a locale is a registry row + generated JSON, never per-file imports (the old ~124-manual-import smell is retired).
- **ML-7 В· One authored locale; every other locale is tool-scaffolded + resumable.** Only **`en`** (`frontend/src/i18n/locales/en/`) is hand-authored вҖ” it is the single source of truth for UI copy. Every other UI locale is **GENERATED** by `scripts/i18n_translate.py` (drives a local model via LM Studio; the same verify-by-effect / no-silent-drop discipline the rest of the repo uses): per-key **verify** = JSON valid В· key-set identical to `en` В· every `{{placeholder}}`/`$t()`/`<tag>` preserved В· target-script present; **self-heal** re-prompts the model with the exact defects (вүӨ3 rounds); an unresolved key is written to `<lang>/_FAILED.json` and left as the `en` source вҖ” **never a silent/blank key**; **resumable** вҖ” a namespace whose output already re-verifies is skipped, so a re-run fills gaps and **never re-translates from scratch**. **Never hand-edit a non-`en` locale JSON** (the tool overwrites it and it drifts from `en`). To change copy: edit `en/<ns>.json`, then `python scripts/i18n_translate.py --ns <ns>` regenerates that namespace across all locales. Chunk size is an **attention-quality** limit (the model degrades on long inputs / drops keys), not a context-capacity one.

## Checklist вҖ” any language-dependent code
- [ ] Takes an explicit `lang` / detects it; dispatches per-language or degrades-open (ML-1)
- [ ] Normalizes via the shared spine (NFKC+casefold+CJK), not `.lower()`; native-script honorifics/stopwords (ML-2)
- [ ] No space-assuming split / `[A-Z][a-z]+` / `\w` on prose; CJK counted per-char (ML-3)
- [ ] Rule-based checks cover every supported language (sanitizer shape), no Latin-only regex (ML-4)
- [ ] `ensure_ascii=False` on all user-text serialization (ML-5)
- [ ] Language tagged on the payload; i18n via key+params, no English default; locale set from the Registry SoT `languages.ts` (ML-6)
- [ ] Only `en` authored by hand; other locales generated via `scripts/i18n_translate.py` (resumable, verify+self-heal, no silent blanks); non-`en` JSON never hand-edited (ML-7)
- [ ] A golden fixture per supported language proves it (ja/ko included)
