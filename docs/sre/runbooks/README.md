# SRE Runbook Library

> **Layer:** L7.B (RAID cycle 35) · **Spec:** SR03 §12AF · **LOCKED:** Q-L7B-1

This directory is the **canonical, 3am-readable** runbook library for the LoreWeave
platform. Every SEV0-capable alert MUST link to a runbook here (or to
`generic/escalation-chains.md`). The CI lint `scripts/runbook-verification-lint.sh`
enforces this rule.

## Index

The auto-generated alert→runbook map lives in [`INDEX.md`](INDEX.md). It is
regenerated by `scripts/runbook-index-generator.sh` on every PR touching this
directory. The CI hook ensures `INDEX.md` is always committed current.

## How to use a runbook (at 3am)

1. PagerDuty incident lands on your phone with `runbook_url`.
2. Click the URL — it deep-links to a markdown file in this directory.
3. Read the **TL;DR (30 seconds)** section first.
4. Run the **Diagnostic Commands** (always `--dry-run` first for destructive ops).
5. Follow **Mitigation Steps → Quick mitigation → Full resolution**.
6. If unsure, escalate per [`generic/escalation-chains.md`](generic/escalation-chains.md).

## How to add a new runbook

1. Copy [`TEMPLATE.md`](TEMPLATE.md) into the appropriate subsystem directory.
2. Fill in YAML frontmatter (REQUIRED fields validated by lint).
3. If the runbook is born from a postmortem, set `born_from_incident_id`.
4. Run `bash scripts/runbook-index-generator.sh` to refresh `INDEX.md`.
5. Commit with PR description: "Adds runbook for alert `XYZ`".

## Required frontmatter fields

| Field | Type | Purpose |
|---|---|---|
| `runbook_id` | string | `<subsystem>/<short-name>` (e.g. `auth/token-flow-broken`) |
| `version` | int | Bumped on substantive changes |
| `owner` | string | Team handle (CODEOWNERS) |
| `applies_to_alerts` | list | Prometheus alert names that link here |
| `applies_to_services` | list | Services the runbook touches |
| `last_verified` | date | YYYY-MM-DD — when an SRE last read this end-to-end |
| `last_verified_by` | string | Person who did the read-review |
| `verification_method` | enum | `reading_review` \| `tabletop` \| `chaos_drill` \| `stub` |
| `next_verification_due` | date | Default last_verified + 90 days |
| `severity_hints` | list | Suggested severity per scenario |
| `dry_run_required_for_destructive` | bool | Set true unless `dry_run_not_available: true` |
| `external_access_needed` | list | Required SSO grants (see access-inventory.md) |

## Verification methods

- **`reading_review`** — owner + one other SRE read the runbook end-to-end and
  confirm it matches current code/architecture. Baseline for most runbooks.
- **`tabletop`** — team walks through the scenario verbally (timed). Required
  quarterly for the top-10 most-fired alerts.
- **`chaos_drill`** — actually trigger the condition in staging and follow the
  runbook end-to-end (SR7 chaos integration).
- **`stub`** — placeholder runbook (Q-L7B-1 V1 launch allowance). MUST be
  upgraded before the corresponding feature ships to paid tier. `last_verified`
  is `1970-01-01` to flag overdue.

## Drift detection

Three CI scripts protect the library:

- `scripts/runbook-verification-lint.sh` — block alert without a runbook link;
  flag runbooks past `next_verification_due`.
- `scripts/runbook-drift-check.sh` — detect service renames/removals not
  reflected in runbook `applies_to_services`.
- `scripts/runbook-index-generator.sh` — re-generate INDEX.md on every change.

## External-access docs

`infra/external-access-docs/` mirrors a curated subset of runbooks to an
S3 bucket fronted by CloudFront. This is the fallback when CloudWatch / Vault
/ Grafana / SSO are themselves the incident (SR3 problem 9). Access URL +
break-glass procedure documented in `external-access-docs/README.md`.

## Q-L7B-1 stub allowance

The V1 launch gate (SR3) is "27 runbooks present". Per Q-L7B-1:

> V1 ships ALL 27; stub OK if `last_verified: 1970-01-01` + `verification_method: stub`.

This avoids the trap of either (a) shipping with fewer than 27 (false "ready")
or (b) blocking launch on producing 27 fully-tabletop'd runbooks. Stubs are
honest — they say "this is a placeholder; the next-on-call should read and
verify it before the next page". The lint counts them as PRESENT but flags
them as `OVERDUE_STUB` in dashboards.
