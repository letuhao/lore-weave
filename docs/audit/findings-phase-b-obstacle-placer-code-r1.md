## Finding 1 — BLOCK — the §9 Q3 fallback is unimplemented; a Sea zone gets zero Tree biomes against a mandatory `count_min: 1`
- **Location:** `services/tilemap-service/src/engine/biome_select.rs:81-84` — `select_biomes`. Spec D3, AC-1, AC-4.
- **Problem:** D3 pins the §9 TMP-BIOME-Q3 fallback verbatim — "if a needed `object_type` has zero matching biomes, fall back to *all* library templates of that type". The code does the opposite: `let pool = by_type.get(&rule.object_type)…; if pool.is_empty() { continue; }` silently skips the rule. The `select_biomes` docstring rationalizes the skip with "every §2.3 rule that can be unmatched has `count_min: 0`" — which is **false**: the §2.3 Tree rule is `count_min: 1`, and TMP_005 §6's Water row has no Tree biome (AC-1 states this explicitly). So a `Sea`/`Water` zone selects **0** Tree biomes — a mandatory rule left unsatisfied, contradicting D3 + AC-4, and AC-1's own "a `Sea` zone's mandatory `Tree` rule takes the §9 Q3 fallback".
- **Why it matters:** AC-4 *is* the §9 Q3 acceptance criterion; the build ships without the behaviour it names. The headline Phase-B outcome ("a generated tilemap has mountains, trees, rocks … filling each zone") silently drops trees from every water zone.
- **Test that codifies the bug:** `no_matching_biome_yields_an_empty_selection_no_panic` asserts `sel.is_empty()` for a no-native-biome terrain — it pins the *skip* behaviour, so the missing fallback reads as intended and the AC-4 gate is false-green.
- **Suggested fix:** when `by_type` has no entry for `rule.object_type`, build the pool from all library `BiomeSet`s of that `object_type` (terrain filter dropped). Rewrite the Subterranean test to assert a non-empty fallback selection; add a `Water`-terrain test asserting ≥1 Tree.

## Finding 2 — BLOCK — `fill_is_largest_first` is false-green: the area oracle floods adjacent obstacles into one blob
- **Location:** `obstacle_placer.rs:333-380` — `fill_is_largest_first` + the `occupied_footprint_area` helper. Spec AC-6.
- **Problem:** the test recovers each placement's footprint area with `occupied_footprint_area`, a 4-connected flood-fill of `Occupied` tiles from the anchor. `fill_zone` packs obstacles densely into the `Obstacle` region, so adjacent obstacle footprints are 4-connected — the flood-fill merges them and returns the same whole-blob count for every anchor. `areas` collapses to a near-constant vector, so `w[0] >= w[1]` is trivially true **regardless of fill order**: the test cannot fail even if `fill_zone` sorted ascending.
- **Why it matters:** AC-6's "placed largest-first" is the only ordering guarantee for obstacle fill, and it currently has zero real coverage. The dead `for p in &st.object_placements { let _ = p; }` loop is a tell that the test author knew the area derivation was unfinished.
- **Suggested fix:** have `fill_zone` return each placed obstacle's template `area()` in placement order (the placed grid alone cannot reveal it — footprints merge); assert that sequence is non-increasing and shows a real size gradient. Delete the dead loop and `occupied_footprint_area`.

## Finding 3 — WARN — `erosion_never_seals_a_gap` is tautological and omits the AC-5 fixtures
- **Location:** `obstacle_placer.rs:284-310` — `erosion_never_seals_a_gap`. Spec AC-5.
- **Problem:** the test verifies the erosion result with `would_seal_a_gap` — the *exact* primitive the `erode_zone` gate uses per candidate tile. Because the gate refuses any tile whose removal would seal a gap, `would_seal_a_gap(all-eroded, pre-passable)` is false **by construction**: the test passes for any gate-respecting `erode_zone` and verifies nothing the gate does not already enforce. It also omits the two fixtures AC-5 names — (a) a 2-wide sole-link corridor that must erode to 1-wide, (b) a 2-wide dead-end appendage that must erode away fully — and the "independent reachability oracle" AC-5 mandates.
- **Why it matters:** erosion is ObstaclePlacer's one connectivity-critical step; lesson 9ba274f5 is precisely about not trusting the same primitive to both act and verify. `erode_zone` itself is correct on inspection — this is a coverage gap, hence WARN not BLOCK.
- **Suggested fix:** add an independent 4-connected flood-fill oracle; assert the post-erosion passable region is one component (it starts as one). Add the two AC-5 ASCII-grid fixtures.
