## Finding 1 — WARN — D6 does not scope its `place_and_connect_object → NoSpace` emergency-increment to the *pile* call; a literal reading charges a guard `NoSpace` against the pile loop's failure budget
- **Location:** §3 D6 (interaction with D5).
- **Problem:** D6 increments `emergency` on "`compose_pile → None` ... or `place_and_connect_object → NoSpace`". Two `place_and_connect_object` calls occur per iteration — the **pile** (D4) and, for `value ≥ 2000`, the **guard** (D5). D5 states a guard `NoSpace` is benign (the pile is already placed; an unguarded pile is valid, TMP_006 §5.3). D6 names "`place_and_connect_object → NoSpace`" unqualified — an implementer can charge a guard skip against the `target_count`-capped failure budget.
- **Why it matters:** TMP_006 §3.3's `emergency_loop_counter` increments only on a failed pile composition/placement; a successful pile never touches it. A guard-`NoSpace` increment mildly under-fills any guarded-tier zone on tight guard-adjacency geometry. WARN, not BLOCK — bounded by `target_count`, only the guarded-tier-tight-geometry case.
- **Suggested fix:** D6 scopes the `emergency` increment to the **pile** `place_and_connect_object` call (and `compose_pile`); a guard `NoSpace` is explicitly not a loop failure.

## Finding 2 — WARN — D2 names `MAX_COMPOSE_ATTEMPTS = 100` but the stated loop condition omits the attempt term, dropping TMP_006 §3.3's explicit `attempts < 100` conjunct
- **Location:** §3 D2 (`compose_pile`).
- **Problem:** D2 says the loop is "bounded by `MAX_COMPOSE_ATTEMPTS` (100, §3.3)" but writes the condition as "while `running_sum < tier.min` or `object_count == 0`" — no attempt term. TMP_006 §3.3's loop is `WHILE pile_value < T.min AND attempts < 100`. D2 declares the constant but the written predicate never reads it — an internal inconsistency, and an unstated divergence from the CANDIDATE-LOCK §3.3 pseudocode. (No hang — every pool object has `value > 0`, so `running_sum` strictly increases and the loop self-terminates.)
- **Why it matters:** an unstated divergence from a source-spec pseudocode is the silent-drift class the spec elsewhere flags explicitly (D9's REPLACE-vs-UNION note, DEFERRED #023). WARN — the behaviour is sound either way.
- **Suggested fix:** D2 either writes the condition with the explicit `&& attempts < MAX_COMPOSE_ATTEMPTS` conjunct (matching §3.3), or drops the constant and records the deliberate divergence with the termination argument.

## Finding 3 — WARN — AC-7's `≥ 1 MonsterLair` no-op detector still runs over several seeds with only a *pool-reachable* `min ≥ 2000` tier; an unlucky seed can compose-fail and false-RED a correct build
- **Location:** §4 AC-7; §3 D5/D6.
- **Problem:** the r5 fix made AC-7 a hand-built fixture (geometry pinned). But AC-7 still "runs over several seeds" with the `min ≥ 2000` tier only "pool-reachable" (the AC-5(e) shape — reachable for *some* compositions). With `target_count == 1`, one `compose_pile` decides the high tier; an unlucky seed returns `None`, `emergency` hits `1 == target_count`, zero high-tier piles place, zero `MonsterLair` — and AC-7's `≥ 1 MonsterLair` assertion RED-fails a correct build.
- **Why it matters:** the converse of lesson 1e524dee — a gate that intermittently false-REDs a correct build gets weakened or deleted under pressure, taking the no-op detector with it. r5's WARN-2 fixed the geometry half, not the composition half.
- **Suggested fix:** the AC-7 fixture's `min ≥ 2000` tier gets a wide `[min, max]` (`max` well above the pool's total value, so `compose_pile` reliably succeeds for every seed) and a `density` giving `target_count ≥ 2` (headroom) — so the `≥ 1 MonsterLair` / `≥ 1 Treasure` assertions are robust.
