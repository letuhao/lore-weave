The Phase-C spec is structurally complete and D4's Phase-A API references use the *real* `place_and_connect_object` signature (`state, zone_idx, template, kind, search_area, min_distance, optimize`), not the obsolete TMP_006 §5.2 sketch. But two defects would ship a wrong-behaving placer or a false-green acceptance suite, and a third non-blocking gap exists.

## Finding 1 — BLOCK — D6 emergency-bound diverges from TMP_006 §3.3; it throttles successful placement
- **Location:** §3 decision D6 (and §4 AC-5, which depends on it).
- **Problem:** D6 specifies the per-tier loop as `while placed < target_count && attempts < target_count` and states explicitly that `attempts` increments on **every** iteration. TMP_006 §3.3's pseudocode is `WHILE pile_count < target_count AND emergency_loop_counter < target_count`, and `emergency_loop_counter` increments **only on a failed composition** (`IF pile_value < T.min OR pile_value > T.max: emergency_loop_counter += 1; CONTINUE`). A successful compose+place does NOT touch the counter in the source. D6 inverts this: every successful placement also consumes one `attempt`. Because the bound equals `target_count`, the loop can never iterate more than `target_count` times total — so the realized pile count drops below `target_count` the moment any composition or placement fails (each success *and* each failure burns budget). The source reserves the whole `target_count` budget for *failures* and lets an unbounded run of successes proceed.
- **Why it matters:** direct contradiction of TMP_006 §3.3. It silently under-fills every zone whenever placement is even slightly contended — which TMP_006 §4.4 says is the *normal* case. It also makes AC-5's "per-zone count ≤ summed tier `target_count`" trivially always-true, so AC-5 cannot detect the regression.
- **Suggested fix:** match TMP_006 §3.3 — a separate `emergency` counter incremented **only** when `compose_pile` returns `None` *or* `place_and_connect_object` returns `NoSpace`, never on a successful placement; loop `while placed < target_count && emergency < target_count`. Add an AC clause that a zone with `target_count` reachable, placeable tiers actually reaches `placed == target_count` when no placement fails.

## Finding 2 — BLOCK — AC-5 and AC-6 are single-fixture; `target_count` integer-truncation makes the load-bearing clause vacuous
- **Location:** §4 AC-5 and AC-6 (root cause in D6's `target_count` formula).
- **Problem:** D6 computes `target_count = (tier.density as f32 * zone.assigned_tiles.count_ones() as f32 / 1000.0) as u32`. The `as u32` truncates toward zero. For any zone where `density × tiles < 1000`, `target_count == 0` and the per-tier loop never executes — zero piles, regardless of `Open` area (e.g. `density: 4` on a 200-tile zone → `0.8` → 0). AC-5 asserts the per-zone count is "> 0 for a zone with a reachable tier and ample `Open` area" but pins no zone size; AC-6 asserts "at least one `MonsterLair`" for a ≥2000 tier, again with no zone-size floor. Both run against a single fixture — the lesson-1e524dee trap: a universal property checked on one fixture shape. A build that places treasure in no zone under 250 tiles is indistinguishable from a correct build.
- **Why it matters:** `target_count == 0` for small-but-valid zones is real behaviour the spec never acknowledges (TMP_006 §2: density is "piles per zone-tile-thousand"; sub-thousand-tile zones legitimately exist). The acceptance gate is false-green.
- **Suggested fix:** (a) decide and document small-zone behaviour (truncate-to-0 is defensible — document it); (b) AC-5/AC-6 pin the fixture — hand-built zones of known tile count so `target_count` is exact — plus a boundary case exercising `target_count == 0`; (c) AC-5 asserts the realized count against the *computed* `target_count`, not merely `> 0`.

## Finding 3 — WARN — D5 feeds `choose_guard` a terrain stored as `Option<TerrainKind>` with no `None` policy; guard/pile templates unlisted in §5
- **Location:** §3 decision D5; §5 module design.
- **Problem:** D5 calls `choose_guard(zone_terrain, …)`. `choose_guard`'s real signature is `choose_guard(terrain: TerrainKind, strength: u32)` — a bare `TerrainKind` — but the build state stores `zone_terrain: Vec<Option<TerrainKind>>`, initialised `None`, populated by TerrainPainter. D5 names `terrain_painter` as a dependency, but the type is still `Option` and the spec gives no rule for `None`. Separately, D4/D5 reference `treasure_pile_template()` and a guard "`&unit` template" but the §5 module map lists neither.
- **Why it matters:** non-blocking, but a latent panic-vs-default decision left to the implementer, and an unreviewable construction detail.
- **Suggested fix:** state D5's `None`-terrain policy explicitly (recommend the dependency-edge-guaranteed `.expect(...)`, mirroring `ObstaclePlacer::process` — a `None` is a pipeline-wiring bug, a panic is the correct surfacing); add `treasure_pile_template()` / the guard template to §5 with their footprint (1×1 blocking).
