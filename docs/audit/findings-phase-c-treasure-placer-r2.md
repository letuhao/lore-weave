## Finding 1 — BLOCK — TMP_006 §2's `min == max == 0` filler tier is unhandled: `compose_pile` returns `Some(0,0)`, D4 places a phantom 0-value `Treasure`, AC-2 is falsified
- **Location:** §3 D2 (`compose_pile`), D4 (`process`); §4 AC-2; §2 "Out of scope".
- **Problem:** TMP_006 §2 explicitly permits a filler tier — "Author can specify a tier with `min == max == 0, density == 1` to add a filler 'empty' tier." The Phase-C design never reconciles this. D2's loop stops when `running_sum ≥ tier.min`; with `min == 0` the loop never iterates → `running_sum == 0`, `object_count == 0`. D2's return rule is "`Some` iff the sum is in `[min, max]`" → `0 ∈ [0,0]` → it returns `Some(TreasurePile { value: 0, object_count: 0 })`, not `None`. D4 then places that as a 1×1 blocking `Treasure` via `place_and_connect_object`. A `density:1` filler tier on a ≥1000-tile zone emits a phantom blocking `Treasure` carrying zero objects/value. §2 "Out of scope" lists no exclusion, so the filler tier is in scope by omission.
- **Why it matters:** (a) a phantom blocking `Treasure` ships into `place_tilemap` output / the rebaselined golden / every `object_placements` consumer; (b) AC-2 asserts `compose_pile` returns `Some` with `object_count ≥ 1` — false for the §2-permitted `{0,0,1}` input, so AC-2's universal claim is wrong.
- **Suggested fix:** `compose_pile` returns `None` for any `max == 0` tier (the filler/empty tier composes nothing); the loop guarantees `object_count ≥ 1` for a non-filler tier; the return requires `object_count ≥ 1`. AC-2 adds the filler-tier `None` clause and scopes `object_count ≥ 1` to non-filler tiers.

## Finding 2 — BLOCK — AC-6(a) is a single-shape "≥ 1 `MonsterLair`" check; D5's `Open`-only guard adjacency can `NoSpace`-skip every guard, so the AC false-greens an adjacency bug
- **Location:** §4 AC-6(a) (root cause §3 D5 `guard_search_area`); also AC-5(b).
- **Problem:** D5 builds `guard_search_area` from the `Open` tiles 4-adjacent to the pile footprint and places via `place_and_connect_object`; a `NoSpace` skips the guard. A just-placed 1×1 `Treasure` has ≤4 neighbours; neighbours on `free_paths` are `Walkable` (not `Open`, excluded), the pile tile is `Occupied`. Whether any `MonsterLair` is placed depends entirely on the one hand-built fixture's geometry. AC-6(a) pins the tier and `target_count ≥ 1` but pins nothing about geometry guaranteeing a guardable adjacency, and asserts only "≥ 1 `MonsterLair`" — the lesson-1e524dee trap. An adjacency off-by-one (`neighbors8` vs `neighbors4`, anchor-vs-footprint) passes the favourable fixture; a correct impl can false-red on an unfavourable one. AC-5(b) has the mirror weakness — it asserts high-`max`-first ordering but pins nothing making the *lower* tier's `target_count ≥ 1` and range pool-reachable, so an empty lower tier makes the ordering claim vacuously true and an inverted sort passes.
- **Why it matters:** the guard gate verifies only that guards *can* place in one favourable zone, not that every ≥2000 pile triggers a guard that lands when an adjacent tile is free. A guard-adjacency regression ships green.
- **Suggested fix:** AC-6(a) pins a fixture geometry guaranteeing every placeable pile has ≥ 1 non-sealing `Open` tile 4-adjacent (a wide all-`Open` zone, `free_paths` confined to one edge), and asserts the `MonsterLair` count **equals** the placed ≥2000-pile count, each lair `neighbors4`-adjacent to a `Treasure` footprint. Apply symmetric fixture pinning to AC-5(b).

## Finding 3 — WARN — D4's "sort tiers by `max` descending" pins no tie-break; equal-`max` tiers leave the RNG draw order, hence the map, dependent on the sort-stability choice
- **Location:** §3 D4 ("sort its tiers by `max` descending"); D7 (TMP-A4 determinism).
- **Problem:** an author may declare two tiers with equal `max` but different `min`/`density`. Tier order drives pile-composition order → the per-zone `ChaCha8Rng` draw sequence → the realized map. `slice::sort_by` is stable, `sort_unstable_by` is not; D4 names neither. The rest of the engine pins every tie-break (`object_manager` `(score desc, flat asc)`, `obstacle_placer::fill_zone` biome-id-then-name, `ModificatorRegistry` lexicographic, D1/D2 the fixed-order pool `Vec`) — D4 is the one implicit one.
- **Why it matters:** non-blocking — `ac4_same_seed` runs the same binary twice and the golden self-reproduces, so neither catches it — but a real TMP-A4 hole: a future switch to `sort_unstable` silently changes output of an equal-`max`-tier tilemap with no failing test.
- **Suggested fix:** D4 states the full sort key — e.g. `(max desc, min desc, density desc)` — so the order is total and stability-independent.
