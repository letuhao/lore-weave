## Finding 1 — WARN — the xor-rate test substitutes `count 1..=1` rules; the shipped §2.3 water-feature rate is never verified, and D3/AC-3's "≈0.5" is inconsistent with §2.3's `count 0-1`
- **Location:** `services/tilemap-service/src/engine/biome_select.rs` — `xor_pair_yields_all_three_outcomes_never_both`. Spec D3, AC-3, AC-2.
- **Problem:** the test builds **custom** `Lake`/`Crater` rules with `count_min: 1, count_max: 1` — not `engine_default_biome_selection_rules()`, which ships `Lake 0-1` / `Crater 0-1` per §2.3 (AC-2). With `count 1..=1`, a chosen type always yields ≥1 biome, so the measured `P(neither) = 0.5` and the `(140..=260)` band passes. But the engine that actually ships in `place_tilemap` uses the §2.3 defaults: after the xor coin picks `Some(Lake)` (P ≈ 0.25), the `Lake` rule draws `count = rng.gen_range(0..=1)` — a 50 % chance of **0**. So the *shipped* rates are `P(Lake appears) = P(Crater appears) ≈ 0.125` and `P(no water feature) ≈ 0.75` — not the `neither ≈ 0.5` AC-3 states and D3 pins ("the `xor` pair behaves as one optional `First`-frame slot at ≈50 %"). The round-3 BLOCK fixed the *Crater-unstocked* halving (0.5 → 0.25); the `count 0-1` multiplier is a *second*, independent halving that no test exercises.
- **Why it matters:** AC-3 makes the two-coin rate an acceptance criterion and D3 is a pinned design decision with an explicit probabilistic contract; the one test of that rate measures a non-shipped rule configuration. This is not a shipped *code* bug — `select_biomes` faithfully implements §2.3 (AC-2) and D3's two-coin decision — it is a spec inconsistency: D3's ≈50 % is the *xor decision* rate, but the phrasing reads as a feature-present rate, and the test papers over the gap rather than exposing it. Lake is also Phase-E's river-sink discovery tag (D7/AC-7), so the realized water-feature rate is load-bearing downstream.
- **Suggested fix:** reconcile the spec — state explicitly that D3's ≈0.5 is the *decision* rate and the §2.3 `count 0-1` then independently gates each chosen type (realized ≈0.25 per type). Add a test that runs the xor against `engine_default_biome_selection_rules()` asserting the realized band. Keep the existing coin-isolation test but make its comment say it tests the *xor decision*, not the shipped rate.

## Finding 2 — WARN — AC-8's `ZoneSpec` *with* `biome_selection_rules` is unverified; `BiomeSelectionRules` / `BiomeSelectionRule` have zero serde round-trip coverage
- **Location:** `services/tilemap-service/src/types/template.rs` (the `ZoneSpec` test module) and `src/types/biome.rs` (the `biome.rs` test module). Spec AC-8, D8.
- **Problem:** AC-8 is explicit — "additive schema round-trips: … `ZoneSpec` with/without `biome_selection_rules`". The `without` path is covered, but the **`with`** path is not: every `ZoneSpec` the suite constructs sets `biome_selection_rules: None`. Worse, `BiomeSelectionRules { use_engine_default, rules }` and `BiomeSelectionRule { object_type, count_min, count_max, xor_with, priority }` have **no serde round-trip test anywhere** — `biome.rs` round-trips only `BiomeSet`. `BiomeSelectionRule.xor_with` carries `#[serde(default, skip_serializing_if = "Option::is_none")]`; the `Some`-and-`None` serialization asymmetry that combination produces is exercised by nothing.
- **Why it matters:** D8's `biome_selection_rules` is the author's only hook to override obstacle behaviour per zone (`ObstaclePlacer::process` reads it); a regression to the derives — a field rename, an `xor_with` tag change, a `skip_serializing_if` interaction — would round-trip silently wrong and be caught by no committed test, yet AC-8 names this exact schema. The derives look correct on inspection (hence WARN, not BLOCK), but an AC-named round-trip with zero coverage is a real gap.
- **Suggested fix:** add a `ZoneSpec` round-trip test with `biome_selection_rules: Some(BiomeSelectionRules { use_engine_default: false, rules: vec![ /* one rule with xor_with: Some(..), one with None */ ] })`, and a `BiomeSelectionRules` / `BiomeSelectionRule` serde round-trip in `biome.rs` asserting both `xor_with` cases survive.

## Finding 3 — WARN — the AC-5 erosion corpus only ever builds a single-component pre-erosion passable region; the multi-component case AC-5's oracle exists for is untested
- **Location:** `services/tilemap-service/src/engine/modificators/obstacle_placer.rs` — `erosion_never_seals_a_gap` and the two ASCII fixtures. Spec AC-5.
- **Problem:** `erosion_never_seals_a_gap` builds each random zone with `state(9, 9, &free)` — which assigns **every** tile of the 9×9 grid to one `Wilderness` zone, so the pre-erosion passable region is the full rectangle: always exactly **one** 4-connected component. The two ASCII fixtures likewise carve a single connected region. AC-5 specifies the oracle "`would_seal_a_gap(eroded, pre_erosion_passable)` must be `false`" and the lesson-9ba274f5 rationale is precisely the **multi-component** case — splitting one component while eliminating another leaves the count unchanged. No erosion test ever feeds `erode_zone` a multi-component pre-erosion region.
- **Why it matters:** in production `erode_zone`'s `passable` starts as the zone's entire `assigned_tiles`, and a Penrose nearest-centroid assignment is not guaranteed to hand a zone a 4-connected tile set — so a real zone *can* present `erode_zone` a multi-component passable region. `would_seal_a_gap` itself is correct for that case (`would_seal_a_gap_matches_the_independent_reachability_oracle`, 600 multi-component pairs), so this is a coverage gap, not a shipped bug — hence WARN — but the *erosion* path's behaviour on a multi-component zone is asserted by nothing. Lesson 1e524dee: a single-topology corpus for a universal property is falsely confident.
- **Suggested fix:** add a multi-component erosion test — carve a wall band so the zone holds two disjoint Open pockets — and assert `would_seal_a_gap(eroded, pre_passable) == false` (the oracle AC-5 names) plus a per-component reachability check.
