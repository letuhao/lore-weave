## Finding 1 — BLOCK — AC-10(b), the regression gate for D9's non-transitivity rule, is vacuous; a transitive-chase resolver (which D9 relies on for cycle-safety) ships green
- **Location:** §4 AC-10(b); §3 D9.
- **Problem:** D9's non-transitivity is load-bearing twice — (i) it fixes effective-tier resolution (`X` inheriting from `Y` takes `Y`'s *literal* `treasure_tiers`, not what `Y` itself inherits), and (ii) D9 explicitly states it "makes an inheritance cycle structurally impossible". AC-10(b) — "the resolution is non-transitive — a `Y` that itself declares `inherit_treasure_from` is not chased" — pins **no fixture** and states **no assertion**. Unlike AC-10(a) (r3-tightened to a disjoint-own-tiers fixture with an explicit value-range assertion), AC-10(b) describes no construction that *fails* when non-transitivity is violated. A transitive-chase resolver (`loop { resolve }` instead of one hop) is observably wrong only against a fixture where `Y` declares `inherit_treasure_from: Some(Z)` with `Z`'s tiers pool-reachable and disjoint from `Y`'s own.
- **Why it matters:** lesson 1e524dee — a non-exercising test for a universal property is falsely green. The transitive-chase bug is *worse* than the prefer-own bug r3 caught: on an authoring cycle `X → Y → X` a transitive resolver never terminates (infinite loop / stack overflow), aborting `place_tilemap`. AC-10(b) cannot catch the wrong-tier output on a chain *or* the hang on a cycle — both D9 guarantees can regress green.
- **Suggested fix:** AC-10(b) pins a three-zone chain — `X → Y`, `Y → Z`, `Y`'s own `treasure_tiers` non-empty / pool-reachable / disjoint from `Z`'s — and asserts every placed `Treasure` in `X` lands in `Y`'s own range and none in `Z`'s; plus a cycle `X → Y → X` fixture asserting `place_tilemap` terminates deterministically with no panic / no hang.

## Finding 2 — WARN — AC-7 asserts ≥ 1 `MonsterLair` over per-seed-varying `place_zones` geometry not pinned to guarantee a guard-placeable adjacency — a false-RED risk
- **Location:** §4 AC-7; §3 D4/D5/D6.
- **Problem:** the r4 fix removed AC-7's empty-`treasure_tiers` vacuity and added `object_placements` ≥ 1 `Treasure` **and** ≥ 1 `MonsterLair`. But AC-7 verifies the full `place_tilemap` pipeline "over several seeds" — `place_zones`-generated geometry that varies per seed — and pins no geometry. A `MonsterLair` lands only if (a) the `min ≥ 2000` tier has `target_count ≥ 1` for some zone (`density × tiles / 1000`, `as u32`-truncated — not guaranteed by "carries the tier") and (b) the guard's `place_and_connect_object` finds a non-sealing `Open` tile 4-adjacent to the pile (D5 — `NoSpace` legitimately skips it). For some seed neither may hold → zero `MonsterLair` → a **false-RED on a correct build**.
- **Why it matters:** non-blocking — the real connectivity gate is the flood-fill — but an AC that can fail a correct build gets weakened/deleted under pressure, taking the no-op detector with it.
- **Suggested fix:** give AC-7 the AC-6 treatment — a hand-built fixture whose geometry guarantees `target_count ≥ 1` for the `min ≥ 2000` tier and a guard-placeable adjacency — or drop the `≥ 1 MonsterLair` count to a conditional and keep `≥ 1 Treasure` as the no-op detector.

## Finding 3 — WARN — D5's `guard_search_area` is the first custom-subset `search_area`; D5 does not state it must be grid-dimensioned, which `place_and_connect_object`'s `debug_assert_eq!` requires
- **Location:** §3 D5; `object_manager.rs` `place_and_connect_object`.
- **Problem:** `place_and_connect_object` opens with `debug_assert_eq!((search_area.width(), search_area.height()), (grid.width, grid.height), …)` and its doc warns the `fits ⇒ in-bounds` `expect` is sound only when the two agree. Every existing caller / D4's treasure call builds `search_area` from `zone_area_open` / `zone_passable` (grid-sized). D5's `guard_search_area` ("the `Open` tiles 4-adjacent to the pile footprint") is the **first custom-subset** `search_area`; D5 does not state it must be grid-dimensioned. An implementer building it as a bounding-box-sized mask trips the `debug_assert` in test builds and risks a mis-projection in `--release`.
- **Why it matters:** non-blocking — `cargo test` is debug, so the `debug_assert` surfaces it — but an unreviewable construction detail in a new pattern the spec introduces.
- **Suggested fix:** D5 states `guard_search_area` is a **grid-dimensioned** `TileMask` — build it as `zone_area_open(zone_idx)` intersected with the placed pile footprint's 4-neighbourhood (stays grid-sized, mirrors the `zone_area_open` precedent).
