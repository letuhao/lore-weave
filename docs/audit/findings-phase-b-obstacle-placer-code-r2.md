## Finding 1 — BLOCK — `ac10_walkable_skeletons_stay_connected_end_to_end` is false-green: AC-10 mandates the passable region, the test checks the inert `Walkable` skeleton with the forbidden oracle
- **Location:** `services/tilemap-service/tests/determinism.rs:181-202` — `ac10_walkable_skeletons_stay_connected_end_to_end`. Spec AC-10.
- **Problem:** AC-10 is explicit — "after `place_tilemap` no zone's **passable region** (`Walkable ∪ Open`) has been split — verified with the **split-OR-eliminate oracle** (`would_seal_a_gap` ... or an independent all-pairs reachability oracle), **not** a `connected_components` count delta." The committed test does the two things AC-10 forbids: it checks `connected_components(&z.free_paths) == 1` — the **`Walkable` skeleton only** (`free_paths`), not `Walkable ∪ Open` — using **`connected_components`**, the count oracle AC-10 names as wrong. ObstaclePlacer's erosion only turns `Open → Obstacle` and fill only `Obstacle → Occupied`; **it removes no `Walkable` tile**, so `z.free_paths` is byte-identical before and after ObstaclePlacer. The assertion re-checks a property `place_zones`/`fractalize` already guaranteed and that Phase B physically cannot break — it would pass identically with `ObstaclePlacer` deleted from the pipeline. This is lesson 9ba274f5's "wrong region" (a connectivity check on the Walkable skeleton is inert when the placer acts on `Open` tiles) plus lesson d29dbaba's vacuous assertion in one test.
- **Why it matters:** AC-10 is the *only* criterion exercising connectivity through the **integrated** `place_tilemap` pipeline (multi-zone, real `place_zones` output, erosion fed a real per-zone passable region). `erosion_never_seals_a_gap` tests `erode_zone` in isolation on synthetic single-full-grid zones; it cannot catch an integration regression — a pipeline-ordering change, or a future placer that strands an `Open` courtyard. AC-10 — the criterion the spec says "the same correct oracle is reused, live, by TreasurePlacer in Phase C" — has **zero real coverage**.
- **Suggested fix:** the test cites a real constraint — `TileState` is build-internal, not on `TilemapView`. Resolve it with a vehicle that *can* see `TileState`: run the multi-zone fixture through `place_zones` → `TilemapBuildState::from_zones` → a directly-invoked `ObstaclePlacer::process`, then for every non-Forbidden zone assert the post-pipeline `state.zone_passable(idx)` is one component under an independent flood-fill (or `would_seal_a_gap` against the per-zone total of tiles removed from the pre-pipeline passable region). Keep the `ac10_*` name only on a test that checks `Walkable ∪ Open`.

## Finding 2 — WARN — `fill_is_largest_first` re-derives `fill_zone`'s own sort postcondition; it cannot fail for any front-to-back placement loop
- **Location:** `services/tilemap-service/src/engine/modificators/obstacle_placer.rs` — `fill_is_largest_first`; `placed_areas` built inside `fill_zone`. Spec AC-6.
- **Problem:** `fill_zone` sorts `items` by `b.area.cmp(&a.area)` (area **descending**), then iterates `for item in &items` front-to-back and does `placed_areas.push(item.area)` for every placed item. `placed_areas` is therefore, by construction, a forward subsequence of an area-descending-sorted `Vec` — **non-increasing for any placement logic that walks `items` in order**. The test's `for w in areas.windows(2) { assert!(w[0] >= w[1]) }` re-asserts that structural postcondition; it cannot fail unless `items.sort_by` itself produces a non-descending order. It *does* catch the most plausible regression — an inverted comparator — and the `areas.min() < areas.max()` guard keeps the corpus non-degenerate, so this is not fully vacuous (hence WARN). But AC-6's actual artifact is the placement record: "a placed object's footprint `area()` ≥ every later-placed **object's**" — the ordering of `state.object_placements`. The test verifies the side-channel `placed_areas`, never `object_placements`.
- **Why it matters:** "placed largest-first" is the only ordering guarantee AC-6 gives obstacle fill; the check meant to pin it instead pins the sort comparator's own output — sharing the actor's structure (sort → forward iterate → report in that order), the pattern lessons 9ba274f5 / d29dbaba warn against. The round-1 F2 fix removed the merged-blob oracle but replaced it with a self-derived sequence rather than an independent reconstruction.
- **Suggested fix:** reconstruct each placed obstacle's footprint area from `object_placements` independently of `fill_zone`'s loop (look up the placement's biome templates / have `fill_zone` record the chosen template identity per placement), then assert *that* sequence — in `object_placements` order — is non-increasing and shows the size gradient. Pin a non-tautological anchor (e.g. the first placed area equals the largest template area in the selection, computed independently from the library).

## Finding 3 — WARN — `fill_places_obstacles_only_in_the_obstacle_region` inspects only the anchor tile; the obstacle-placer test module never exercises D5's inter-zone boundary fade
- **Location:** `services/tilemap-service/src/engine/modificators/obstacle_placer.rs` — `fill_places_obstacles_only_in_the_obstacle_region`; helpers `state` / `carved_state`. Spec AC-6, D5.
- **Problem:** AC-6's core invariant is "every placed obstacle's **footprint** lies entirely within the zone's `Obstacle` region (no footprint tile overlaps `Open`/`Walkable`/`Occupied`)." The test inspects only `p.anchor`: `assert!(obstacle_before.get(p.anchor))` + `assert_eq!(st.tile_state_at(p.anchor), Occupied)` — one tile of a 2–9-tile footprint. `footprint_fits` is correct on inspection, so this is a coverage gap, not a live bug — but a regression admitting a footprint whose *non-anchor* cells land on `Open`/`Walkable` passes untouched. Separately, every test in this module builds its zone via `state()` / `carved_state()`, both of which make **one zone covering the whole grid**. So D5's "a neighbour counts as wall if it is **not a member of this zone's `assigned_tiles`** ... a tile of a neighbouring zone — TMP_005 §4.3's zone-boundary fade" — the `!zone_assigned.get(n)` clause of `is_wall_adjacent` — is **only ever triggered by the grid border**, never by a real inter-zone boundary, in the entire unit-test module. The one multi-zone test is `ac10_*` (Finding 1), which is inert.
- **Why it matters:** footprint-containment and boundary-fade are the load-bearing behaviours of D6 fill and D5 erosion; both are verified only indirectly or not at all. A `footprint_fits` off-by-one or a dropped `!zone_assigned.get(n)` clause would be caught by no committed test.
- **Suggested fix:** in `fill_places_obstacles_only_in_the_obstacle_region`, assert **every** `Occupied` tile (the union of all placed footprints) was in `obstacle_before` — not just the anchors. Add a test with a genuine two-zone build state (two zones sharing an interior boundary, neither touching that boundary's grid edge) that exercises the `!zone_assigned.get(n)` boundary-fade clause.
