## Finding 1 ‚Äî BLOCK ‚Äî AC-5(a), the regression gate for the r1 D6-bound fix, is vacuous
- **Location:** ¬ß4 AC-5(a); ¬ß3 D6.
- **Problem:** AC-5(a) specifies the scenario as "places **exactly** `target_count` `Treasure` objects **when no composition/placement failure occurs**", and claims it "pins D6's corrected bound; the old per-iteration counter would stop short". That parenthetical is false. The rejected r1 design ‚Äî a single counter incremented on every iteration, bound `attempts < target_count` ‚Äî under-fills *only when a failure burns a slot*. In a run with **zero** failures, every iteration is a success, the per-iteration counter and `placed` rise in lockstep, and the rejected loop *also* reaches `placed == target_count`. The corrected and rejected bounds produce **identical** output for the exact scenario AC-5(a) specifies ‚Äî they diverge only on the failure path, which AC-5(a) deliberately excludes.
- **Why it matters:** AC-5(a) is the only acceptance clause meant to pin the r1 fix; as written it cannot fail when the property is violated (lesson 1e524dee). The r1 BLOCK can silently regress.
- **Suggested fix:** AC-5 adds a clause with a fixture that forces a mid-run failure (a tight tier range the pool reaches only for some compositions ‚Üí ‚â• 1 `compose_pile ‚Üí None`) with ample placement room, asserts the realized count still reaches `target_count`, and independently confirms ‚â• 1 failure occurred (replay `compose_pile`, a pure function). Reword AC-5(a)'s parenthetical ‚Äî it is the clean-zone baseline, not the D6-fix gate.

## Finding 3 ‚Äî WARN ‚Äî D7 justifies determinism by claiming per-zone placement is "order-independent", but `place_and_connect_object` reads the map-wide distance oracle ‚Äî placement is order-*dependent*
- **Location:** ¬ß3 D7; ¬ß2 item 7.
- **Problem:** D7 states "all RNG (composition sampling) draws from it, so per-zone placement is order-independent and the TMP-A4 axiom holds." Only *composition sampling* uses the per-zone sub-seeded RNG. The *placement* step, `place_and_connect_object`, scores anchors against `state.nearest_object_distance` ‚Äî a **map-wide** oracle that every prior placement in any zone mutates (the D10 full-grid refresh; the `nearest_object_distance_oracle_spans_zone_boundaries` test). So the anchor chosen in one zone depends on what earlier zones placed: per-zone *placement* is order-dependent. `place_tilemap` is still deterministic ‚Äî but because zones are processed in a fixed order over a single-threaded pipeline, not because placement is order-free.
- **Why it matters:** non-blocking today, but TMP-TR-Q5 / ¬ß4.2 anticipate per-zone *parallel* placement; D7's "order-independent" phrasing is exactly the rationale a future author would cite to parallelize `TreasurePlacer`, which would race on the shared mutable `nearest_object_distance` oracle and break TMP-A4.
- **Suggested fix:** D7 corrects the rationale ‚Äî `place_tilemap` is deterministic because (a) composition RNG is per-zone sub-seeded **and** (b) zones are processed in a fixed deterministic order, single-threaded ‚Äî and states explicitly that placement is *not* order-independent, so per-zone parallelization is blocked until the oracle is made placement-order-insensitive.
