## Finding 1 — BLOCK — `Lake xor Crater` under-delivers water features on the production library; the AC-3 xor test is false-green via library substitution
- **Location:** `services/tilemap-service/src/engine/biome_library.rs:94-117` (`Crater` never stocked) + `src/engine/biome_select.rs` (xor + Q3-fallback loop) + `src/engine/biome_select.rs` (`xor_test_library` / `xor_pair_yields_all_three_outcomes_never_both`). Spec D3, AC-3.
- **Problem:** `engine_biome_library()` stocks only `Mountain / Tree / Rock / Plant / Lake` per terrain — **`Crater` is never stocked anywhere**. The §2.3 rules ship `Lake 0-1 xor Crater` and `Crater 0-1 xor Lake`. The D3 two-coin decision lands on `Some(Lake)` / `Some(Crater)` / `None` at ≈0.25/0.25/0.5. When it lands on **`Some(Crater)`**: the `Lake` rule already `continue`d (`decided != Some(Lake)`), and the `Crater` rule then hits the Q3-fallback `None` branch whose filter `b.object_type == Crater` is **empty for the whole library** → `pool.is_empty()` → `continue`. Net: a "Crater" coin yields **nothing AND suppresses Lake**. So on the real library the xor's "feature" slot is filled with P ≈ 0.25, not the ≈0.5 D3 explicitly pins ("the `xor` pair behaves as one optional `First`-frame slot at ≈50 %"). `xor_pair_yields_all_three_outcomes_never_both` does not catch this: it builds `xor_test_library()`, which **deliberately stocks both `grass_lake` and `grass_crater`**, and asserts `(140..=260).contains(&neither)` against that synthetic library. Run the same assertion against `engine_biome_library()` for any land terrain and `crater_only` is 0 — FAIL.
- **Why it matters:** D3 is a pinned design decision with an explicit probabilistic contract; AC-3 makes the two-coin rate an acceptance criterion. Lake is also the Phase-E river-sink discovery tag (D7/AC-7). Water features ship at half the designed rate, and the only test of the rate is green only because it substitutes a library without the gap — lesson 1e524dee (passes on a hand-picked input, fails on the real one) plus lesson d29dbaba (vacuous-by-substitution).
- **Suggested fix:** decide intent. (a) Stock a `Crater` biome per land terrain (the §6 table lists Crater cells) so the two-coin model delivers ≈0.5 — and assert the rate against `engine_biome_library()`, not a synthetic one; or (b) if Crater is a conscious V1+30d cut, change the `Lake`/`Crater` default rules so the surviving type carries the full optional-slot probability (drop `Crater`, make `Lake` a plain `0-1 First`), and update D3/AC-3. Either way add a test exercising the xor against the production library.

## Finding 2 — WARN — `ac10_place_tilemap_pipeline_never_splits_a_zone_passable_region` runs a single seed for a universal property
- **Location:** `services/tilemap-service/src/engine/mod.rs` — `ac10_place_tilemap_pipeline_never_splits_a_zone_passable_region`, drives the fixture at `seed = TilemapSeed(0xC0FFEE)` only. Spec AC-10.
- **Problem:** the r2 fix correctly relocated AC-10 onto the real pipeline, checks `Walkable ∪ Open`, and uses an independent flood-fill — that part is sound. But AC-10 is a **universal** property and the test exercises exactly **one seed**. `place_zones` is seed-driven (FR → Penrose → fractalize); a different seed yields different zone shapes and erosion patterns. A regression splitting a passable region only for certain geometries (a concave zone, a narrow Penrose isthmus) passes untouched on `0xC0FFEE`. Lesson 1e524dee.
- **Why it matters:** AC-10 is the *only* integrated-pipeline connectivity check; the spec says its oracle is reused live by Phase-C TreasurePlacer. Erosion is well-covered elsewhere (`erosion_never_seals_a_gap`, 200 zones; gate correct on inspection) — hence WARN — but the *integration* path is pinned to one geometry.
- **Suggested fix:** loop the AC-10 body over a fixed seed list (e.g. `[0xC0FFEE, 1, 2, 7, 99]`) or several fixtures with differing zone counts/roles — reproducible without being single-input.

## Finding 3 — WARN — AC-7's Lake-object tagging is unverified; the test named for it asserts only `Mountain`
- **Location:** `services/tilemap-service/src/engine/modificators/obstacle_placer.rs` — `mountain_and_lake_biomes_tag_their_object_type`. Spec AC-7.
- **Problem:** AC-7 is explicit — "a `Mountain` ... biome yields `Some(Mountain)`, a `Lake` biome `Some(Lake)`." The test named `mountain_and_lake_biomes_tag_their_object_type` asserts only `kinds.contains(&Mountain)`; there is **no `Lake` assertion** here or anywhere (`fill_places_obstacles_only_in_the_obstacle_region` checks `is_some()`, not the variant). The Lake half of AC-7 has zero coverage while the test name advertises that it does. Lake was likely skipped because the §2.3 `Lake` rule is `0-1` behind the xor (non-deterministic per seed) — but that is a reason to build a deterministic Lake fixture, not to leave the clause untested under a misleading name. Lesson d29dbaba (a test whose name implies coverage it does not provide).
- **Why it matters:** `Some(Lake)` is Phase-E RiverPlacer's river-sink discovery handle (D7) — the exact reason AC-7 names Lake. A future regression to Lake tagging is caught by nothing. Production code is correct on inspection (the tag is `selection.by_type`'s key, same path as Mountain) — hence WARN.
- **Suggested fix:** add a deterministic Lake assertion (drive `fill_zone` with a hand-built `BiomeSelection` containing a `Lake` biome and assert `biome_object_type == Some(Lake)`), or rename the test.
