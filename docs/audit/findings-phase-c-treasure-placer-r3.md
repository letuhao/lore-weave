## Finding 1 ‚Äî BLOCK ‚Äî AC-10 pins no fixture where zone X has distinct, non-empty own tiers; D9's "ignoring its own" cannot fail when violated
- **Location:** ¬ß4 AC-10; ¬ß3 D9.
- **Problem:** D9's core rule is REPLACE ‚Äî a zone `X` with `inherit_treasure_from: Some(Y)` takes `Y`'s `treasure_tiers` and `X`'s own is ignored. AC-10 asserts "places treasure per zone `Y`'s `treasure_tiers`, ignoring its own" but pins nothing about `X`'s *own* tiers. If the fixture's `X` has an empty own `treasure_tiers` (the natural shape), the assertion is vacuous ‚Äî correct REPLACE, a prefer-own bug (`if X.tiers.is_empty() { Y } else { X }`), and a UNION bug (`X ++ Y`) all produce identical output. Only a fixture where `X` carries a non-empty own tier set whose `[min,max]` is pool-reachable and disjoint from `Y`'s distinguishes them.
- **Why it matters:** "ignoring its own" is the most error-prone half of D9 (prefer-own / union is the natural mistake), and the gate meant to catch it is false-green ‚Äî lesson 1e524dee. A resolver that quietly prefers `X`'s own tiers ships green.
- **Suggested fix:** AC-10 pins a fixture where `X` declares `inherit_treasure_from: Some(Y)` **and** a non-empty own `treasure_tiers` with a pool-reachable range disjoint from `Y`'s, and asserts every placed `Treasure` in `X` lands in `Y`'s range and none in `X`'s own.

## Finding 2 ‚Äî BLOCK ‚Äî D10's call-site census is wrong: adding `value` to `TilemapObjectPlacement` breaks the `obstacle_placer.rs` struct literal, unlisted in ¬ß5
- **Location:** ¬ß3 D10; ¬ß5 module-design table.
- **Problem:** D10 adds `value: Option<u32>` to `TilemapObjectPlacement`. `#[serde(default)]` affects deserialization only ‚Äî a Rust struct literal must name every field, and `TilemapObjectPlacement` derives no `Default`. D10's census ‚Äî "the only other call sites are Phase A's `object_manager.rs` tests" ‚Äî counts callers of the *function* `place_and_connect_object`, but adding a *struct field* breaks every struct-literal site. `ObstaclePlacer::fill_zone` (`obstacle_placer.rs`) pushes a `TilemapObjectPlacement { kind, anchor, canon_ref, biome_object_type }` literal **directly** ‚Äî Phase B D6 deliberately bypasses `place_and_connect_object`. That literal fails to compile the moment `value` is added; D10's census omits it and ¬ß5 does not list `obstacle_placer.rs` as MOD.
- **Why it matters:** a PLAN built from ¬ß5 + D10's census misses `obstacle_placer.rs`; the omission surfaces only as a `cargo build` failure. The spec makes a factual claim that is false.
- **Suggested fix:** D10 distinguishes `place_and_connect_object` *callers* (get the new param) from `TilemapObjectPlacement` *struct literals* (get the new field ‚Äî `object_manager.rs` body, `object.rs` tests, **and `obstacle_placer.rs::fill_zone`**, each `value: None`); add `engine/modificators/obstacle_placer.rs` to ¬ß5 as MOD.

## Finding 3 ‚Äî WARN ‚Äî D9 adopts REPLACE against TMP-TR-Q3's UNION-with-override, and never specifies the "both fields set" case
- **Location:** ¬ß3 D9; TMP_006 ¬ß8.5 TMP-TR-Q3.
- **Problem:** TMP-TR-Q3's locked decision is UNION-with-override ("full object pool copy ‚Ä¶ author overrides specific entries ‚Ä¶ more-specific zone-level wins"). D9 specifies REPLACE ("`X`'s own `treasure_tiers` is ignored") and never says what a zone declaring **both** `inherit_treasure_from` and a non-empty own `treasure_tiers` gets ‚Äî under D9 the own tiers become silent dead config. REPLACE is *defensible* ‚Äî `treasure_tiers` is an unkeyed `Vec`, so per-entry override has no clean meaning ‚Äî hence WARN, not BLOCK.
- **Why it matters:** an unstated divergence from a CANDIDATE-LOCK source decision is the silent drift ¬ß2 pulled D9 in to avoid; an author writing both fields gets a no-op with no diagnostic.
- **Suggested fix:** D9 states explicitly it chooses REPLACE over UNION-with-override for the unkeyed `treasure_tiers` `Vec` (with the unkeyed-Vec justification), documents that both-set makes the own tiers inert, and notes the divergence in DEFERRED #023 so the ¬ß3.2 filtering layer reconciles it.
