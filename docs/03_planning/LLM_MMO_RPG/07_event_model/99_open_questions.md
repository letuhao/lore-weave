## Phase 1 residuals (to resolve in Phase 2-5)

### EVT-Q1 — Producer-rule binding for multi-role services

**What:** [EVT-A4](02_invariants.md#evt-a4--producer-category-binding) requires each EVT-T* to have authorized producers via JWT `produce: [...]` claims. Some services produce many categories (world-service produces PlayerTurn, NPCTurn, AggregateMutation, CalibrationEvent). Open: do those land as one big claim list per service, or as per-operation capability tokens (different JWT per request)?

**Why deferred:** Implementation detail of capability JWT; needs Phase 2 EVT-P* design to decide.

**Resolves in:** Phase 2 [`04_producer_rules.md`](04_producer_rules.md).

**Default for design:** assume one JWT per service with a list claim until decided.

### EVT-Q2 — Validator pipeline order for AdminAction

**What:** [EVT-A5](02_invariants.md#evt-a5--validator-pipeline-runs-in-fixed-order-no-skips) locks fixed order. AdminAction has a different validator subset (capability + S5 dual-actor + audit; no A6 / no canon-drift) — but the order WITHIN that subset still needs locking. Specifically: does world-rule lint run BEFORE or AFTER S5 dual-actor confirmation?

**Why deferred:** Phase 3 EVT-V* design will surface tradeoffs (running lint first reveals overrides the operator may want to dispel; running dual-actor first authenticates before running anything).

**Resolves in:** Phase 3 [`05_validator_pipeline.md`](05_validator_pipeline.md).

### EVT-Q3 — LLMProposal bus retention vs proposal_id reuse

**What:** [EVT-T6 LLMProposal](03_event_taxonomy.md#evt-t6--llmproposal) has 3 terminal states (Validated / Rejected / Expired). Default expiry 60s. Open: how long does the bus retain Validated proposals for diagnostics? How long does it retain proposal_id for dedup window? Both impact bus retention sizing.

**Why deferred:** Phase 3 LLM bus protocol needs to size retention against expected proposal volume (1-10/sec/active-session × ~hundreds of sessions V1).

**Resolves in:** Phase 3 [`07_llm_proposal_bus.md`](07_llm_proposal_bus.md).

### EVT-Q4 — CalibrationEvent producer ordering

**What:** [EVT-T7 CalibrationEvent](03_event_taxonomy.md#evt-t7--calibrationevent) producer is world-service. When a FictionClockAdvance crosses N date boundaries (e.g., 23 days = 23 day_passes + 1 month_passes), open: does world-service emit them all atomically in the SAME transactional cluster as the FictionClockAdvance, or as immediate follow-ups in separate transactions?

**Why deferred:** DP-K5 has `t3_write_multi` for atomic clusters but NOT for T2; CalibrationEvent should be T2 per its Channel-event semantics. Phase 4 will pick a pattern.

**Resolves in:** Phase 4 [`08_scheduled_events.md`](08_scheduled_events.md) + [`09_causal_references.md`](09_causal_references.md).

**Default for design:** assume immediate follow-ups in chronological order within the same channel writer's epoch.

### EVT-Q5 — Causal-ref multi-parent semantics

**What:** [EVT-A6](02_invariants.md#evt-a6--causal-references-are-typed-single-reality-gap-free) defines causal-refs as `Vec<CausalRef>`. Open: when is multi-parent semantically meaningful? BubbleUpEvent obviously aggregates multi-source. Other categories (NPCTurn) likely single-parent. Open: should we restrict multi-parent to specific categories at validator time, or allow uniformly?

**Why deferred:** Phase 4 causal-ref design will surface tradeoffs (uniform = simpler enforcement; restricted = catches modeling bugs at design review).

**Resolves in:** Phase 4 [`09_causal_references.md`](09_causal_references.md).

---

### EVT-Q6 — Replay filter for flavor narration

**What:** [EVT-A8](02_invariants.md#evt-a8--flavor-narration-is-not-events) excludes flavor narration from canonical event log. But session catch-up replay (Phase 4 EVT-L3) needs to deliver narration text to UI for a reconnecting client. Open: is flavor stored in audit log + replayed from there, or regenerated by UI hitting a narration endpoint with `(turn_id, prompt_context)` and re-prompting LLM?

**Why deferred:** Tradeoff between cost (re-prompt = LLM call) and consistency (audit-log replay = same text). Per SPIKE_01 obs#16, the user may regenerate flavor narrative for varied retells; that argues for re-prompt. But re-prompt = $$$ on reconnect storms.

**Resolves in:** Phase 4 [`10_replay_semantics.md`](10_replay_semantics.md).

**Default for design:** assume audit-log replay for V1, re-prompt as feature flag for V1+30d.

### EVT-Q7 — Schema version field placement per envelope vs per payload

**What:** I14 + R3 mandates additive-first schema evolution. Per-event schema version field can live: (a) on the EVT-T* envelope (single version per category), (b) on each sub-shape payload (per-sub-shape independent versioning), (c) both.

**Why deferred:** Phase 4 EVT-S* design needs to choose. Per-payload (b) is more flexible but more bookkeeping; envelope (a) is simpler but couples sub-shape evolution.

**Resolves in:** Phase 4 [`11_schema_versioning.md`](11_schema_versioning.md).

**Default for design:** assume both — envelope tracks category-level breaking changes, payload tracks sub-shape additions.

---

### EVT-Q8 — Idempotency key composition for WorldTick / NPCRoutine across big jumps

**What:** [EVT-T11 WorldTick](03_event_taxonomy.md#evt-t11--worldtick-v130d) and [EVT-T10 NPCRoutine](03_event_taxonomy.md#evt-t10--npcroutine-v130d) idempotency must cover: a 23-day FastForward triggers all relevant pre-scheduled WorldTicks in chronological order, exactly once each, even if scheduler is replayed. Open: idempotency key = `(reality_id, world_tick_id)`? `(reality_id, schedule_id, fired_at_fiction_ts)`? Trade-offs around schedule mutation (author moves a beat).

**Why deferred:** Phase 4 scheduled-events design needs to settle.

**Resolves in:** Phase 4 [`08_scheduled_events.md`](08_scheduled_events.md).

### EVT-Q9 — DP gap: SchemaVersionMismatch resolution

**What:** [DP-K3](../06_data_plane/04a_core_types_and_session.md#dp-k3--dperror-enum) has `SchemaVersionMismatch` variant; DP Q5 was deferred to "Phase 2 CP migration". Phase 0 brief §S9 explicitly ties EVT-S* to resolving DP Q5. Open: do we resolve DP Q5 fully here, or do parts of it stay in DP-C5 expand/migrate/contract?

**Why deferred:** Boundary question. DP-C5 owns the multi-instance migration coordination protocol; EVT-S* owns the per-event schema versioning + upcaster contract. Both are needed. Phase 4 EVT-S* will draw the line.

**Resolves in:** Phase 4 [`11_schema_versioning.md`](11_schema_versioning.md). Cross-references DP-C5.

**Default for design:** EVT-S* owns the per-event-type schema, DP-C5 owns the multi-instance migration choreography. Boundary at "what shape is the data" (EVT) vs "how do replicas reconcile" (DP).

### EVT-Q10 — Time-travel debug replay determinism guarantee

**What:** Brief §S8 third use case: operator queries "what happened in cell X between turn N and M" — must be deterministic. Open: deterministic against what? Same input event log + same fiction-clock state + same RNG seed (DP-Ch27) → same output state. But flavor narration is NOT in event log per EVT-A8 — so debug replay shows structural deltas exactly + flavor as best-effort from audit log.

**Why deferred:** Phase 4 replay design needs to specify the operator UX: "structural-only" mode vs "with-narration-best-effort" mode.

**Resolves in:** Phase 4 [`10_replay_semantics.md`](10_replay_semantics.md).
