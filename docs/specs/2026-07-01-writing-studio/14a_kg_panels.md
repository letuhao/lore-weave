This spec covers the Knowledge/KG surface, using the same two-phase shape [`13_glossary_panels.md`](13_glossary_panels.md) proved out, per [`docs/standards/dockable-gui.md`](../../standards/dockable-gui.md).

## Why a two-phase shape again (not a straight fanout)

Scoping investigation (2026-07-04) found the KG/Knowledge surface is **larger** than Glossary's was: ~4,300 lines across `pages/KnowledgePage.tsx`, `pages/ProjectDetailShell.tsx`, `pages/book-tabs/KnowledgeOntologyTab.tsx`, and `features/knowledge/**` (17 raw tab/section slots across the two hub pages, collapsing to ~13 unique capabilities once shared components are recognized). Same three DOCK-standard violation classes as Glossary, plus one Glossary didn't have:

- **DOCK-8** — two independent page-level tab-switches, not one: `KnowledgePage.tsx` (8 tabs: projects/jobs/global-bio/entities/timeline/raw-drawers/insights/privacy) and `ProjectDetailShell.tsx` (9 sections: overview/entities/timeline/evidence/proposals/gap/insights/schema/graph). Both are **route-driven** (each tab is its own URL, one section mounts per route — not a same-component ternary), so they don't hit the *letter* of DOCK-8's `GlossaryTab` example, but they hit its *spirit*: two pages each host N unrelated capabilities behind internal tab furniture the palette/agent-enum/dock can't see. `KnowledgeOntologyTab.tsx` (a third, smaller offender: `view` state switch over `adopt`/`schema`/`views`/`sync`) already partially self-corrected — its `schema` tab redirects to `ProjectDetailShell`'s `schema` section rather than mounting a duplicate editor.
- **DOCK-9** — grep initially flagged 4 files; one turned out to be a false positive on closer read. `VersionsPanel.tsx` has two genuine hand-rolled modals (`role="dialog"` preview, `role="alertdialog"` rollback-confirm — centered, `bg-black/50`, real DOCK-9 violations, straightforward FormDialog/ConfirmDialog adoption). `ontology/GenerateSchemaDialog.tsx` and `ontology/SchemaWorkbench.tsx` are likewise genuine (same centered/dimmed/z-50 shape), scoped to Phase B (K8 below). **`MemoryIndicator.tsx` is NOT a DOCK-9 violation** — its `fixed inset-0 z-40` is a transparent click-outside catcher for a small anchored dropdown popover (button → chip menu), the identical shape `NotificationBell.tsx` already uses repo-wide and unflagged. DOCK-9 targets a hand-rolled stand-in for a **modal** (centered, dimmed, tab-trapped); an anchored popover is a different, accepted UI pattern with no shared primitive to adopt yet (`@radix-ui/react-popover` isn't even a dependency) — "adopt, don't rebuild" (the standard's own fix mandate) doesn't apply when there's nothing to adopt. Not pulled into this cycle's scope; a future cross-cutting "popover primitive" pass (NotificationBell + MemoryIndicator + SessionSidebar + ~10 more) is a separate, unscoped concern. (Everything else under `features/knowledge/components/**` — `EntityEditDialog`, `EntityMergeDialog`, `BuildGraphDialog`, `ProjectFormModal`, `ChangeModelDialog`, `RegenerateBioDialog`, `EventEditDialog`, `RelationEditDialog`, `ErrorViewerDialog`, `EntityDetailPanel`/`JobDetailPanel`/`DrawerDetailPanel` drawers — is already `FormDialog`/raw `Dialog.*`-compliant; no work needed there.)
- **DOCK-10** — no hoist for the one truly cross-panel piece of state: which Knowledge *project* is "the book's project." `KnowledgeOntologyTab.tsx` re-derives it inline (`projects.find(p => p.book_id === bookId)`) and nothing else reuses that lookup — every future book-scoped KG panel needs the same resolution, currently uncentralized.
- **New — DOCK-7 route-coupling, 3 instances**: `ProposalsInboxTab.tsx` (`<Link>`), `shell/OverviewSection.tsx` (2× `<Link>`), `ontology/CreateSchemaEntry.tsx` (`<Link to="/books/${bookId}/kg-ontology?view=adopt">`). Unlike Glossary's single root-level route defect (A6), these are each local to one future panel slice — no shared root, so each rides with its own Phase-B slice instead of being pulled to Phase A.
- **New — tenancy shape Glossary didn't have**: a Knowledge *project*'s `book_id` FK is **optional** (`docs/03_planning/KNOWLEDGE_SERVICE_ARCHITECTURE.md` — a project "can link to an existing book… or be standalone"). `KnowledgePage.tsx` is a **global, cross-book** hub (every project the user owns, plus user-level surfaces: global bio, privacy, cross-project extraction jobs). Studio is book-scoped (`/books/:bookId/studio`). Per the human's direction this session, **the whole surface still migrates into Studio** — the global pieces become **user-scoped panels** (openable from any book's Studio, same precedent as `usage`/`notifications`/`settings`/`trash` from wave 1 — those already show cross-book, user-owned data from inside one book's dock), not book-scoped ones.

Per [[fanout-independent-slices-parallel-build-serial-integrate]], the shared pieces (hub launcher, the book→project hook, the two load-bearing DOCK-9 widgets, Lane-B wiring) land **serially first** (Phase A); the ~12 remaining capability panels are then **provably disjoint files** and fan out in parallel (Phase B).

**Decision (post-Phase-B, 2026-07-04) — classic routes are NOT retired.** `pages/KnowledgePage.tsx`, `pages/ProjectDetailShell.tsx`, and `pages/book-tabs/KnowledgeOntologyTab.tsx` stay exactly as they are, coexisting with the new dock panels — reversing this spec's earlier assumption that they'd shrink into redirects. Reasoning: wave-1's own spec (`11_dockable_migration.md`) already made and documented this exact call for `usage`/`notifications`/`trash`/`settings` ("existing pages/routes stay — multi-device + non-studio entry points; panels are additional mounts, not replacements") and those classic routes are still live and unretired today. Knowledge's case is *harder* than wave-1's, not easier: `KnowledgePage` (global hub) and `ProjectDetailShell` (project-id-keyed, and `Project.book_id` is genuinely nullable — a standalone project has no book to redirect into) have no reliable "which book's Studio" target, and there is no "last active book"/default-book mechanism anywhere in this codebase to invent one from. Studio is also documented desktop-first (`00_OVERVIEW.md` "Out of scope") — retiring these routes would remove the only mobile/multi-device-reachable path to this data for a mostly-cosmetic win. `KnowledgeOntologyTab` (a book-tab, `bookId` already known) is the one case that COULD redirect cleanly, but doing so needs a new Studio deep-link mechanism (auto-open a panel via URL on mount) that doesn't exist yet either — out of scope unless a future session specifically wants to build that.

## Out of scope

- The extraction engine / wizard itself (shared with Glossary's extraction entry point, #13 already scoped its glossary call site) — only the Knowledge-invoked call sites of any shared wizard are in scope here, not the wizard's shared step machinery.
- Wiki pages — separate feature/route (#12 cycle queue lists it independently).
- Any BE/MCP tool schema changes — this cycle is FE dockable migration only; if a Lane-B gap surfaces in A4 that needs a new tool, that's tracked separately (No-Defer-Drift gate 4 — buildable now if the signal exists, not "blocked").
- Server-sync of any new panel's layout (existing D6 localStorage convention applies).
- A per-project mode for Mining Insights (K7) — only if a real use case appears.

## Testing discipline

Per [`00_OVERVIEW.md`](00_OVERVIEW.md) and [`13_glossary_panels.md`](13_glossary_panels.md)'s precedent: unit + E2E per panel/hook, `/review-impl` at Phase A close and at Phase B integration close. Phase B's cross-panel E2E (e.g. hub → overview → entities drill-down) may defer to the debt stack until ≥2 Phase-B panels exist.
