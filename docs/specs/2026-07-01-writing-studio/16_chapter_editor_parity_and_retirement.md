# 16 · Chapter Editor Parity & Retirement (COHERENCE)

> Component of [Writing Studio (v2)](00_OVERVIEW.md). Status: ✅ COMPLETE 2026-07-05 — Phase 1 (data-safety parity) shipped 2026-07-04; Phase 2 (editor-craft UX), Phase 3 (Translate workmode), Phase 4a (M6 responsive-chrome), Phase 4b (M9 — kept `ChapterEditorPage.tsx`, marked deprecated, not deleted) all shipped 2026-07-05.
> Origin: the "Cursor-for-novels" missing-pieces register, item #1 COHERENCE (memory `writing-studio-fragmented-not-underbuilt`) — the platform runs **two live, uncoordinated chapter-editing surfaces** and doesn't feel like one product.
> Builds on [#04](04_manuscript_editor.md) (manuscript hoist — several of its own "build notes" are still open and fold into this plan), [#08](08_studio_state_architecture.md) §"Migration: editorBridge → bus + reconciler" (the target architecture for the write-back fix was already decided there, just not built), [#09](09_agent_gui_reconciliation.md) (Lane C `applyProposedEdit`, G7 dirty-guard).

## Problem

Two routes both do "edit a chapter" and neither knows the other exists:

| Route | Page | How users reach it |
|---|---|---|
| `/books/:bookId/chapters/:chapterId/edit` | `ChapterEditorPage.tsx` (legacy) | **Every** chapter-row click and the pencil icon in `ChaptersTab.tsx` — the default, most-used path |
| `/books/:bookId/studio` | `WritingStudioPage.tsx` (Studio v2) | One promoted header CTA button on the book detail page |

A new user who clicks a chapter (the natural action) never sees Studio. A user who clicks "Studio" gets a materially different, less-capable chapter-editing experience for anything beyond drafting prose + basic AI co-writing. This is not a cosmetic gap — a full capability audit (2026-07-04) found **15 legacy-only capabilities with no Studio equivalent**, several of them data-safety features (Checkpoints, Revision History, Publish Gate).

**Decision (user-approved 2026-07-04, CLARIFY):** retire `ChapterEditorPage` — Studio becomes the **sole** chapter-editing surface once parity is reached. This spec is the parity-then-retirement roadmap. Per this track's own build-while-plan convention (see [`00_OVERVIEW.md`](00_OVERVIEW.md)), each phase below gets fleshed out into its own detailed spec **when that phase starts** — this document locks the decisions, the phase order, and the two prerequisite architecture fixes that block everything else.

## Prerequisite architecture fixes (block Phase 1 — do these first)

These aren't new capabilities; they're compliance debt in code that **already shipped** (including today's `fb98f161f` APPLY-DIFF fix), flagged by the audit as landmines that would get worse the more capability we pile onto the current wiring.

| # | Fix | Why it blocks Phase 1 |
|---|---|---|
| **P1** | **Lane-C compliance** — Studio's `EditorPanel`/`ComposePanel` stop using the global `editorBridge` singleton (`registerEditorTarget`/`getEditorTarget`) for `propose_edit` Apply. Add `applyProposedEdit(diff)` as a real action on `ManuscriptUnitProvider` (Tier-4) per spec [#08](08_studio_state_architecture.md)'s own "Migration: editorBridge → bus + reconciler" steps 1–3 and spec [#04](04_manuscript_editor.md)'s "Agent write-back (Lane C)" section — both already describe the target shape, neither is built. `ProposeEditCard`'s Apply path in the studio surface calls the hoist action instead of the bridge. **`ChapterEditorPage` keeps using `editorBridge` unchanged** (spec 08 step 5: "do not delete in this track") — it's being retired anyway, not worth touching. | A global mutable singleton (`registerEditorTarget`) is exactly the kind of implicit cross-surface state that gets more dangerous as Checkpoints/Revision-History (Phase 1) start reading "what chapter is currently open" from more places. Fix it before adding more consumers, not after. |
| **P2** | **Reuse the G7 dirty-guard pattern for the new Phase-1 restore paths.** Correction to the initial 2026-07-04 audit finding: G7 (spec [#09](09_agent_gui_reconciliation.md)) is **already implemented** for the Lane-B reconciler — `bookEffects.ts`'s `bookDraftEffect` checks `ctx.isChapterDirty?.(chapterId)` before reloading (confirmed in code, not just spec). P2 is narrower than "build the guard from scratch": Checkpoints-restore (1.2) and Revision-History-restore (1.3) are **two new reload-capable code paths**, and each must call the same `isChapterDirty()` check before restoring over the live document — not a new mechanism, just don't let two new call sites skip the pattern their sibling already follows. | Skipping this on a "just this once, it's not the reconciler" assumption is exactly how a guard pattern gets silently unenforced on new code paths — cheap to require now, expensive to discover after a user loses work to a restore. |

**Gate:** P1 lands with a regression test proving `applyProposedEdit` genuinely replaces `editorBridge` on the Studio path (no residual `registerEditorTarget` call site in `EditorPanel.tsx`/`ComposePanel.tsx`). P2 is folded into 1.2/1.3's own test suites (each restore path gets a "blocked by a dirty hoist" test) rather than being a separate merged commit — there's no shared new code to gate on, just a shared rule to enforce at two new call sites.

## Phase 2 — Editor-craft UX parity (kickoff CLARIFY done 2026-07-04 — detailed below)

**Kickoff audit (2026-07-04, 2 parallel Explore agents over real code)** confirmed the shared `TiptapEditor` component (`components/editor/TiptapEditor.tsx`) already exposes real, implemented props/ref-methods for 11 of the 12 legacy items (`grammarEnabled`, `editorMode`, `focusMode`, `selectionMenu`/`aiLayer` slots, `setGlossaryEntities`/`setGlossaryEnabled`, `setHeatmapTerms`/`setHeatmapEnabled`, `insertAtCursor`/`replaceSelection` with `ProvenanceAttrs`, `markAllProvenanceReviewed`, `setProvenanceVisible`, `getUnreviewedProvenanceCount`) — the gap is entirely in the host layer (`EditorPanel.tsx`/`ManuscriptUnitProvider.tsx`), which today passes only `ref`/`content`/`onUpdate` to `<TiptapEditor>` and calls none of these methods. Full audit detail lives in the session transcript, not duplicated here; findings are folded into the task table below.

**M7 resolved: Classic/AI editor-mode toggle (`useEditorMode`) is NOT ported.** User decision 2026-07-04 — Studio's Compose is always tool-calling-capable via `propose_edit`; a separate Classic/AI axis (which in legacy also gates media-block editability + slash-menu richness) adds a control surface nobody asked for. Dropped from scope, not deferred — a conscious won't-fix (CLAUDE.md defer-gate #5), tracked here so it isn't re-litigated.

**Two items are architecture redesigns, not ports — folded into Phase 2 as new prerequisite-style tasks (2.7, 2.8 below):**
- **Image/video upload-context + version-history (legacy item #10).** Legacy stores upload context (`{token, bookId, chapterId}`) and history-open callbacks as **module-level singletons** (`ImageBlockNode.tsx`'s `setImageUploadContext`/`setOnOpenHistory`, `VideoBlockNode.tsx`'s `setOnOpenVideoHistory`) — safe only because exactly one `ChapterEditorPage` is ever mounted at a time. Studio's dockview allows **multiple chapter tabs open simultaneously**, each with its own `EditorPanel`/`TiptapEditor` instance; the singleton would silently misattribute uploads/history-opens to whichever tab last set it. User decision 2026-07-04: **redesign now, don't port the landmine.** Approach: move the context onto `editor.storage` (mirroring the pattern `TiptapEditor.tsx:199-220` already uses for `editorMode` — `editor.storage.mediaGuard.editorMode`), so each `Editor` instance owns its own upload context / history-callback, keyed by nothing but the instance itself. `ImageBlockNodeView`/`VideoBlockNodeView` read `this.editor.storage.mediaUpload` (or similar) instead of the module-level getter. Legacy `ChapterEditorPage.tsx` continues to work unchanged as long as it still sets the storage on its own single editor instance (same net behavior, no cross-page contract change since it only ever has one instance anyway).
- **Popout Compose window (legacy item #11, popout-insert-relay).** User decision 2026-07-04: wants a **true separate OS/browser window** (multi-monitor use), not a same-page dockview float (which already exists natively via dockview's split/float and needs no new work). The existing `openPopoutChannel(bookId, chapterId)` primitive (`features/composition/workspace/popoutChannel.ts`) is already generic and correctly scoped per-(book,chapter) — reusable as-is, not composition-specific in its channel/message shape. **The real new problem:** Studio's Compose is Chat-based (`ComposePanel.tsx` renders `<Chat>`, not `CompositionPanel`), and `ProposeEditCard`'s Apply path calls `getEditorTarget()` from the module-singleton `editorBridge` (`features/chat/context/editorBridge.ts`) — which is **per-window** (a popout is a separate JS realm/React root; `getEditorTarget()` there returns `null`, so Apply would silently no-op today). Legacy's popout (`PopoutHost.tsx`/`PopoutBridge.tsx`) never had this problem because `CompositionPanel`'s solo-mode `onAccept` doesn't go through a human-gated diff-review card at all — it's a direct generate-and-accept, not `propose_edit`. Studio needs a genuinely new cross-window Apply path: `ProposeEditCard` must detect "I'm running inside a popout" (e.g. a context flag set by the popout host route) and, when true, `channel.post({kind:'insert-prose', text, provenance})` instead of calling `target.applyProposedEdit` directly; the **opener**'s `EditorPanel.tsx` calls `usePopoutInsertRelay(bookId, chapterId, onInsert)` (hook already exists, generic, untouched) wired to `unit.applyProposedEdit(...)`. New files: a Studio popout route (mirroring `PopoutHost.tsx`'s shape — separate root, reuses `App.tsx`'s providers, renders `<Chat>` solo) + a `StudioPopoutBridge` (mirrors `PopoutBridge.tsx`) invoked from `ComposePanel.tsx`'s toolbar. This is the single largest item in Phase 2 — sized and gated on its own (2.8).

## Phase 4a — M6 resolved: responsive Studio chrome ✅ COMPLETE 2026-07-05

**User's product call on M6 (2026-07-05):** NOT a separate mobile shell, NOT `ChapterEditorPage`'s `MobileEditorShell` kept alive, NOT read-only degrade. Studio's existing dockview frame already renders acceptably on a single narrow-viewport panel — "the hard thing is only the tab control and individual panel responsive." Direction: make the existing dockview-based Studio responsive, not build a second UI.

**Live-verified root causes** (chrome-devtools emulation at a genuine 390×844 mobile viewport — an earlier `resize_page` call silently floored at 500px width and produced false readings; `emulate({viewport: '390x844x2,mobile,touch'})` gave the real CDP override):

1. **dockview's own tab-strip overflow mechanism already works correctly** — `.dv-tabs-overflow-dropdown-root` (a "⌄ N" chevron+count) appears once tabs overflow and opens a proper touch-sized list. No code change needed; the "tab control" complaint is about a subtle, easy-to-miss affordance, not brokenness. Not fixed (low ROI to patch a working library internal for discoverability alone) — noted for a possible future `dockviewTheme.css` polish pass, not tracked as a defer row (cosmetic, no user-report backing it beyond the general "tab control" framing already addressed by the actual reproducible bugs below).
2. **`EditorPanel`'s action-toolbar row** (Grammar/Heatmap/Focus/Scenes/Save/Publish/…) had no overflow handling — on a narrow viewport, buttons past the visible width (confirmed: Save, Publish) were rendered off-screen with **no way to reach them at all**. Fix: `overflow-x-auto` (horizontal scroll) on the row.
