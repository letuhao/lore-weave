## Locked decisions

| # | Decision | Why |
|---|---|---|
| U1 | **Two panels, not one:** `jobs-list` (palette + agent-openable) and `job-detail` (hidden-from-palette, params-retargeting singleton — same shape as `json-editor`/`skill-editor`). | List and detail are the same capability at different depth (not DOCK-8's "unrelated capability" case), but a user plausibly wants to watch one job's progress in a tab while the list keeps updating in another — collapsing them into one panel loses that. `job-detail` takes `{service, jobId}` params (F1), same precedent as `SkillEditorPanel`'s `{skillId}`. |
| U2 | **`JobsStreamProvider` wraps both new panels**, reused as-is — no extraction needed. | It's already a router-free context provider (the correct Tier-3/4 hoist); today's two pages already each wrap it independently, so panels do the same. |
| U3 | **Shared leaf components (`JobRow`, `JobMonitor`, `JobsMobile`) take an injectable navigate-callback prop instead of forking.** Standalone pages keep passing a real `navigate()`/`<Link>`; panels pass a callback that calls `host.openPanel('job-detail', { params: { service, jobId } })` (list→detail) or nothing (list panel has no "back" target). | Mirrors the exact `NotificationsPanel` → `NotificationItem.onClick` → `followStudioLink` pattern (`onItemClick` prop, not a fork) — DOCK-2's "reuses AS-IS" is satisfied by dependency injection, not a copy. Avoids inventing a second navigation abstraction beyond the F3 resolver already proven for Settings/Notifications. |
| U4 | **Standalone `/jobs`, `/jobs/:service/:jobId` routes stay** (same reasoning as Settings/Notifications — jobs are user-scoped, reachable with no active book/studio mounted, e.g. from the extraction wizard's `navigate('/jobs')` after starting a job). | No forced route retirement; this is additive (a new in-studio surface), matching U-none precedent set by wave-1. |
| U5 | **`jobs-list` joins the `ui_open_studio_panel` agent enum; `job-detail` does not** (params-based singleton, same as `json-editor`/`skill-editor` staying outside the enum per DOCK-6). | An agent plausibly wants to open "my jobs" for the user to see; opening one specific job by id isn't a case any current MCP tool surfaces (no tool returns a bare `{service, jobId}` today) — add to the enum only when a concrete tool needs it, not speculatively. |

## Scope amendment (user call, post-CLARIFY): Books + Leaderboard ARE in scope

The structural-mismatch findings above stand as accurate engineering facts, but the user explicitly wants both docked anyway, at full scope (not the slimmer variants offered as lower-risk alternatives): **full `BooksPage` port** (list+search+create+coverage, reframed as browse-then-read — see Phase C) and the **full global, unscoped 4-tab Leaderboard**. Phases C and D below design how to do that without silently violating DOCK-7/DOCK-8. Books turned out not to need any rule exception at all once reframed as a pure view/read capability (see Phase C's design correction); Leaderboard needs the DOCK-8 fix (4 catalog entries instead of one internal tab-switch) but no exception either — both land as ordinary, standard-compliant panels.

### Phase C — Books: Browser + Reader (revised, NO navigate-away — DOCK-12 dropped)

**Design correction (user call, supersedes the earlier DOCK-12 draft):** the actual desired capability is narrower and safer than "switch to a different book's studio" — it's **browse + read another book without ever leaving your own active workspace**. The writer's current studio (bookId=A) stays mounted, untouched, for the entire session; browsing/reading book B is a pure, non-mutating VIEW action with zero effect on book A's state. Since nothing here needs to leave the current studio, **no DOCK-7 exception is needed at all** — this is a normal two-panel dockable feature, not a rule carve-out. (Only a future social layer — comment/like on another book — would touch real mutation, and that doesn't exist yet; out of scope until designed.)

**Sequencing strategy for this spec (mirrors the KG effort's own stated plan):**
1. **Build all new-file work first** (every hook + panel in the tables above) — zero collision risk, safe to do immediately regardless of what else is in flight.
2. **Hold the spine-file edits** (catalog rows, i18n keys, the `jobs-list` enum row, any effect-handler registration) until immediately before each phase's commit — re-check `git diff` on those 4 files right before editing them (not just at BUILD start) so an append lands cleanly against whatever Glossary Phase A/B, the Inspector GUI, or the KG effort has already landed, rather than silently overwriting a concurrent addition.
3. **Land each phase (B / C+D) as its own commit** — smaller diffs on the shared spine files are easier for a human or another agent to rebase past than one giant combined commit.
