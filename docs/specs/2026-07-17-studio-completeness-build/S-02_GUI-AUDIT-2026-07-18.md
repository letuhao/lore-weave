# S-02 Manuscript Parts — live GUI/UX audit (user's-eye)

> **Method:** rebuilt book-service (:8205) with the S-02 routes + served the new FE on an isolated
> port (:5199 → gateway :3123), logged in as the test account, and **operated the feature by hand** in
> a real browser (chrome-devtools). Every finding below is from using it, not reading code.
> Screenshots: `scratchpad/s02-studio-01.png`, `s02-after-drag.png`.

## What works (verified live — NOT a hollow shell)
- Create act (New-act button → act appears) · Drag chapter → act (nests, count badge updates,
  Unassigned decrements) · Trash act holding a chapter → the chapter returns to **Unassigned**
  (un-filed, not deleted; confirm message is clear) · Chapter click → **Editor opens** (no view-only
  dead end) · grouped tree renders clean — **no overlap, no clipping, counts/nesting correct**.
- No console errors from S-02 code (only pre-existing Router/Tiptap warnings).

## Findings

| # | Severity | Finding | Backend ready? |
|---|---|---|---|
| **1** | HIGH | **Reorder acts has NO GUI.** No control, act headers not draggable, hook doesn't expose `reorderAct`. DOM: `reorderActControl:0`, `actHeadersDraggable:[false]`. A user with 2+ acts cannot reorder them. | ✅ route + `book_part_reorder` + `partsApi.reorder` |
| **2** | HIGH | **Restore a trashed act has NO GUI.** Only active acts shown; no "show trashed" / restore, no undo. DOM: `restoreOrTrashView:0`. Trash-by-mistake = the act's name/grouping is unrecoverable from the UI. | ✅ `?include_trashed` + restore route + `book_part_restore` + `partsApi.restore` |
| **3** | MED | **Acts mislabeled "arc" in the footer** ("1 arc · 4 ch"). `counts.arcs` reused for the parts count; footer prints "arc". | n/a (FE) |
| **4** | MED | **Native `prompt()`/`confirm()`** for create/rename/trash — un-styled OS dialogs that block the page and break the polished look. (Known pattern-debt: ChapterBrowser flags the same, "upgrade to ConfirmDialog".) | n/a (FE) |
| **5** | MED | **Hover-only affordances break on touch.** Rename/trash reveal on `:hover`. Tablet/mobile (a stated platform) has no hover → unreachable. | n/a (FE) |
| **6** | LOW | **Empty act gives no drop-target cue**; no drag-over highlight while dragging (the draft's active-drop-target style was dropped). | n/a (FE) |
| **7** | LOW | **Two adjacent "+"-like buttons** (FolderPlus New-act vs Plus Plan) both read "add"; only tooltips disambiguate. | n/a (FE) |
| **8** | LOW | **Can't create a chapter INTO an act** — New-act makes acts only; new chapters route through Plan, then must be dragged in. | partial |

## Scorecard
Usability 6.5 · Completeness(CRUD) 6 · Ease-of-use 7 · Beauty 6 · Consistency 6.5 · Accessibility/touch 5 ·
Robustness 8 → **Overall ≈ 6.4/10.** Real & working, but two backend-complete verbs have no button
(reorder, restore) and rough edges (arc-label, OS dialogs, touch).
