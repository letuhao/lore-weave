# LOOM T3.4 — Grounding Pin / Exclude (spec)

> **Status:** DESIGN · **Size:** XL · **Branch:** `feat/composition-service` · **Date:** 2026-06-24
> **Workflow:** /loom v2.2 human-in-loop. CLARIFY approved (PO, 2026-06-24).
> **Product spec source:** `docs/specs/composition-studio-mockup-v3.html` (Grounding subtab, `data-st="gr"`).

## 1. Goal & user value

In the co-writer's **Grounding** panel the author sees what will be injected into the
next generation. T3.4 lets them **steer** it per item:

- **Pin** an item → force-keep it in the injected context even when the budget would
  otherwise trim it (and mark it as deliberately retained).
- **Exclude** an item → drop it from the injected context entirely.

The author's choice **persists per-scene** (server SSOT, cross-device) and is honored
by BOTH the grounding **preview** and the actual **generation** — same `pack()`
chokepoint, so preview == what the model receives ("no silent seam", CLAUDE.md).

## 2. Scope (locked at CLARIFY)

| Decision | Locked value |
|---|---|
| Granularity | **Per item/line** |
| Addressable item types | **`present` (cast entities), `canon` (canon rules), `lore` (lore hits)** — the three with stable source ids the PO named |
| Persistence | **Per-scene, server SSOT** (`outline_node_id` keyed) |
| Non-addressable blocks | `beat / threads / recent / memory / open_promises / guide` stay opaque (no stable per-line id) — rendered as today, **not** pin/excludable in v1 |

### Out of scope (tracked as deferrals, not silently dropped)
- `D-T3.4-TIMELINE-PIN` — pinning individual `memory`/timeline events (events carry ids
  but aren't in the PO's named set; revisit if authors ask).
- `D-T3.4-BLOCK-LEVEL` — a "exclude the whole `lore` block" coarse toggle (per-item covers it).

## 3. Stable item identity (the addressable key)

Pin/exclude key on a **canonical, language/derivative-stable id** — never a localized
label (so a pin survives a reader-language switch or a derivative override):

| Type | `item_id` source | Notes |
|---|---|---|
| `present` | `present[].entity_id` | already the **glossary anchor** (stable), not knowledge's rename-sensitive canonical_id — see `lenses.gather_present` DI3 |
| `canon` | `CanonRule.id` (uuid) | composition-owned rule id |
| `lore` | lore hit stable id = `source_id` (+ `:chunk_index` when the hit carries one) | a hit without a resolvable id is **not** pinnable (rendered read-only) |

`(item_type, item_id)` is the unique address within a scene.

## 4. Persistence — `scene_grounding_pins`

New table in composition-service (owns its DB):

```
scene_grounding_pins(
  id              uuid PK default gen_random_uuid(),
  user_id         uuid NOT NULL,
  project_id      uuid NOT NULL,
  outline_node_id uuid NOT NULL,          -- the scene
  item_type       text NOT NULL CHECK (item_type IN ('present','canon','lore')),
  item_id         text NOT NULL,          -- stable id (uuid or source_id[:chunk])
  action          text NOT NULL CHECK (action IN ('pin','exclude')),
  created_at      timestamptz NOT NULL default now(),
  UNIQUE (project_id, outline_node_id, item_type, item_id)
)
```

- Scoped to `project_id + outline_node_id` (the scene). `user_id` carried for tenancy
  filtering (matches sibling repos).
- One row per addressed item; `action` is `pin` XOR `exclude` (the UNIQUE key flips it
  via upsert — pinning an excluded item replaces the row, never two rows for one item).
- "Clear" = DELETE the row (item returns to default budget behavior).
- Migration is idempotent (DO-block / `IF NOT EXISTS`), per repo convention.

## 5. Backend mechanism (`pack()` — the single chokepoint)

`pack()` is consumed by the grounding endpoint **and** the engine generation path, so
reading the pin/exclude set inside `pack()` makes both honor it with no call-site able
to forget it (lesson: audit-all-callsites — we avoid the trap by centralizing).

New dep: `grounding_pins_repo` (optional, like `canon_repo`). When wired, after the
lenses gather + the **spoiler filters**, `pack()`:

1. Reads the scene's set: `pins`/`excludes` per `(item_type, item_id)` for `node['id']`.
2. **Exclude** — drop matching items from `bundle.present` / `bundle.canon` / `bundle.lore`
   **before** `build_segments`. (Exclude is always spoiler-safe: it only removes.)
3. **Pin** — mark matching items' segments `protected=True` so `enforce_budget` keeps
   them. Most meaningful for **lore** (default droppable, `PRIO_LORE`); `present`/`canon`
   are already protected, so a pin there is a no-op for budget but still recorded
   (UI intent + future ordering).

**Ordering invariant (spoiler-safe):** gather → **spoiler filter** → exclude/pin → budget.
A pinned item that the spoiler filter already removed (beyond the cutoff) never reaches
the pin step, so a pin can NEVER resurrect a spoiler-filtered item (preserves T2.3).

**Sanitize invariant:** pinned `lore`/override text still flows through the existing
`sanitize_lore` in `build_segments` (pin only changes `protected`, not the sanitize path).

### Surfacing addressable items to the FE
`get_grounding` today returns opaque `blocks{}`. Add an addressable list to the response
(derived from the post-exclude bundle):

```
grounding_items: [
  { type: 'present'|'canon'|'lore', id: str, label: str,
    pinned: bool, excluded: bool }
]
```

- `label` = the human line (entity name+summary / rule text / lore snippet) for display.
- `excluded` items are still returned (so the FE can show + un-exclude them) but are
  flagged + were NOT packed into `blocks`.
- The opaque `blocks{}` stays for the non-addressable blocks (back-compat; the FE keeps
  rendering them as `<details>`).

## 6. API

```
PUT /v1/composition/works/{project_id}/scenes/{outline_node_id}/grounding-pins
    body: { item_type, item_id, action: 'pin'|'exclude'|'none' }
    - 'none' clears (DELETE the row); 'pin'/'exclude' upsert
    - 200 { item_type, item_id, action } ; 404 if work/scene not owned
GET  …/grounding  (existing) → now also returns `grounding_items[]` (pin/exclude state folded in)
```
Both routes go through the existing work/scene ownership gate (SEC2). No new gateway
route needed (catch-all `/v1/composition/*` auto-proxies).

## 7. Frontend

- `GroundingPanel.tsx` — for `present`/`canon`/`lore`, render `grounding_items[]` as
  per-line rows with a **pin** (📌) and **exclude** (🚫) toggle; excluded rows shown
  struck-through/dimmed with an un-exclude affordance. Non-addressable blocks render as
  today.
- New hook `useGroundingPins` — `setAction(item, action)` → optimistic update + PUT,
  invalidates the grounding query so preview re-packs.
- `api.ts` — `setGroundingPin(projectId, sceneId, body, token)`.
- i18n ×4 (`en/vi/ja/zh-TW`, namespace `composition`): `pin.*` keys (pin/unpin/exclude/
  restore/pinnedHint).

## 8. Acceptance criteria

1. Author pins a lore hit → it survives a budget trim that would otherwise drop it
   (preview shows it pinned; a generation with a tight budget still includes it).
2. Author excludes a cast entity → it disappears from the grounding preview AND is
   absent from the packed prompt the engine generates with.
3. Pin/exclude **persists** across reload and on another device (server SSOT).
4. A pin can **not** surface an item past the spoiler-safe cutoff (T2.3 invariant holds).
5. Pinned untrusted `lore` text is still sanitized.
6. Clearing returns the item to default budget behavior.
7. Non-derivative and derivative packs both honor the set; multilingual label change
   does not orphan a pin (id is canonical, not the localized label).

## 9. Test plan (evidence gate)

- **BE unit:** repo upsert/flip/clear + tenancy; `pack()` honors exclude (item absent),
  honors pin (protected through a forced budget trim), spoiler-cutoff-beats-pin,
  sanitize-still-applies; grounding response carries `grounding_items[]`.
- **BE integration (real PG):** migration idempotent; repo CRUD round-trip.
- **FE vitest:** panel renders items + toggles; exclude strike-through; optimistic
  set + invalidate; persist (mocked PUT).
- **Live-smoke (≥2 services — composition + knowledge/glossary):** pin a real lore hit
  on a scene → grounding preview keeps it under a tightened budget; exclude an entity →
  absent from `grounding` blocks. Token: `live smoke: <one-liner>`.

## 10. Risks / notes

- Lore hit id stability: if `search_drawers` hits lack a per-chunk id, `source_id` alone
  means pin/exclude acts at the **lore-source** granularity (all chunks of a chapter) —
  acceptable, documented in the panel copy. Verify hit shape in BUILD.
- `CanonRule.id` assumed present (DB model) — verify in BUILD.
