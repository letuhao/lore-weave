# T5.4 — Dock / float / pop-out windowing model — design spec

**Track** 5 (Editor shell) · **Phase** P2 · **Type** [FE] (large) · **Depends-on** all the panels (it hosts them) · **Status** DRAFT
**Program:** [composition-design-program](../../plans/2026-06-10-composition-design-program.md). **Mockup:** `composition-studio-mockup-v3.html` the `+` picker "Dock as panel" / "Open as view" (lines 298–309) + the dockable right sub-tabs (291–310).

> Replace the **fixed** sub-tab strip (`CompositionPanel`: compose/assemble/planner/grounding/canon/quality/settings + the new Threads/Cast/Style/References/Flywheel/Progress) with a flexible **dock / float** windowing model: the author picks which panels to dock, floats them, and opens the big views (T5.5) full-width. This is the IDE chrome that frames everything.

## 1. UX

```text
toolbar: [▤ left] [▦ right]            + (add component ▾)
                                        ┌ Dock as panel ─────────┐
right rail (docked, reorderable):       │ ⚖ Canon   👥 Cast       │
  [⚖ Canon][👥 Cast][🧵 Threads][🎚 Style]│ 🧵 Threads 🎚 Style ♻ Fly│
  ┌─ floating ─────────┐                 ├ Open as view ──────────┤
  │ 📊 Progress   ✕ ⤢   │  ← drag/resize  │ 🎬 Scene 🕑 Time ⌗ Beat │
  └────────────────────┘                 └────────────────────────┘
layout saved per-device · panels never unmount (CSS show/hide)
```

- A **component picker** (`+`): "Dock as panel" (compact panels → the rail) and "Open as view" (big views → the T5.5 overlay).
- **Dock**: panels live in a reorderable rail (drag via dnd-kit). **Float**: pop a panel into a draggable/resizable in-app window. **Pop-out (PO: in scope)**: a separate OS window (`window.open`) hosting a panel, synced to the main app via **BroadcastChannel/postMessage**.
- The author's **layout persists per-device** (which panels docked/floated, order, sizes) — localStorage (per-device UI state, allowed).
- **No-unmount invariant:** docking/floating/hiding a panel must use CSS show/hide, never remount (stateful panels: streams, chat, editor — CLAUDE.md rule). This is the central constraint.

## 2. FE shape

- **context** `features/composition/context/WorkspaceLayoutContext.tsx` — the layout model (panels: docked order, floating windows {pos,size}, which views open); persisted to localStorage; **stable context** (changes rarely) separate from volatile streaming context (split-by-update-frequency rule).
- **live-state owner (AH-2, REVIEW-ARCH)** `context/LiveStateContext.tsx` (and/or a SharedWorker) — owns the singleton stateful instances (co-writer **stream**, chat **SSE**, Tiptap **editor**); every panel (docked/floated/popped) is a **thin view** that subscribes. An OS-popped window syncs via **BroadcastChannel**. This is what reconciles OS pop-out with not-losing-state.
- **components** — `WorkspaceShell` (renders the rail + floating layer + the T5.5 overlay), `ComponentPicker`, `FloatingWindow` (drag/resize), `DockRail` (dnd-kit reorder). Each existing panel becomes a registered "component" mounted once, shown/hidden by the layout.
- **hook** `useWorkspaceLayout` — dock/float/close/reorder/open-view actions.
- **api / types** — none (per-device layout local). `PanelId` registry.
- **host** — refactors `CompositionPanel` from a fixed strip into the registered-panel model. **Migration:** keep all current panels working as docked-by-default (parity) before adding float/pop-out.

## 3. BE delta

**None for V1** (layout is per-device UI state → localStorage). *Optional later:* sync layout to `/v1/me/preferences` for cross-device layout — deferred (per-device is the documented default for UI state).

## 4. Data contract

Local `WorkspaceLayout = { docked: PanelId[], floating: {id, x, y, w, h}[], openView: ViewId|null }` in localStorage. No wire types V1.

## 5. Acceptance criteria

- The author docks/undocks panels, reorders the rail, and floats a panel to a movable/resizable window — layout persists per device.
- Big views open full-width via the T5.5 overlay.
- No panel loses state (stream/chat/editor) when docked/floated/hidden (no remount).
- Falling back to the default layout restores the current fixed-strip behavior (parity).

## 6. Test outline

- **vitest**: dock/float/reorder update the layout + persist; a floated panel keeps its hook state (no remount — visibility-lifecycle test); default layout = parity with today; picker adds/removes panels.
- **Playwright** `composition-windowing.spec.ts`: float a panel → drag → persists on reload; reorder the rail; open a power-view.
- **i18n ×4** keys: `ws.add_component`, `ws.dock`, `ws.float`, `ws.popout`, `ws.open_view`, `ws.close`, `ws.reset_layout`.

## 7. Decisions (cleared 2026-06-10)

- **Full dock + float + OS pop-out (PO)** — including a true OS pop-out (`window.open`) synced to the main app via **BroadcastChannel/postMessage**. The most complex FE item.
- **Sequence LAST** (after all panels exist), behind a feature flag, with the fixed sub-tab strip as the **parity fallback**. (Lead)
- **State hoisting (AH-2, REVIEW-ARCH — PO):** React can't move a live tree across browsing contexts, so the live state of stateful panels (co-writer **stream**, chat **SSE**, Tiptap **editor**) is hoisted into a **shared owner** (live-state context / SharedWorker); docked / floated / OS-popped windows are **thin views** synced via BroadcastChannel. The invariant becomes "**the live state never unmounts; views are re-creatable**" — not "the component never remounts". The largest FE undertaking.
- **Per-device layout** (localStorage) V1; server-prefs sync later. (Lead)
