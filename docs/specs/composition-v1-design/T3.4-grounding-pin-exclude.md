# T3.4 — Grounding pin / exclude — design spec

**Track** 3 (Write & AI) · **Phase** P1 · **Type** [FS] · **Depends-on** `/scenes/{node}/grounding` + the packer (exist) · **Status** DRAFT
**Program:** [composition-design-program](../../plans/2026-06-10-composition-design-program.md). **Mockup:** `composition-studio-mockup-v3.html` §C Grounding pane (`data-st="gr"`, lines 331–345 — present/lore/canon/heatmap).

> Give the author control over the grounding pack: **pin** an item (force it into the context) or **exclude** it (keep it out). Today `GroundingPanel` is **read-only** — the packer decides everything. Pins/excludes must **persist per scene** and the **packer must respect them**.

## 1. UX

```text
┌─ 🔎 Grounding · Ch.12·S3 ······················ grounded · 1240 tok
│  PRESENT IN SCENE
│   📌 Kael — swordless (Ch.9)            [pinned]   ✕exclude
│      Mira — hidden allegiance           pin📌      ✕exclude
│  LORE
│      The Gate of Ash                    pin📌      🚫excluded ↺
│  ACTIVE CANON RULE
│      Kael's sword is lost — 1 conflict  pin📌      ✕exclude
└─  pinned items are always injected · excluded never · rest = auto
```

- Each grounding item (within a block: present / lore / canon / threads / …) gets **pin** and **exclude** toggles.
- **Pinned** → always injected (survives budget trimming). **Excluded** → never injected. **Neither** → the packer's normal relevance/budget logic.
- Pins/excludes are **per scene** (the unit the pack is built for), persisted server-side, and **re-applied on every generate**.
- A budget warning if pins exceed the context budget.
- Reuse the existing block accordion; add the toggles + a "pinned/excluded" state chip.

## 2. FE shape

- **hook** — extend `useGrounding` to also read/write the scene's **pin/exclude prefs**; mutations `pinItem` / `excludeItem` / `clear` (optimistic). Item identity = `{ block, ref }` (an entity_id / lore_id / rule_id — see Open Q1).
- **component** `GroundingPanel.tsx` (extend) — per-item pin/exclude controls + state chips; budget warning.
- **api.ts** — `getGroundingPrefs` / `setGroundingPref(sceneId, {block, ref, state})`.
- **types.ts** — `GroundingPref { block, ref, state: 'pin'|'exclude' }`.

## 3. BE delta

**NEW (composition-service):**
- A **grounding-prefs store** per scene node — pins/excludes keyed by `{block, ref}`. Start as a `outline_node`-scoped blob or a small `grounding_prefs` table (Open Q2).
- The **packer respects prefs**: pinned items are force-included (priority over budget trimming); excluded items are dropped before packing. Wire into `packer/pack.py` (the same place blocks are assembled).
- Endpoints: `GET/PUT /v1/composition/works/{project_id}/scenes/{node_id}/grounding-prefs`.

## 4. Data contract

`GET /scenes/{node_id}/grounding-prefs` → `{ prefs: GroundingPref[] }`. `PUT` body `{ block, ref, state | null /* clear */ }`. The grounding pack response (`/grounding`) gains per-item `ref` ids + a `pref` marker so the FE can render state. The packer reads prefs at pack time (both `/grounding` preview and the real generate).

## 5. Acceptance criteria

- Author pins/excludes specific grounding items; choices persist per scene.
- A pinned item is always in the pack (even under budget pressure); an excluded item never is — verified on the next generate, not just the preview.
- Pins over budget → a clear warning.
- Clearing a pref returns the item to auto behavior.

## 6. Test outline

- **vitest**: pin/exclude toggles call `setGroundingPref`; state chips reflect prefs; budget warning when pins exceed budget.
- **BE pytest** (composition): the packer **force-includes pins + drops excludes** (assert on the packed blocks, not just the endpoint) — pinned survives a tight budget; excluded absent even when relevant.
- **Playwright** `composition-grounding-pin.spec.ts`: pin an item → it persists → appears in the grounding preview; exclude → gone.
- **i18n ×4** keys: `grounding.pin`, `grounding.exclude`, `grounding.pinned`, `grounding.excluded`, `grounding.clear`, `grounding.budget_warn`.

## 7. Decisions (cleared 2026-06-10)

- **Pack emits structured items with `ref`s** (entity_id / lore_id / rule_id) — the pref key. ⚠ **REVIEW-IMPL MED-4:** the pack today returns `blocks: dict[str,str]` (rendered text, `pack.py:68`); per-item refs require exposing the upstream **structured segments** (`bres.kept`, before `segments_to_blocks`) — a real pack-API + grounding-contract change (text→structured), not an additive field. Feasible (the segments exist) but scope it as a contract change. (Lead)
- **Store:** a `grounding_prefs` table keyed by (node_id, block, ref) — clean upsert/clear. (Lead)
- **Pins win budget:** keep all pins (author intent), warn when over budget. (Lead)
- **Packer respects prefs** in both the `/grounding` preview AND the real generate (test on the packed blocks, not just the endpoint). (Lead)
