# Book Structure Pipeline — centralize the scattered manuscript entity model

**Status:** DRAFT — design cycle, 2026-07-20. For review. **No code in this pass.**
**Origin:** dogfood Bug 4 (a manuscript *Part* vanishes after reload). Root-caused to scattered
book-structure ownership + an implicit Work-coupling + a **silent read-skip**. Confirmed against the
user's real book `019f7719-…` (its `composition_work` row — `project_id 019f772e-…` — survived the
data-loss incident and matches the chat session's `project_id`, proving the book was in outline mode).
**User directive:** *"centralize our logic — it's scattered and it causes this. Need a centralized
pipeline that auto-creates missing entities… no one wants an app with a lot of create-requirement, no
check and silent error — for both humans and agents."*

---

## 1. The canonical symptom (Bug 4)

A user with an LLM-authored book (chapters generated by the co-writer) groups Chapter 1 & 2 into
"Part 1". It shows during the session; on reload it is **gone** from the Manuscript navigator.

- The data is **fine** — API repro proved the part row + the chapter→part links persist.
- The FE `useManuscriptTree` loads parts **only** in `'chapters'` source mode
  (`if (source !== 'chapters') return;`, useManuscriptTree.ts:188). `rawSource = projectId ? 'outline'
  : 'chapters'` — so any book with a composition **Work** is in `'outline'` mode and its parts are
  **never fetched**. They survived the session only via optimistic local state.
- **No error, no check, no owner** of "this book's manuscript structure." The part just vanishes.

This is not a one-file bug. It is the shape of the whole model.

---

## 2. The current fragmented model (the map)

A single conceptual "book manuscript" is split across **two services**, with independent scope keys,
a cross-service FK-by-id, **3+ scattered auto-provision seams**, and **5+ FE read paths**.

### Entities & ownership
| Entity | Service · table | Scope key | Created by | Notes |
|---|---|---|---|---|
| Book | book-service · `books` | `owner_user_id` | `POST /v1/books` | manuscript root; soft-delete lifecycle |
| Chapter | book-service · `chapters` | `book_id` | `POST /v1/books/{id}/chapters` + co-writer tools | carries `structure_node_id` → a composition part |
| Part (act) | composition-service · `structure_node` kind=`part` | **`book_id`** (flat integer rank) | `create_part` (arc.py:642) | **Work-independent** — needs no project |
| Outline (arc/scene) | composition-service · `structure_node` kind=`arc`/`scene` | Work hierarchy (`parent_id`, LexoRank) | plan-forge compile | the planned structure |
| Work | composition-service · `composition_work` | `book_id` (+ `project_id`) | `_ensure_work` / `_ensure_pending_work` / `create_work_for_book` | at-most-one canonical + at-most-one pending (partial unique) |

### The three fault lines
1. **Cross-service FK by id, unenforced.** `chapters.structure_node_id` (book-service) points at a
   `structure_node` (composition). No FK, no cascade, no single transaction. A chapter can reference a
   part in another service that a reader never joins to.
2. **Independent scope keys that the reader couples anyway.** Parts are scoped by **`book_id`** and are
   **Work-independent**. The Work/outline is scoped by **`project_id`**. Yet the FE decides *whether to
   even read parts* from **Work presence** — coupling two things the data model keeps independent.
3. **Auto-provision exists but is scattered + partial.** `_ensure_work` (plan_forge_service.py:1739) is
   a clean, idempotent, book-keyed ensure — but there are **3+ copies** (`_ensure_pending_work` in both
   works.py and mcp/server.py, `create_work_for_book`), and **only plan-forge flows call any of them**.
   `create_part` provisions nothing. So a book's "does it have a Work" state is implicit and path-dependent.

### The read fan-out (what a consumer must juggle to render one manuscript)
`useWorkResolution` + `useActiveWorkId` → resolve `projectId` → branch mode → then EITHER
`compositionApi.listOutlineChildren` (outline) OR `partsApi.list` + `booksApi.listChaptersPage` +
`groupChaptersByParts` (chapters). **Five reads and a mode guess** to answer "what is in this book."
Agents face the same maze with no guidance (Bug 2: the co-writer made a *chapter* when asked for a
*description* — no single structure surface told it what the book needs).

---

## 3. Root cause (one sentence)

**No component owns "the book's manuscript structure"** — so every consumer (FE and agent) reverse-
engineers it from scattered, independently-scoped reads, and when the reverse-engineering guesses a mode
that skips an independently-scoped entity, that entity **silently disappears**.

---

## 4. Design — the centralized book-structure pipeline

Three principles, each directly answering the user's three complaints.

### 4.1 One unified structure read  ("scattered logic → one place")
A single **book-structure resolver** returns the whole manuscript as one coherent tree, keyed by
`book_id`, so no consumer branches on Work-existence or juggles five reads:

```
GET /v1/books/{book_id}/structure   →
{
  book_id, has_work: bool,
  acts: [ { part_id, title, sort_order,
            chapters: [ { chapter_id, title, sort_order, has_scenes } ] } ],
  unassigned: [ chapters… ],        // chapters with no/na part — never dropped
  outline?: { … arcs/scenes … }     // present only when a Work exists; ADDITIVE, not a mode switch
}
```
- Parts are **always** included (they are `book_id`-scoped; Work presence is irrelevant to reading
  them). Bug 4 cannot recur: there is no code path that reads chapters-without-parts.
- `has_work` / `outline` are **additive facets**, not an either/or mode. A planned book shows its acts
  *and* can surface its outline; an unplanned one just omits `outline`.
- **Owner:** the resolver lives where the structure lives — composition-service (it already owns
  `structure_node` + the `_ensure_work`/parts logic), exposed through `api-gateway-bff` like the other
  `/v1/composition/*` and `/v1/books/*` routes. The FE `useManuscriptTree` collapses to *one* fetch;
  the agent gets *one* `book_get_structure` MCP tool (helps Bug 2's class).

### 4.2 Idempotent auto-provision  ("auto-create missing entities")
Consolidate the scattered ensures into **one idempotent `ensure_*` per entity**, invoked at every write
entry — so no human or agent must create prerequisites in order:
- **One** `ensure_work(book_id)` (fold the 3 copies into the `_ensure_work` book-keyed pattern; it is
  already race-safe via the partial unique indexes).
- Writes self-provision: `create_part`, "set up outline", "add a description" each call the relevant
  `ensure_*` first. Creating a part on a Work-less book just works; asking for an outline on a bare book
  bootstraps the Work; there is no "create a Work first" ceremony.
- **Idempotent + at-most-one** (respect the existing partial-unique invariants; never fork a second Work
  — the F5 fork bug `_ensure_work`'s docstring already guards).
- **Tenancy:** every ensured/queried row stays `book_id`/`owner_user_id`-scoped (User Boundaries); a
  grantee never forks their own shadow Work (the fix `_ensure_work` already encodes).

### 4.3 No silent seams  ("no check and silent error")
- A read **never** drops an entity because of a mode guess (4.1 removes the guess).
- A write that cannot attach (e.g. a part-move to a non-existent/foreign part) returns a **typed error**,
  never a no-op that reads as data loss — the frontend-tool-contract "resolver never silently no-ops"
  rule, extended to the structure surface.
- The GUI stops offering an affordance whose result cannot render (or renders it) — no "create a Part
  that immediately vanishes."

## 5. Incremental delivery (do NOT big-bang)

| Phase | Delivers | Fixes |
|---|---|---|
| **P1 — unify the read** | the `/structure` resolver + FE `useManuscriptTree` reads it once; parts always render | **Bug 4** immediately |
| **P2 — consolidate ensure** | one `ensure_work`; writes self-provision; retire the 3 scattered copies | the "create-requirement" friction |
| **P3 — surface errors** | typed errors on unattachable writes; kill silent no-ops; GUI affordance gating | the "silent error" class |
| **P4 — agent structure tool** | one `book_get_structure` + guided create tools through the same pipeline | Bug 2's class (agent picks the right entity) |

P1 is a shippable, self-contained fix for Bug 4 that also lays the resolver every later phase builds on.

---

## 6. Decisions (resolved with the user, 2026-07-20)

1. **Parts vs outline — keep DISTINCT; use "part" terminology, never "act"/"arc" in the manuscript rail.**
   Acts-vs-arcs confused users and the UI was already renamed to **"part"** to fix that. The manuscript
   view is **parts → chapters**; the outline (arcs/scenes) is a **separate planning lens** (plan-hub),
   **never merged** into the manuscript rail. `/structure` returns `parts` (+ `unassigned`) as the
   manuscript, and `outline` only as a distinct, separately-consumed facet a consumer may ignore.
2. **Resolver home: composition-service** (it owns `structure_node` + `_ensure_work`), exposed via the
   gateway, joining book-service for chapter titles/order.
3. **Provision timing: ensure-on-write** (a read never mutates) — **and because book-create is the first
   write, provision the canonical structure THEN.** So a book's structure is always ready (the user's
   "easier"): the resolver never meets a null-structure case, and no human/agent must "create a Work first."
4. **Migration state.** Assume the C-merge C4 write-cutover endpoints (already live); do NOT re-open the
   C-merge (parts book-service→composition C1–C4).
5. **Scope of "structure" in P1.** Parts + chapters (+ the additive `outline` facet). Scenes/beats stay in
   the plan-hub lens.

## 7. Open risk — sequencing (LOAD-BEARING)

**Eager book-create provisioning (decision #3) MUST land after P1, not before.** Giving every book a Work
today flips it to the FE's `'outline'` mode — where parts are not read — so eager-provisioning *before*
the unified read would make **Bug 4 universal** (every book loses its manuscript-parts view). Order:
**P1 (unified read, mode-coupling gone) → then P2 folds in eager provisioning.** The phase table (§5)
already sequences P1 before the auto-provision consolidation; this note makes the dependency explicit so it
is never reordered.
