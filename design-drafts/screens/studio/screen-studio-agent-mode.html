Agent Mode — "mission control" for an autonomous multi-chapter authoring run (register #4, unnumbered) — draft v2

A human starts a bounded run over an ordered list of chapters; a server-side driver drafts them one at a time, a critic scores each draft, and the human reads + accepts/rejects before the next unit proceeds. Backend (authoring_run_service.py, 1346 lines) is fully built; frontend is 0% — this is the first artifact toward it.

✅ CLARIFY complete 2026-07-05 — spec: docs/specs/2026-07-01-writing-studio/20_agent_mode.md. Locked: MCP tools ARE being built (chat can supervise a run); "auto-pause after each unit" (default ON) is now a server-side pause_after_each_unit flag the driver itself honors at its unit-boundary check — NOT the client-poll-and-call-/pause approach still shown running below (that design was found to silently no-op for any run started via MCP with no Studio panel open, which defeats the point of the MCP tools existing at all). Keyboard triage (accept/reject/next) is also locked in for v1. This mockup's interactive demo below is frozen at the point that surfaced the gap — it is a UX/data reference, not a literal build target for the pause mechanism; follow the spec, not this page, where the two disagree.

Confirmed 2026-07-05 (was an open question in v2's first pass): the driver does not wait for accept/reject — it drafts every unit back-to-back until budget/scope/critic-severe stops it. Accept/Reject/Revert-all are server-gated to report_ready / failed / paused only — reviewing "as you go" means the human must actively pause.
