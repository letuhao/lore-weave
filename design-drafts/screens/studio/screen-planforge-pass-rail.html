Spec: 35_planforge_studio.md (wave 5). One new panel, plan-passes, over a compiler that is ~85% backend-complete: all four Pass-Rail routes ship, and all four have zero FE consumers. Two of the seven passes — cast and beats — are BLOCKING: the runner stops until a human accepts them. The only door that clears them (POST …/checkpoint) has no GUI, so the human — the only oracle the design says can answer those questions — is the one party who cannot answer them. The agent can. Descends from design-drafts/planforge/2026-07-06-planner-panel-redesign-mockup.html and keeps its locked principle: the writer never edits raw structured data, at any failure point.

② The gate the ledger hides — this is the load-bearing finding (F-P4). PF-7 refuses to accept cast until its glossary seed proposal is applied (plan_forge_service.py:686). The worker creates that proposal and stores its id on the pass entry — and derive_view never returns it (plan_pass_service.py:306-318 carries artifact_id and job_id, and stops). So a GUI sees blocked_at:"cast", offers Approve, gets a permanent 409, and has no way to discover which proposal to apply — there is no list-proposals-for-run route. A stored-but-never-returned field, 275 lines from the comment in the same file warning that such a field "is indistinguishable from a dropped one." BE-20 emits bootstrap_proposal_id + decided_by + decided_at; all three are already on PassEntry. It is XS, and without it this panel is a lie. Click “review seed →” above — the two-step it unlocks (review → approve → apply → Approve) is the wave's DoD step (c).
PASS_ORDER is not the dependency graph — and the first draft of this mock got two rows exactly backwards. world.depends_on = ("cast",), so while cast sits pending, world is BLOCKED — the first draft drew the paid Run button on it, and that button 409s. beats.depends_on = ("motifs",) only, so with motifs accepted beats is RUNNABLE — the first draft drew it as blocked. It is the one pass a GUI-only user can still fire, straight into a second checkpoint they also cannot clear. Four of seven passes are gated on an accepted cast. And pass_cursor is contiguous fresh AND accepted (is_accepted = decision in ("accepted","auto"), plan_pass_service.py:213), so it breaks at cast and reads 1 — not 2, not the spec sketch's 3. Never pin a literal. Render what the ledger derives.
Pass 7's output kind is scene_plan, and that is deliberate. PASS_REGISTRY["self_heal"].output_kind == "scene_plan" — it re-emits the healed scene plan, and the registry says why in-line: "which is exactly why inputs must resolve by POINTER: under a latest-by-kind rule it would read its own output as its input." So B2's schema covers six kinds for seven passes — correctly — and scene_plan must validate for both producers. One consequence for the viewer: the artifact list is DISTINCT ON (a.kind), so once pass 7 runs, the scene_plan row shows self_heal's body, not pass 6's. Say so in the panel, or the user thinks their scenes vanished.

③ The beats checkpoint — edits there is no "hold" and the cascade

edits deep-merge into the pass artifact → a new artifact → everything downstream stales by derivation, with nothing written to mark it. That is the compiler's best property and its most surprising one, so the form says it before you commit, and names the passes by name. Per the parent mockup's locked principle, this is a structured form, not a JSON patch box (OQ-2: cast + beats are the two shapes we know; anything else falls back to the read-only viewer + “re-run this pass”). The primary button is Approve with edits — ONE call ({approved:true, pass_id, edits}): the gate runs before the write, so a refused approve mutates nothing. Save edits is the secondary, and it marks the pass rejected — there is no third decision.

⑥ The ledger that lies about itself F-P3 · BE-21 — same run, two answers

derive_view(run, *, package_artifact_id=None) re-derives each pass's input fingerprint, and for the 5 passes with reads_package=True the package id is one of the inputs. Three call sites. Two pass it. _serialize_run (plan_forge_service.py:383) does not — so GET /runs/{id}, the LIST, POST /checkpoint and PATCH /novel-system-spec all report a run that has made zero progress. It has bitten nobody only because no FE reads that block — which is exactly the gap this wave closes.

What this mock does NOT propose. (1) No raw JSON/spec editor as a fallback — the parent mockup's locked principle, and it holds here: the artifact viewer is read-only, because edits (the only sanctioned artifact mutation) must go through POST /checkpoint's deep-merge so it is fingerprinted. A writable viewer would be a second, un-derived write channel into the pass ledger — it breaks PF-3, and it is the exact "make the writer fix the raw plan" failure the 2026-07-06 audit banned. (2) No OCC (PS-8) — pass_state has no version column; last-write-wins is deliberate, and the derived ledger makes the outcome visible on the next read. An If-Match here would be a new confirmation convention, and AN-8 says finding one is finding a defect. (3) No Delete-a-pass (PS-1). (4) No estimate route and no new /actions descriptor (F-P8) — three invented estimate paths already 404 in production; whether plan_run_pass should become Tier-W is OQ-1, for the PO, because it would change the agent's channel too. (5) No "Hold" (F-P10) — the checkpoint route has two decisions, accepted and rejected. A third one drawn in a mock is a state the builder would then have to invent in the client, and it would drift from the ledger on the first read.
The agent half — and the inverse gap this wave deliberately leaves open. Every action above is already an MCP tool (plan_run_pass · plan_pass_status · plan_review_checkpoint · plan_link · plan_handoff_autofix), and this spec changes none of them. Two consequences the builder must not miss: (a) registering plan-passes means two edits in frontend_tools.py — the panel_id enum and the tool's description prose, which is the model's only hint the panel exists — then regenerate contracts/frontend-tools.contract.json, never hand-edit it (57→58 == 58 == 58). (b) there is no plan_* effect handler today, so without planEffects.ts (X-4) the agent runs a pass and the Rail sits stale — mirror bookEffects.ts including unwrapToolResult, or it stays green in unit tests and never fires live. And after BE-3 the human can read an artifact body while the agent still cannot (no plan_get_artifact tool) — a real GG-2 inversion, recorded as OQ-3 rather than silently grandfathered.
