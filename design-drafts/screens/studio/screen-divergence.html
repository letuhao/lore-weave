The immutability constraint — and what this mock therefore does NOT propose.
(1) No entity_override editor. The divergence spec and its overrides are CREATE-ONCE: written inside the /derive transaction (works.py:378-405), never touched again by any route, repo method, or tool. An override editor is unbuildable on today's backend — it needs BE-13 (PATCH /divergence-spec + POST|PATCH|DELETE /overrides), deferred to v2 as D-DIVERGENCE-SPEC-EDIT. This mock draws the v1 that respects the backend: a read-only spec that names the line and the missing route. (2) No divergence MCP tools. Zero today, zero in this wave — an INVERSE gap recorded, not hidden (D-DIVERGENCE-MCP-TOOLS): a derive forks a whole knowledge partition, which is a Tier-W-shaped action that deserves its own AN-8 confirm design, not a bolt-on. (3) No hard delete. Archive is status='archived'; the chapters and the partition survive. (4) No <CompositionPanel soloPanel="divergence" />. Reuse the leaves — DivergenceWizard, DerivativeBanner, useDerivativeContext, useWhatIfPromotion — never the shell (EC-6). (5) No fourth critic_model_ref picker in ⑦. Models have one home (settings.model_roles) — SET-8. Writing both keys to dodge BE-21 is two homes, the same violation wearing a hat. (6) No blanket If-Match on patchWork. It is one shared function over five call sites, one of which drags a world map. Opt in per call site (EC-4b). (7) No entity_override editing, and no story-template authoring. Both are backend-shaped (BE-13, BE-12) and both are said out loud rather than shipped as a pencil that 405s.
