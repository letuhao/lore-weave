An archived arc has no derived block — so render “—”, never a computed 0. derived_blocks returns rows for live nodes only (structure.py:255). A panel that COALESCEs the missing block to 0 tells the user this arc has zero chapters when it has 18. Same shape as the chapter-blocks-null-nontext-COALESCE scar: a null that means not computed is not a zero. Two traps ride on this. (1) The detail route does not behave this way today — it calls span(), which happily computes a span for an archived node; the “—” only becomes true under BE-A1, whose fix must emit span:null, chapter_count:null for an archived node and not reuse the list route's empty block (whose chapter_count: 0 is the computed zero). (2) The panel therefore gates the “—” on is_archived, not on falsiness — 0 and null must never collapse into one branch.

What this mock does NOT propose. (1) No LLM, anywhere. Every action here is deterministic CRUD, $0, no propose→confirm — and adding one would be a defect. The "suggest an arc for this premise" button (composition_arc_suggest) is Wave 4: it has no REST route, so the browser gets a 403 from the BFF's FE_BRIDGE_TOOL_ALLOWLIST and fails closed at integration time. Its landing site is this header — but it lands with its backend. If a later agent adds a paid action here, it goes through the generic /actions/preview → /actions/confirm spine; three bespoke per-action estimate routes 404 in production today for exactly this reason. (2) No server-side schema for tracks/roster. They are list[dict[str, Any]] at both doors; tightening them would reject writes the agent already makes. AI-3 makes the panel safe today; the schema is deferred as D-ARC-TRACKS-ROSTER-SCHEMA (gate #2 — structural). (3) Do not derive status from member chapters. No engine branches on it (grep found none) — it is an author's label. Recorded here so the next agent does not "helpfully" auto-compute it.
The Lane-B effect handler is mandatory, not conditional (X-4). registerEffectHandler(/^composition_arc_(create|update|delete|restore|move|assign_chapters)/, …) → invalidateQueries(['plan-hub']). Without it the agent edits an arc, the open inspector keeps showing the stale row, and the user's next save 412s against a version they were never shown. This handler is the difference between "the agent and the human share one object" and "they fight over it." MCP parity needs nothing new: all 8 tools exist — the human side was the hole, and after this ships there is no INVERSE gap for arcs (which is why BE-A3 must land on both doors — a GUI-only unassign would just invert the bug).
DoD #5 — the assertions that prove this panel matters, one per prompt-bearing write. (a) Set an arc's tracks → run a draft on a member chapter → assert the prompt changed (test_pack_arc_wired.py's BA12/D2 shape). (b) Do it again for roster_bindings — bind a role to a glossary entity and assert the "Cast bindings: …" line appears (lenses.py:337-344). These are the panel's only two prompt-bearing writes; a tracks-only test leaves the entire Roster/binding editor unproven, which is the write-only-behavior bug this repo bans wearing the costume of a green suite. Do NOT write a pack-effect test for roster (slots) or summary — the packer does not read them; assert roster through extract_template_from_arc instead, and let summary/status be proven by the Hub rendering them. Plus the mandatory live browser smoke (8 steps): the agent's ui_open_studio_panel {panel_id:"arc-inspector"} leg is verified by EFFECT — a mounted dock tab, not a shown:true in the raw stream.
