That last clause is not softened. invalidate_fact sets valid_until; nothing clears it, at any layer. A restore needs a new engine fn + route + tool — filed as D-KG-FACT-RESTORE (OQ-4). Until it lands, this confirm dialog is the mitigation, so the copy has to say so.
The stale-row outcome: the tool returns {invalidated:false, reason:"no matching fact found"} with 200, not 404 — the REST mirror copies that verbatim. The row was already gone ⇒ refetch and say "already gone". Never report success.
Do NOT wire this to the triage queue. POST /pending-facts/{id}/reject (the pre-commit triage queue) and /internal/admin/…/reject-fact are different objects with different lifecycles. A committed fact is not a pending proposal. Wiring Forget to reject would silently do nothing to the fact the user is looking at — and would report success while doing it.

Wiring the handlers is the easy half. The half that matters is making the props optional so a handler-less button cannot render at all — and asserting it in a test.

Without the test, the next noop re-creates this exact defect and every suite stays green. This repo already has a law for that: an item is DONE only when a test asserts its effect (checklist-is-self-report-enforce-by-tests).
Why this is the most important item in slice 8a even though it is the smallest. A missing button is a gap. A button that renders, hovers, focuses, accepts a click and does nothing, with no error is worse than a gap — it is a lie the UI tells about its own capability, and it is indistinguishable from a working button until the user checks whether the row actually went away. That is precisely this repo's silent-success-is-a-bug-not-environment class, shipped to production, in the panel whose job is to show you the state of your knowledge.
What slice 8a costs. Four FE affordances + three thin routes (BE-14a/b/c — each a router over an engine that already exists and is already hardened) + one closed-set unification (BE-14d) + one line of Lane-B registration (/^memory_(remember|forget)$/ — outside ^kg_, so KNOWLEDGE_WRITE_PATTERN genuinely cannot match it). No new panel. No new category. No new table. No migration. No setting. No token spent. 8a is shippable on its own — which is exactly why it is split from 8b.
🟢 And one thing NOT to do. The audit claimed kg_create_node and kg_project_entities_to_nodes have no Lane-B effect handler. That is refuted. KNOWLEDGE_WRITE_PATTERN (knowledgeEffects.ts:18) is a negative lookahead over the whole ^kg_ namespace — both tools already match it and already invalidate ['knowledge-entities'] / ['knowledge-subgraph']. Do not add a handler for them. Do not "fix" the pattern. The genuinely unhandled tools are memory_forget (outside ^kg_) and, in slice 8b, every world_*.
