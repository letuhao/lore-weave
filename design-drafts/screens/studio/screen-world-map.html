🔴 UPDATE DOES NOT EXIST — and "delete + recreate" is REJECTED, not deferred. The tool set is add/remove-only at every layer. The naive fallback fails on four independent grounds:
  1. It churns ids. A marker id is the handle for its inspector selection, its deep link, and the agent's undo hint. Recreating on every drag invalidates all three.
  2. It is not atomic. DELETE succeeds, POST fails on a network blip ⇒ the user's pin is gone. There is no transaction across two HTTP calls.
  3. It is a write storm. A drag is a continuous gesture. Delete+create is 2 writes per drop and unbounded per drag-frame.
  4. It destroys ordering. Both children sort by created_at (mcp_maps.go:280, 303), so a "move" would silently jump the pin to the end of the render order.
Therefore: real PATCH routes with partial semantics — an absent key means unchanged; an explicit null on entity_id / marker_type means clear.

Why a container, and not "just open the map"
DOCK-8: a launcher hosts no capability's content. The world panel does not draw a map, a timeline, or a rollup graph. It answers three questions and hands off:

1 · Which world is this book in? — resolved from book.world_id. No param, so the panel is bare-id openable and lands in the palette, the enum and the guide.

2 · What else is in it? — the member books. Another book is a different studio, so that hop is correctly external.

3 · What maps exist? — the rail that makes world-map reachable without a route hop.

studioLinks.ts gains one rule — and it is not one line, and there is no existing rule to mirror. What we want: /worlds/{worldId} → openPanel('world') only when it is this book's world, else external. 🔴 That is not implementable against today's signature. resolveStudioLink is a pure, synchronous function (no React, no fetch — by design, studioLinks.ts:1-11) and its StudioLinkContext carries only { bookId, titleFor? } (:19-24). It has no way to learn the book's world. So the change is three files: (1) studioLinks.ts — add worldId?: string to the context; match /^\/worlds\/([^/]+)/; resolve to the panel only when ctx.worldId === worldId; when ctx.worldId is absent, fall through to external (today's behaviour exactly ⇒ no caller regresses, and no silent no-op). (2) KgOverviewPanel.tsx:44 — mandatory, not optional: onOpenWorld currently calls followStudioLink('/worlds/'+worldId, host, { bookId: host.bookId }) and throws the world id away. Pass { bookId: host.bookId, worldId }. Without this line the new rule is dead code and the papercut is still there. (3) host/__tests__/studioLinks.test.ts — matching world ⇒ studio · a different id ⇒ external · ctx.worldId absent ⇒ external (the degrade branch a builder forgets). ⚠ And studioLinks.ts:51-57 is a comment explaining why /knowledge/projects/:id is deliberately NOT mapped — it is the reasoning to copy, not a rule.

LWW No OCC on a positional write — this is a decision, not an omission. A drag is a continuous gesture; OCC over a drag produces a 412 storm and self-conflicts on the second rapid edit of the same object (instant-commit-control-over-occ-entity). The scope gate is the trap this repo has already been bitten by. Every child route must verify both world_maps.owner_user_id == caller and marker.map_id == {map_id} — expressed in SQL exactly as the existing deletes do (mcp_maps.go:432: UPDATE … FROM world_maps wm WHERE m.id=$1 AND m.map_id=wm.id AND wm.owner_user_id=$2), treating RowsAffected() == 0 as the uniform not-found. A handler that trusts a bare marker_id is a tenancy defect, not a shortcut (gate-must-derive-scope-from-the-loaded-row). Markers and regions carry no owner column — their tier is inherited through map_id → world_maps.owner_user_id, so the JOIN is the tenancy boundary.

⑦ 🔴 world-map opened BARE — the state the happy-path smoke never touches

The DoD has the agent call ui_open_studio_panel {panel_id:"world-map"} with no params. On a fresh book that resolves to nothing — no world, therefore no maps, therefore no map to select. "It always lands somewhere real" is only true if this is drawn, and a blank panel behind a shown:true is the exact silent-success-is-a-bug class. So world-map reuses world's link-to-a-world block rather than forking a second empty state — the panel that can fix the problem is one click away, in the dock, not a route hop.
