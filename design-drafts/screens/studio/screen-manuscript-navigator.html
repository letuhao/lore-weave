Manuscript Navigator — component #02

The Side Bar's arc → chapter → scene tree (adaptive depth). Built to survive 10k+ chapters / 100k+ scenes via virtualization + cursor paging + lazy expansion (the VS Code Explorer model).

1

Virtualized rows

Only the ~40 rows in view are in the DOM (the panel shows win 100–140). Scrolling re-renders a moving window over a fixed-height list; the top/bottom spacers reserve the off-screen height so the scrollbar is honest. A 10k-chapter book never mounts 10k nodes.

2

Cursor paging

Chapters arrive a page at a time (limit≈100) keyed by an opaque next_cursor. Scrolling near the tail fires the sentinel → fetch the next page → append to the store. No offset arithmetic (which drifts as chapters are added/removed); the cursor is stable. "Reload each browse" = the window re-derives from the store, holding only loaded pages.

3

Lazy scene expansion

A chapter's scenes are fetched only on expand (Ch 0004 has 300 → it loads 100 at a time, its own cursor). Collapsing drops them from the store. So opening one fat chapter never pulls the whole book's scenes.

4

Server-side jump

The filter box is a query (by number or title), answered by the server against all 10k — not a client-side filter of in-memory rows (which would only see loaded pages anyway). Hitting ↵ jumps the window to the match and expands its chapter.

5

Adaptive depth (arc → chapter → scene)

The tree shows only the levels that add grouping. Self-written books get the full arc → chapter → scene (collapse arcs to make 10k chapters navigable by group). Imports have 1 arc + 1 scene/chapter, so both degenerate levels are hidden → a flat chapter list navigated by the jump box. Same component, structure-driven — like VS Code flattening a single-child folder chain. Arcs + scenes come from the composition outline; a book without one renders chapters only.

Why this is mandatory: some manuscripts run 10k+ chapters with piles of scenes. A plain tree (mount everything, client-filter) would freeze the browser and blow memory. VS Code's Explorer scales to giant repos with exactly this recipe — windowed rendering + lazy children + paged fetch — so the navigator mirrors it.

Interaction (the "wiring", tracked as Debt #1): clicking a scene opens/focuses it in a dock group. That link is built with #03 (a dock panel to open into); until then, selection just highlights + the E2E for the link is deferred.
