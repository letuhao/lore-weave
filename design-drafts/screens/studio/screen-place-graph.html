Location-only is a hard constraint, not an oversight. buildPlaceGraph keeps a relation edge only when both endpoints are location entities — a place↔character edge is dropped on purpose. The predicate picker is the closed set contains · borders · route_to (PLACE_LINK_PREDICATES); the port preserves that enum exactly, so the FE resolver and the MCP createRelation contract stay in agreement.

③ Scope — this panel authors PLACES; other kinds live next door deep-link

A worldbuilder also needs characters, factions, concepts, items, events — and relations that are not location↔location (Hàn Lập member_of Thất Huyền Môn). Those do not belong in a place graph. The general Create-Entity / Create-Relation authoring surface is the sibling kg-entity-authoring draft; this panel links to it.

Entity kinds — where each is authored
Here (place-graph):
location
— as map nodes, with location↔location links
Sibling (kg-entity-authoring):
character faction concept item event skill + any custom kind
both call the same createEntity / createRelation — this panel is just the location-scoped, spatial view of it.
Same API, two views. Adding a place here and adding a character in the entity-authoring panel are the same knowledgeApi.createEntity call with a different kind. The place graph is the spatial projection: it renders location entities as draggable nodes and their spatial relations as edges. Nothing here is a second write path — the deep-link keeps one concept with one home (no "authored in eight places" drift).
What this mock does NOT propose. (1) It adds no backend — if a control here implied a new route, that would be a defect in the draft. (2) It is not the raster world_maps editor: no base image, no pixel pins, no polygons — those are book-service, per-world, owner-scoped, and largely unbuilt (see screen-world-map-editor.html). (3) It does not author non-location kinds or non-spatial relations; those deep-link to the entity-authoring panel. Keeping these three apart is the point — merging any two of them is the confusion S7 closes.
