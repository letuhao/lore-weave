What "view-only" actually costs. Because the agent can only add_marker/remove_marker (never move), even the agent cannot nudge a misplaced pin — it must delete and re-create it, minting a new marker_id and dropping any reference to the old one. A human editor that drags pins is therefore not "just a frontend" — it is blocked on an UPDATE contract that does not exist for anyone.

Drag is an UPDATE, and UPDATE is the hole. A pin drag must send a position delta to PATCH /v1/worlds/{id}/maps/{map_id}/markers/{marker_id} and persist on onDragEnd (the same shape the place-graph already uses for node positions). The current toolset has only add_marker + remove_marker; simulating a move as remove+add is not atomic (a disconnect strands the delete) and churns marker_id, breaking anything that referenced it. The mock renders the readout so the contract it needs is unmistakable.
The entity tie is a SOFT cross-service UUID, and rebinding must stay soft. A marker's entity_id points at a glossary location entity but carries no FK (book-service and glossary are different services). "Rebind" swaps that UUID; deleting the glossary entity must NOT cascade-delete the pin — the pin simply renders unbound. This is the same soft-pointer posture world_map_add_marker already takes; the human editor inherits it rather than inventing a hard link.

This is additive, not a re-architecture. The tables, MinIO storage, owner-scoping, and the 8 agent tools all exist and are correct. What is missing is (a) a public REST mirror of the create/add/remove tools and (b) an UPDATE contract that does not yet exist for anyone — three PATCH routes + three matching MCP tools (rename map, move/rebind marker, reshape region). Ship the PATCH tools too, or the agent stays unable to move a pin it placed wrong.
What this mock does NOT propose. (1) It is not the place-graph — no location↔location node edges live here; a raster map's regions are polygons, not graph relations (plan 30 §10). (2) It does not grant sharing — worlds have no E0 grants, so every route stays strictly owner_user_id-scoped, and a foreign map is a 404 with no existence oracle. (3) It does not make the entity tie a hard FK across services. A later agent should not "simplify" drag by looping remove+add, nor add a cross-service foreign key to bind a pin.
