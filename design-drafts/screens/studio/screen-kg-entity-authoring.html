There is no form — even though createEntity (api.ts:1607) and createRelation (api.ts:1617) already ship and are called today by the World Map for kind:'location'. This is a human authoring surface over kg-entities + kg-graph. Pure FE wiring — no new backend.

To add a character today you must run extraction or approve an agent proposal — there is no form, though createEntity is ready. The endpoint exists (POST /entities, payload {project_id, name, kind}, api.ts:1607) and returns a 201 with the fresh entity. Its only human caller is useWorldMap.ts:129, which passes kind:'location'. Nothing on the backend rejects kind:'character' — the button was simply never drawn.

Anchored vs discovered is derived, not chosen. A brand-new entity has no glossary_entity_id, so the server-computed status is discovered (ENTITY_STATUSES = canonical / discovered / archived, api.ts:386). It becomes canonical only when a glossary anchor is attached — this form never lets the user fake that state.
Enum parity is a real, tracked drift. The filter renders kind as a select over 7 values (EntitiesTab:24), but the edit dialog types it as a plain <input> (EntityEditDialog:156) — a free string that could write kind:"charcter" silently. This create form uses the enum picker above; making Edit match is the one-line follow-up flagged in the banner.
What this form does NOT add. There is no DELETE / archive control here. No entity-level delete route exists in api.ts today (only archiveProject; merge-into archives the loser). The S7 audit flagged the kg-overview delete buttons as unverified — this screen links out to Merge (the only archival path) rather than implying a route that isn't proven.

④ The result no silent success · no silent failure

On 201 the new entity appears as a table row and a graph node (query invalidation, no reload) and a success toast fires. On 409/422 the create is rejected and an error toast explains why — the W0-S16 law: never a silent success, never a silent failure.

The graph and table share one source, so both refresh from one write. After the 201, the existing query invalidation (['composition','worldmap',…] for places; the entities/subgraph queries for kg) re-fetches — the new node (outlined amber, LMB above) and the new ally_of edge both appear without a reload. No optimistic fabrication: the row shown is the server's returned entity.
Every failure surfaces — the W0-S16 law. createRelation already returns 409 (endpoint not the caller's entity) and 422 (self-loop); the resolver maps each to a specific error toast, never a swallowed rejection and never a fake success. This mirrors the EntityEditDialog 412-conflict handling that already ships (EntityEditDialog:61).
What this draft does NOT claim to build. Zero backend. createEntity + createRelation exist and are proven by the World Map's live use of them. This is the "API ready but unwired" case: the entire deliverable is the two GUI surfaces above plus their MCP-tool parity. The one honest gap — an entity-level delete/archive route — is left unbuilt and flagged, not silently faked with a button that no-ops.
