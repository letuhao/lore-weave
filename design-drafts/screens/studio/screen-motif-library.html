② The six scope tabs — two are new, and one route feeds two of them

The shipped MotifScopeTabs has four: mine / system / all / drafts. The tab is the tier (§8.1), so the book_shared tier gets one — and its route, GET /motifs/book/{book_id}, has existed all along with zero FE consumers. 3a wires it. The catalog tab is the public, adoptable list. ⚠ But that route is not tier-filtered, and the first draft of this mock said it was.

One route, two tabs — and the Book tab must filter, or it duplicates Mine. GET /motifs/book/{book_id} (motif.py:219) returns merged, with no tier / scope param: the caller's global motifs (the Mine tier) + this book's private labels + the book's SHARED rows. Its projection _book_view (:207-216) does return book_id + book_shared on every row — so the split is a free client-side partition and needs zero backend work: Book = row.book_id === bookId && !row.book_shared · Shared = row.book_shared === true, from the same single fetch. Drop the book_id test and every motif already on the Mine tab is rendered a second time under Book. And note Drafts is a status, not a scope (?status=draft; scope is only mine|system|all). A reviewer who finds two fetches of this route, or an un-book_id-filtered Book tab, has found a defect.
