The cascade IS the panel. A slider that reads 50 when the packer will draft at 78 is a lying control (SET-1), and it is the same dirty-signal-computed-but-not-surfaced class this repo has shipped before. So every row carries its effective value and the tier that supplied it, using the shipped TierChip from chat-ai-settings — one chip, one vocabulary, no second copy. The Clear override button is exactly the live-but-unreachable useDeleteStyleProfile (useStyleVoice.ts:30), rendered through the chip's shipped sibling ClearOverride (same file, :60), and it appears only when a row exists at this exact scope.
⚠ The chip is shipped; these four tiers are not. TierChip's CLS/LABEL/TITLE maps (TierChip.tsx:13-38) carry session · book · account · system · unavailable · no_model_configured — and no scene, chapter, work or neutral. The component does not throw on an unknown tier; CLS[tier] ?? 'bg-muted…' and LABEL[tier] ?? tier render the raw slug in grey, with no tooltip. So the panel would ship looking done, with every chip reading a lowercase string — a quieter failure than the bug it replaces. M1 adds the four rows. The chips above are the acceptance criterion for them. (Note the vocabulary is deliberately distinct: the model cascade's book tier ≠ the style cascade's work tier — one name for one concept, in both directions.)
These two tables have no version column — and that is deliberate. The PUT is an ON CONFLICT (project_id, scope_type, scope_id) DO UPDATE upsert: last-writer-wins by design. The memory instant-commit-control-over-occ-entity-needs-write-serialization does not apply — there is no version to self-412 against. But the second half of it still does: rapid slider commits are debounced and chained, never raced, so two in-flight PUTs cannot land out of order.

The one-way door, rendered honestly (EC-2a). The Work's embedding model is write-through on the first successful add and immutable afterwards. That is correct engineering — one Work, one vector space — reported dishonestly: LIST returns only embed_model_set: bool (references.py:92), never the ref, so a user can neither see which model owns their vectors nor change it. v1 renders it read-only, with the ref, its tier, and one sentence. Silence about a permanent choice is worse than a disabled control that explains itself. ⚠ OQ-4: the "re-embeds all N" copy needs a real N — measure it on the dev DB before writing it.
What these mocks do NOT propose. (1) No reference edit pencil. There is no PATCH /references/{rid}; shipping a control that 405s, or one that silently re-embeds to fix a typo, is worse than the honest strip above (BE-17, deferred to v2). (2) No model switcher once the vector space is set — changing it re-embeds every passage and is a migration, not a select. (3) Neither panel mounts <CompositionPanel soloPanel="style" />. It looks like free DOCK-2 reuse; its body calls ~20 hooks before it reaches the slot table and depends on WorkspaceLayoutContext. Reuse the leaves, not the shell (EC-6). (4) The shelf is not the GroundingPanel. The shelf is the corpus; grounding is what this scene actually used. The split stands (OQ-6); the two deep-link to each other.
MCP parity is the point of the wave, not a footnote (MCP-first invariant). BE-10 ships six Tier-A tools — composition_style_{list,upsert,delete} · composition_voice_{list,upsert,delete} — each returning _meta.undo_hint (on delete: the prior values; on upsert: the prior row, or None if it was unset — because "unset" and "50/50" are different states, and an undo that restores 50/50 to a row that never existed is a new bug). scope_type is an enum, registered in CLOSED_SET_ARGS. A Lane-B effect handler (compositionEffects.ts) invalidates ['composition','style-profiles',pid] and ['composition','grounding',pid] on every agent write — the human path already does this (useStyleVoice.ts:24-25); the agent path must not be the one that goes stale.
