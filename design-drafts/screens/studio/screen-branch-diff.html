Branch Diff — dị bản prose audit — a section inside the `divergence` panel · draft

The Spec tab answers "how was this branch declared?"; this Diff tab answers "what is actually different in the prose?". A dị bản is copy-on-write — its project holds only the scenes the writer diverged, so each scene is inherited (verbatim canon, nothing to diff), changed (two-column diff), or added (all-new). Prose is fetched lazily per selected scene for both projects via one new read route (GET /works/{id}/scenes/{node}/prose, wrapping the existing composition_get_prose resolver). The rail hides inherited scenes by default, so the audit stays short at book scale.

Why a section, not a new panel. The audit is meaningless without a selected branch, which the divergence panel already owns — so it rides the same selection as the Spec tab (one selectedProjectId, two tabs). It also needs no new panel_id enum entry, which is exactly the single-line contention that blocks a fresh panel while sibling sessions add their own. One new backend read route (scene prose by node, both projects), a client-side line diff, and lazy per-scene fetch — no bulk book-diff, no new bi-temporal query.
Scene correspondence is by (chapter_id, story_order), and it must degrade honestly. A canon scene the writer never drafted has no prose — show "canon scene not drafted" on the left, the dị bản prose on the right, and label it added-vs-blank, not a false "changed". A dị bản node whose story_order collides with a different canon scene must NOT be mis-paired — pair only within the SAME chapter, and when in doubt render it as added (all-new), never a wrong diff. A wrong pairing is worse than no pairing.
