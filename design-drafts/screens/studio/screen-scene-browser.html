Spec: 22_scene_model_and_crud.md. Two panels over a scene model that is split across two services on purpose: book-service owns scene identity (Per-book, shared with every collaborator, holds the parsed leaf_text), composition-service owns authoring intent (Per-Work, anchored by a new scene_id). Today identity has no public HTTP route at all and intent exposes 2 editable fields — which is why an imported book's Scene Rail renders empty.

Why the pair, and not one panel. Same reasoning as the Chapter Browser (#15): a table cannot edit 20 fields well and a detail pane cannot triage 148 rows well. SceneRail stays — it is the chapter-local strip beside the editor, a third job again. Selecting a row here drives the inspector; the inspector's ↗ jumps the prose via data-scene-id.
Every write here is OCC'd. The inspector sends expected_version (v7 above); a stale write returns 412 and the panel reloads with the "changed elsewhere — reloaded" notice SceneRail already implements. Identity writes and intent writes hit different services and therefore carry two independent versions — the footer shows the intent version because that is what a human edits most.

Import materializes intent, but the browser never depends on it (SC6/SC11). After parse, a composition internal route upserts one outline_node per parsed scene if a Work exists. If none does, nothing happens and this screen still works. The upsert is idempotent on (project_id, scene_id) — the scoped-unique index is the retry guard. A 200 that upserted 0 rows on a book with parsed scenes is a bug, not a no-op.

What this mock does NOT propose. leaf_text is never editable here. It is a read-only parse projection of chapter.body, rebuilt by the parser. Studio decision D17 — "scene prose stays in chapter.body; scenes[] is outline metadata only" — still holds, and a derived projection is not a second prose store. Prose edits go through chapter_drafts, exactly as today. Stated here because the next agent to open this file will otherwise "fix" the browser by making it editable.
MCP parity is not optional (MCP-first invariant). Everything above is agent-reachable: book_scene_list / get / create / update / delete / reorder on book-service (VIEW read, EDIT write), and composition_outline_node_create/_update widened from 4 writable fields to the full intent set. Closed sets (status, beat_role, origin, pin action) ship as enum, registered in CLOSED_SET_ARGS — the free-string panel_id bug is the reason that rule exists.
