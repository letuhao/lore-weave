The Spine — authoring a novel's structure

A cold-start walkthrough found a first-time user cannot write a single word: the Manuscript's only add button has never been wired, and the Editor points back at it. This draft replaces the dead + — and the tree it belonged to — with a lane timeline that can actually express the model it sits on.

the pointThe hatched band is the overlap: Vesna's arc and Auger's arc both run through chapters 3–4. This is the single thing a tree can never draw — a chapter gets one parent, so the collision your book is actually about becomes invisible. The backend already models it (POST /books/{id}/arcs/assign-chapters); only the GUI was missing.
real valuesChapter fills are the migration's own enum — empty · outline · drafting · done — not invented states. Progress is a texture you read in one glance, not a number you go find.
authorshipAmber beats are yours; teal beats the planner proposed. The third arc is still mono + teal — the machine suggested it and you haven't adopted it yet.

Everything here is the planner's. Nothing is yours until you take it.
honestThe planner's output lands on the same spine, in teal, marked as not-yours. Adopting turns a span amber. The writer is never confused about who wrote what — which is the whole promise the product makes.
measured87.2s and $0.00 are the dry run's real numbers (Gemma-4 26B-A4B QAT, local). The latency is shown, not hidden — a spinner that lies about 87 seconds is worse than a number that doesn't.
openDesign Q1 is still the PO's: does a new book get a Work automatically? If yes, state 01 has a spine to draw on from the first second. If no, "Start with an arc" must create the Work silently — never make the writer learn the word "Work".
