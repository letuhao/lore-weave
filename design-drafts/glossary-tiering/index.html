⚠ The drift we're correcting

Since March the data drifted to a flat genre_tags TEXT[] on a single global kind table — losing the primary design: genre is a first-class, tiered level, and attributes are scoped by (kind × genre) (different genres give a kind different attributes and different attribute descriptions). SS-4 already split kinds into system_kinds + user_kinds and locked system writes — that's a correct slice of the Kind row below, but Genre + the (kind×genre) attribute model still need building.

The 3 × 3 model — 9 combinations

Three tiers (who owns it) × three levels (what it is). Every row resolves System → User → Book, higher shadows lower by code.

Level ↓ · Tier → System admin/seed only User owner_user_id Book book_id + E0 grants

Attributes depend on (kind × genre)

The same kind has a different attribute set — and different descriptions — per genre.

character × universal
name · aliases · description (genre-independent base)
character × xianxia
cultivation_realm · dao_heart · spirit_root · sect
character × romance
love_language · emotional_wound · relationship_status

Multi-genre merge = keep both, namespaced

A book/entity can carry several genres at once. Same attribute code from two genres → both kept, labeled by genre (no silent override).

character · genres = [universal, xianxia, romance]
  • nameuniversal
  • cultivation_realmxianxia
  • rank · conflict, kept both rank·xianxiarank·romance
  • love_languageromance

Resolution within one genre still merges System→User→Book (higher tier shadows lower by code). Across genres = keep both.

Resolution order

System defaults → shadowed by → User overrides/additions → shadowed by → Book overrides/additions

For a given (kind, genre, code): the highest tier that defines it wins. A regular user never edits a System row — they clone or override into their own tier. (CLAUDE.md › User Boundaries & Tenancy.)

Boundary independence — copy-on-adopt, sync on demand (D8)

Each boundary holds its own frozen copy of what it adopts. Upstream changes never push or destroy downstream — you pull them on demand via Updates & Sync.

remove → deprecate (never destroys a consumer) edit → publish (consumer pulls on demand) add → offer (not auto-injected)

A book is self-contained (its own adopted ontology) → reproducible, and the substrate a per-book KG builds against.
