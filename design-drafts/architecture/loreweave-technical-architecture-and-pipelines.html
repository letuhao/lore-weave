Architecture & pipelines — Phase 1–5 delivery vs Phase 6+ design

This page is for engineers: service grouping, data-plane dependencies, and the operational pipelines that exist in the monorepo today. Phase 6+ (Living Worlds / game-oriented extensions) remains a design track—gated on novel-platform maturity and prototype metrics—not an implemented product surface.

A

System architecture (engineering view)

From browsers and other external clients, traffic enters through the NestJS gateway. Domain logic is split across Go microservices; AI-facing workloads use Python services and background workers. Each service owns its Postgres schema; Redis, RabbitMQ, and MinIO provide streams, queues, and object storage; Neo4j is optional for graph projections when Track-2 features are enabled.

B

Operational pipelines (Phase 1–5)

High-level flows that exist in the codebase today. Names reflect containers/services in infra/docker-compose.yml; internal events and job APIs evolve—treat this as orientation, not a frozen sequence diagram. Some strips below are fully operational, while others are partial or planned integrations.

Authoring → persistence
Operational — manuscript SSOT. Chapters and chunks persist through book-service; structured lore and wiki artifacts live in glossary-service. Binary assets route to object storage where applicable (minio in dev).
frontend → api-gateway-bff → book-service → postgres | minio
Translation
Operational (quality depth in progress) — async batch translation. translation-service orchestrates jobs; translation-worker consumes rabbitmq work; models and keys resolve via provider-registry-service.
translation-service → rabbitmq → translation-worker → provider-registry-service
Async LLM jobs & notifications
Operational — provider jobs & terminal events. Long-running LLM work is brokered through provider-registry-service with billing hooks; async completion can surface as notifications via notification-service when messaging is configured (e.g. RabbitMQ).
provider-registry-service → usage-billing-service → notification-service
Retrieval serving pipeline (knowledge → app workflows)
Partial — app-facing retrieval context. In addition to extraction/storage, knowledge-service serves context back to applications (for example chat memory/context assembly). Current chat integration uses context-build calls with degraded fallback behavior when the service is unavailable; broader book-service retrieval integration remains planned.
chat-service → knowledge-service (/internal/context/build) → postgres / neo4j → retrieval response
Knowledge & extraction (Track 1–2)
Operational core + partial consumers. Event-driven extraction enriches and stores knowledge data; retrieval serving is a separate path (see strip above). Book/chat-related changes propagate via outbox patterns and relays (worker-infra) into streams consumers read. knowledge-service owns Postgres SSOT for knowledge rows; worker-ai drives Pass-2-style LLM extraction through the internal gateway; optional neo4j holds derived graph projections when enabled. Extraction is opt-in per project and cost-gated in planning docs—background jobs, not interactive latency.
book-service → outbox → worker-infra → knowledge-service + worker-ai → postgres / neo4j
Chat voice + knowledge collection pipeline
Partial — integrated in parts, depth evolving. Chat voice flows include STT/TTS paths and context assembly, while chat session completion feeds event-driven knowledge collection/storage. The core plumbing exists (voice routes/events, audio adapters/workers, outbox events), but quality and coverage are still being expanded.
voice input → chat-service → provider audio path (STT/TTS) → chat turn complete → outbox → worker-infra → knowledge-service + worker-ai
Image / video generation (ComfyUI)
Partial — ComfyUI-backed media stack (separate repo, integration in progress). Generation workloads are intended to run in local-image-generator-service (ComfyUI): SD 1.5, SDXL, Illustrious, Flux 1/2, Qwen Image, Wan, LTX Video, and custom graphs for game assets, object / sprite sheets, and animation outputs. LoreWeave’s video-gen-service exposes gateway-facing routes and storage/billing hooks while this topology is finalized.
frontend → api-gateway-bff → video-gen-service → local-image-generator-service (ComfyUI) → minio

Current monorepo code paths may still call provider APIs directly for some requests during transition.

C

Planned bridge — book / glossary → game “reality” data

A dedicated production pipeline from authored prose to game-ready datasets is not fully specified. The intent is to extend glossary- and extraction-derived artifacts toward runtime-friendly bundles while keeping canon review in the loop.

Design in progress · wire format not locked

Provisional pipeline sketch

  • AI collection from book text — leverage extraction and summarization paths (existing knowledge/glossary direction) to propose structured candidates: places, factions, timelines, rule-like facts.
  • MCP as an integration surface — connect editors/agents and internal tools via MCP-shaped endpoints so enrichment jobs can be invoked consistently; treat this as a contract boundary, not a single frozen schema.
  • Human-in-the-loop enrichment — curators approve, merge, or reject automated proposals before they become authoritative game inputs (parallel to glossary/wiki curation today).

Job boundaries, storage targets, and migration from Phase 1–5 tables to any future game-specific databases are open engineering work. Authoritative narrative for the extension lives under docs/03_planning/LLM_MMO_RPG/README.md.

D

Phase 6+ game platform areas (design breadth)

The following areas describe planned layering and systems for a text-first, lore-grounded game experience. They are not claimed as implemented stacks in the current monorepo; they organize expected problem domains for future specs.

Design track · Phase 6+

Spatial & world representation

  • Four-layer map model (macro → tactical granularity).
  • World graph nodes linking locations, routes, and topology.
  • World-map “papers” (authored or generated cartographic artifacts).
  • Zone-level 2.5D presentation for exploration contexts.
  • Cell / domain / battlefield framing for combat or set-piece scenes.
Design track · Phase 6+

Characters

  • Background bundles (history, goals, constraints).
  • Personality axes / voice priors for dialogue and narration.
  • Appearance descriptors aligned with continuity checks.
Design track · Phase 6+

Memory systems

Per-character and per-relationship memory aggregates to bound retrieval cost while preserving scene coherence—see open problems around NPC memory at scale in the extension catalog.

Design track · Phase 6+

Event sourcing & chronicle

Append-only event chains for significant world actions; projections for current state; aligns with the extension’s storage engineering direction (event log, reality-scoped databases)—no shipped game event store exists yet.

Design track · Phase 6+

Other systems

Narration safety layers, turn arbitration, multiverse fork semantics, realtime presence, billing tiers, and operational SLOs are catalogued across hundreds of features. Start from docs/03_planning/LLM_MMO_RPG/catalog/_index.md and scope summaries in catalog/99_scope_and_refs.md rather than duplicating the registry here.
